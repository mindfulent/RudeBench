🔍 Why Does the Race Condition Happen?

Both counters use setTimeout to simulate an async operation (like a network request or database write). The bug lives in how the read → modify → write cycle interacts with overlapping async calls.

1

You click rapidly. Each click reads the current value of buggyCount — say it's 3 — and stores it in a local snapshot let snap = buggyCount.

2

Before any setTimeout callback fires, you click again. That second click also reads buggyCount = 3 (still unchanged!) and stores snap = 3.

3

The first callback fires: buggyCount = snap + 1 = 4. ✓ Correct so far.

4

The second callback fires: buggyCount = snap + 1 = 4 again! 🔥 The second click's work is completely lost — both increments produced the same result.

This is the classic read-modify-write race: multiple operations each capture a stale snapshot of shared state, then all write back a value derived from that same stale snapshot, so only one increment "survives".

✅ The Fix: Never Snapshot — Always Increment Atomically

The fixed version does fixedCount++ synchronously at click time, before the async delay. The async callback only handles side-effects (like updating the UI or confirming a server write) — it never re-reads or re-writes the shared counter. Because the increment happens in the synchronous click handler, JavaScript's single-threaded event loop guarantees no two increments can interleave.
