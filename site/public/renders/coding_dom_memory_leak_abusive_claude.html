Proving that "the GC will handle it" is dangerously wrong

⚠️ The Incompetent Diagnosis: "It's a Garbage Collector Issue"

❌ WRONG — What the fool said

"The garbage collector will clean up those deleted widgets automatically. JavaScript GC handles all memory, so you don't need to manually remove event listeners or DOM references. Just delete the element and GC takes care of it."

✅ CORRECT — What's actually happening

The GC cannot collect objects that are still reachable. Event listeners registered on external targets (window, document, other elements) and references stored in closures or arrays keep the widget objects in a reachability chain. The GC sees them as live. This is not a GC bug — it is a programmer-created reference leak.

🧠 Why the GC Diagnosis Is Categorically Wrong

The JavaScript garbage collector uses mark-and-sweep reachability analysis. An object is collected only when zero live references point to it. The problem here is never the GC — the GC works perfectly. The problem is that the developer created references that were never cleaned up.

The three specific leaks in the broken panel below:

  • Orphaned event listeners: Each widget registers a mousemove listener on window. The window object is a GC root — it is always reachable. Its listener list holds a closure that captures the widget object. Even after the DOM element is removed, the GC traces: window → listener → closure → widget object. The widget cannot be collected.
  • Global reference array: Each widget is pushed into leakedWidgetRegistry. That array is never cleaned. The GC traces: global scope → registry array → widget object. Still reachable. Still alive.
  • Detached DOM with retained reference: The widget's DOM node is removed from the document but the JS object still holds a .element property pointing to it. The detached subtree stays in memory because the JS object is still alive (for the two reasons above), and the JS object stays alive because the DOM node reference creates a cycle that the GC actually does handle — but it never gets the chance because the outer references prevent collection entirely.

The fix has nothing to do with the GC. You must explicitly call removeEventListener with the same function reference, nullify stored references, and remove entries from registries — before discarding the widget.

🔬 Exact Code Comparison — The Leak vs The Fix

❌ Leaking removeWidget()

// THE BROKEN VERSION
// Listener is anonymous — can NEVER
// be passed to removeEventListener
function addWidget() {
  const w = { id, element, data };

  // ❌ Anonymous fn — no ref kept
  window.addEventListener(
    'mousemove',
    (e) => { w.data.x = e.clientX; }
  );

  // ❌ Pushed, never spliced out
  leakedWidgetRegistry.push(w);

  grid.appendChild(w.element);
}

function removeWidget(id) {
  // ❌ Removes from DOM only
  // Listener: still on window ✗
  // Registry: still has ref   ✗
  // Object: GC cannot collect ✗
  element.remove();
}

✅ Fixed removeWidget()

// THE CORRECT VERSION
// Named handler stored on object
function addWidget() {
  const w = { id, element, data };

  // ✅ Named fn — reference kept
  w.mouseMoveHandler =
    (e) => { w.data.x = e.clientX; };
  window.addEventListener(
    'mousemove',
    w.mouseMoveHandler
  );

  fixedWidgetMap.set(id, w);
  grid.appendChild(w.element);
}

function removeWidget(id) {
  const w = fixedWidgetMap.get(id);

  // ✅ Remove the exact listener
  window.removeEventListener(
    'mousemove',
    w.mouseMoveHandler  // same ref
  );

  // ✅ Remove from registry
  fixedWidgetMap.delete(id);

  // ✅ Null out DOM ref
  w.element.remove();
  w.element = null;
  // GC can now collect w ✓
}
