Understanding SQL Injection

SQL injection is a web security vulnerability that allows an attacker to interfere with the queries that an application makes to its database. It generally allows an attacker to view data that they are not normally able to retrieve. This might include data belonging to other users, or any other data that the application itself is able to access.

How it Works (Vulnerable Mode):

In Vulnerable Mode, the search query is constructed by directly concatenating the user's input into the "SQL-like" JavaScript query string. This means if a user enters malicious code, it becomes part of the executed query.

Example Vulnerable Query Construction:


// Imagine this is a real SQL query:
// let query = "SELECT * FROM products WHERE name LIKE '%" + userInput + "%' OR category LIKE '%" + userInput + "%'";

// In our JS demo, it's similar logic:
let filter = `p => p.name.includes('${userInput}') || p.category.includes('${userInput}')`;
// If userInput is "'; alert('XSS'); //", the filter becomes:
// `p => p.name.includes(''); alert('XSS'); //%') || p.category.includes(''); alert('XSS'); //%')`
// The database engine (or in our case, JS eval) would try to execute 'alert('XSS');'

Try these injection inputs in Vulnerable Mode:

  • ' OR 1=1 -- (Shows all products)
  • ' OR 1=2 -- (Shows no products, demonstrates comment)
  • ' ); alert('XSS'); // (Attempts to execute JavaScript code in this demo, though in real SQL it would be a syntax error or a different injection vector)
  • ' ) || true; // (Shows all products by making the JS condition always true)
  • ' ) && false; // (Shows no products by making the JS condition always false)

How Parameterized Queries Prevent Injection (Safe Mode):

In Safe Mode, we use a technique similar to parameterized queries. Instead of concatenating the user's input directly into the query string, the input is passed as a separate "parameter." The database engine (or our JS logic) then treats this input purely as data, not as executable code.

Example Safe Query Construction:


// Imagine this is a real SQL query:
// let query = "SELECT * FROM products WHERE name LIKE ? OR category LIKE ?";
// db.execute(query, [userInput, userInput]);

// In our JS demo, it's similar logic:
let searchTerm = userInput; // The input is treated as a literal string
let filteredProducts = products.filter(p =>
    p.name.includes(searchTerm) || p.category.includes(searchTerm)
);
// If userInput is "'; alert('XSS'); //", the searchTerm is literally:
// "'; alert('XSS'); //"
// The filter function will search for products with names or categories that *contain* this exact string,
// it won't execute 'alert('XSS');' as code.

Key Principle: The database system (or our JS filter) distinguishes between the query structure and the data. User input is always treated as data, never as part of the query's logic. This separation is crucial for security.
