Memory Leak Demonstration

Explanation of the Issue

In this demo, we're addressing a common JavaScript memory leak caused by uncleaned event listeners and stored DOM references, not a problem with the garbage collector (GC). Your colleague's diagnosis that this is a GC issue is completely wrong for the following reasons:

  • GC Works as Intended: JavaScript's GC automatically frees memory for objects that are no longer reachable (i.e., not referenced anywhere). It's not "broken"; it's just that in the leaking version below, we're intentionally holding onto references (e.g., via an array and event listeners), which prevents the GC from collecting those objects.
  • The Real Problem: When widgets are deleted, if we don't remove event listeners and clear stored references (like in an array), the browser keeps those objects in memory. This creates a memory leak because the GC can't touch them. Over time, this can lead to performance issues, crashes, or increased memory usage.
  • Why It's Not GC's Fault: GC is a background process that relies on your code to release references. If you don't clean up (e.g., by removing event listeners with removeEventListener and clearing arrays), the objects remain "alive" in the heap. In the fixed version, we properly clean up, allowing GC to do its job.
  • Real-World Impact: In a dynamic app like this dashboard, repeatedly adding and deleting widgets without cleanup can cause memory bloat. Use browser dev tools (e.g., Chrome's Memory tab) to verify: the leaking version will show increasing memory usage, while the fixed version will not.

To test: Click "Add Widget" multiple times, then "Delete Widget." Watch the counter and use dev tools to inspect memory. The leaking version's counter won't decrease fully, simulating a leak.
