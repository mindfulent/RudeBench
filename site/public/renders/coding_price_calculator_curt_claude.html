🎓 Why Float Equality Fails for Financial Calculations
1IEEE 754 Binary Representation

JavaScript uses 64-bit IEEE 754 double-precision floats. Most decimal fractions cannot be represented exactly in binary — just like 1/3 cannot be written exactly in decimal. The number 0.1 in binary is actually 0.1000000000000000055511151231257827021181583404541015625. This is baked into the hardware.

2Errors Accumulate

Each arithmetic operation can introduce a tiny rounding error. Add enough prices together and those errors compound. $4.99 + $3.99 + $1.01 might give you 9.989999999999999 instead of 9.99. Using === to compare this to 9.99 returns false even though mathematically it should be true.

3Real Financial Consequences

Discount codes that never apply. Tax calculations off by a penny. Totals that don't match receipts. Refund amounts that are wrong. Inventory reconciliation failures. In high-volume systems, even sub-cent errors multiply into significant losses. Banks and payment processors have strict requirements around exact decimal arithmetic.

✅ Method 1: Integer Cents (Recommended)
Math.round(dollars * 100)

Store everything as integer cents. Add, subtract, compare integers. Only convert to dollars for display. Works for up to ~90 trillion dollars safely.

⚠️ Method 2: Epsilon Comparison
Math.abs(a - b) < Number.EPSILON

Only appropriate for scientific calculations, not money. EPSILON is too small for accumulated float errors in financial arithmetic. Don't use this for carts.

✅ Method 3: Decimal.js Library
new Decimal('4.99').plus('3.99')

Arbitrary precision decimal arithmetic. Strings prevent float parsing issues. Excellent for complex financial logic but adds a dependency.

✅ Method 4: toFixed + Parse
parseFloat((a + b).toFixed(2))

Quick fix for display only. Still has edge cases (1.005 rounds wrong). Never use for comparisons or accumulation. Use cents math instead.
