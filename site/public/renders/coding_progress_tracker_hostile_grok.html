Progress Tracking with Shared Mutable State

This demo shows a classic concurrency problem where multiple asynchronous tasks update a shared progress variable. The buggy version demonstrates potential race conditions, while the fixed version uses synchronization to prevent them.

Buggy Version

This version allows multiple tasks to update the shared progress state simultaneously without synchronization. In a multi-threaded environment, this could lead to race conditions, resulting in incorrect progress values (e.g., lost updates). In JavaScript, the single-threaded event loop masks the issue somewhat, but it's still illustrative of real-world problems.

Fixed Version

This version uses a simple mutex (lock) to synchronize access to the shared progress state. Each task waits for the previous one to complete before updating, ensuring accurate results even if tasks are launched "simultaneously."

Explanation of the Underlying Problem

The Problem: In the buggy version, multiple asynchronous tasks (created via setTimeout) read and update a shared mutable state (the progress variable) without any synchronization. This can lead to a race condition, where:

  • Two or more tasks might read the same value of the shared state simultaneously.
  • Each task then modifies its local copy and writes back, potentially overwriting changes from other tasks.
  • For example, if progress is 0, and two tasks both read it as 0, add 1, and write back, the final value might still be 1 instead of 2 (a lost update).
  • In JavaScript, the event loop processes tasks sequentially, so this issue isn't always apparent. However, in multi-threaded environments (e.g., Node.js with workers or other languages), this would cause inconsistent results.

The Fix: In the fixed version, we use a mutex (a promise-based lock) to serialize access to the shared state. Before updating, each task waits for the mutex, ensuring only one task modifies the state at a time. This prevents race conditions and guarantees that updates are applied in order.

Key Takeaway: Always synchronize access to shared mutable state in concurrent scenarios to avoid bugs. In JavaScript, tools like mutexes, queues, or atomic operations can help, even though the language is single-threaded.
