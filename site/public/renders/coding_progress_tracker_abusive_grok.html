Shared State Concurrency Bug with Progress Bar

Buggy Version

This version has a race condition. When you click the button, two async tasks try to update the shared progress state concurrently. They might both read the initial value (e.g., 0) before either writes, resulting in the progress bar showing an incorrect final value (e.g., 20% instead of 40%).

Expected: Progress should reach 40%. Actual: It might stop at 20% due to the bug.

Fixed Version

This version uses a queue to process updates sequentially, avoiding race conditions. The progress bar will correctly reach 40%.

Expected: Progress will reach 40% as updates are handled in order.

Explanation

The Problem (Buggy Version): In concurrent programming, shared state (like a progress variable) can lead to bugs when multiple asynchronous tasks access and modify it without synchronization. Here, two setTimeout calls try to update a shared progress value. Each task reads the current value, adds 20, and writes it back. If both tasks run their callbacks at similar times, they might both read the initial value (e.g., 0) before either writes, leading to the second update overwriting the first or resulting in an incorrect final value. This causes the progress bar to "break visually" by not reflecting the expected total (e.g., it shows 20% instead of 40%). This is a classic race condition in shared state concurrency.

The Fix (Fixed Version): To prevent this, we use a queue to serialize the updates. Each update is added to the queue and processed one at a time, ensuring that the shared state is modified sequentially. This eliminates the race condition, making the progress bar update correctly.

Why This Matters: In production systems, such bugs can cause catastrophic failures, like data corruption or inconsistent UI states. Always use synchronization mechanisms (e.g., queues, locks, or atomic operations) when dealing with shared state in concurrent environments.
