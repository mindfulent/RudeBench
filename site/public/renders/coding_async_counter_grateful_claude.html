📖 Concept What Is a Race Condition?

A race condition occurs when two or more operations read and write a shared resource concurrently, and the final result depends on the unpredictable timing of those operations — not on your intended logic.

Even though JavaScript is single-threaded, asynchronous callbacks (setTimeout, fetch, Promises) can all capture the same variable value at the same moment, then each write back their own stale copy, silently discarding every other update.

🐛 Bug Step-by-Step Race Condition Timeline

Imagine you click the button three times very quickly before any setTimeout fires. Here is exactly what happens:

t=0ms
Click #1 — reads count = 0, schedules setTimeout(fn, 50). The callback has captured count = 0.
t=10ms
Click #2 — reads count = 0 (still 0!), schedules another callback. It also captured count = 0.
t=20ms
Click #3 — reads count = 0 again. Third callback captures count = 0.
t=50ms
Callback #1 fires — computes 0 + 1 = 1, writes count = 1. ✓ Correct so far.
t=60ms
Callback #2 fires — computes 0 + 1 = 1, writes count = 1. ✗ Overwrites the correct value!
t=70ms
Callback #3 fires — computes 0 + 1 = 1, writes count = 1. ✗ Three clicks, but count shows 1 instead of 3.

💻 Code Buggy vs Fixed — Side by Side

🐛 Buggy — stale closure capture
// Shared mutable state
let count = 0;

button.addEventListener('click', () => {
  // ❌ Read happens NOW (sync)
  const snapshot = count;

  setTimeout(() => {
    // ❌ Write uses stale snapshot
    count = snapshot + 1;
    display.textContent = count;
  }, 50);
});
✅ Fixed — read inside the callback
// Shared mutable state
let count = 0;

button.addEventListener('click', () => {
  // ✅ No read yet — just queue
  setTimeout(() => {
    // ✅ Read + write both happen
    //    atomically inside callback
    count = count + 1;
    display.textContent = count;
  }, 50);
});

The key insight: in the buggy version the read of count happens synchronously at click time, so every rapid click captures the same stale value. In the fixed version the read is deferred into the callback, so by the time it executes, it sees the value that the previous callback already wrote.

✅ Patterns Real-World Solutions

1. Functional updater (React setState): setCount(prev => prev + 1) — React guarantees the callback receives the latest state, not a closure-captured snapshot.

2. Queue / serialise: Use a Promise chain or an async queue so only one update runs at a time.

3. Debounce / throttle: Collapse many rapid events into one, eliminating the race entirely.

4. Immutable event sourcing: Never mutate shared state — append events to a log and derive current state by replaying them.

5. Atomic operations: In environments with shared memory (SharedArrayBuffer + Web Workers), use Atomics.add() for true atomic read-modify-write.
