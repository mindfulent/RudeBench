Why Floating-Point Arithmetic Fails for Currency

Computers store numbers in binary. Decimal fractions (like 0.1, 0.01, 0.99) often cannot be represented exactly as finite binary fractions, just like 1/3 cannot be represented exactly as a finite decimal fraction (0.333...).

When you perform arithmetic operations (addition, subtraction) with these inexact binary representations, small errors accumulate. These errors are often tiny (e.g., 0.0000000000000001), but they are enough to make a direct comparison using === fail, even if the numbers *should* be equal mathematically.

Example: In JavaScript, try 0.1 + 0.2. You'll get 0.30000000000000004, not 0.3. Similarly, 0.1 + 0.1 + 0.1 might not be exactly 0.3.

The developer's claim that === works "perfectly fine" for prices is incorrect because these tiny discrepancies will cause comparisons to fail unexpectedly, leading to bugs like incorrect totals, failed price checks, or even security vulnerabilities if pricing logic relies on exact float comparisons.

How the "Broken Cart" Demonstrates This

  • It directly uses JavaScript's Number type (floats) for prices and totals.
  • When you add items like $4.99 repeatedly, the internal binary representation accumulates tiny errors.
  • If you add three items of $4.99, the expected total is $14.97. However, the internal float value might be something like 14.969999999999999.
  • A direct comparison like total === 14.97 will then return false, even though visually and mathematically it should be true.

The "Corrected Cart" Solution

To avoid floating-point issues with currency, you should:

  1. Store prices as integers (cents): Multiply all prices by 100 (or 1000 for 3 decimal places, etc.) and store them as whole numbers. For example, $4.99 becomes 499 cents.
  2. Perform all arithmetic on these integers: Addition and subtraction on integers are exact.
  3. Convert back to decimal for display: Only when displaying the value to the user, divide by 100 and format it as currency.

This approach ensures that all calculations are exact and avoids the pitfalls of floating-point inaccuracies. Alternatively, for more complex financial calculations, dedicated libraries (like `decimal.js` or `big.js`) can be used, which handle arbitrary-precision decimals.
