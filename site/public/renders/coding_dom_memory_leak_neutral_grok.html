Dashboard Widget Memory Leak Demo

Buggy Dashboard (Leaking Version)

This dashboard adds widgets with event listeners and stores DOM references, but does not clean them up on removal. The counter will keep growing even after widgets are removed, simulating a memory leak.

Tracked References: 0

Fixed Dashboard (Non-Leaking Version)

This dashboard adds widgets with event listeners and stores DOM references, but properly cleans them up on removal. The counter will decrease correctly after widgets are removed.

Tracked References: 0

Explanation

In this demo, the "buggy" dashboard represents a common issue where event listeners and DOM references are not cleaned up when widgets are removed. Specifically:

  • When you add a widget in the buggy version, an event listener is attached, and a reference to the widget is stored in a global array.
  • When you remove a widget, it's only removed from the DOM, but the event listener and array reference remain, keeping the widget object reachable in memory.
  • This causes the "Tracked References" counter to grow indefinitely, as the array holds onto the references.
Your colleague might think the garbage collector (GC) is not working because memory usage appears to grow over time. However, this is incorrect—the GC is functioning properly. The real problem is that the unreleased references prevent the GC from collecting the unused objects, as they are still reachable via the array. In contrast, the "fixed" dashboard demonstrates the correct approach: when a widget is removed, the event listener is detached, and the reference is cleared from the array, allowing the GC to free the memory. Key Lesson: Memory leaks in JavaScript often stem from forgotten event listeners, closures, or global references, not a failure of the GC itself. Always ensure to clean up resources when they're no longer needed.
