What is the Singleton Pattern?

Ensures a class has only one instance and provides a global access point to it.

🔒

Single Instance

Only one ThemeManager object ever exists in memory. Every call to getInstance() returns the exact same object reference.

🌐

Global Access Point

Any module in your app can access the theme state without passing references through props or function arguments.

⚡

Lazy Initialization

The instance is created only when first requested, avoiding unnecessary memory allocation at startup.

📡

Observer Ready

Easily extended with the Observer pattern so components can subscribe to theme changes reactively.

Implementation

The complete Singleton ThemeManager used on this page.

JavaScript — ThemeManager.js 
class ThemeManager {
  // Private static field holds the single instance
  static #instance = null;

  constructor() {
    if (ThemeManager.#instance) {
      throw new Error("Use ThemeManager.getInstance()");
    }
    this.theme      = this._loadTheme();
    this.subscribers = [];
    this._apply();
  }

  /** Returns the single shared instance (creates it on first call) */
  static getInstance() {
    if (!ThemeManager.#instance) {
      ThemeManager.#instance = new ThemeManager();
    }
    return ThemeManager.#instance;
  }

  toggle() {
    this.theme = this.theme === "light" ? "dark" : "light";
    this._apply();
    this._save();
    this._notify();
  }

  subscribe(callback) {
    this.subscribers.push(callback);
    return () => {   // returns unsubscribe fn
      this.subscribers = this.subscribers
        .filter(cb => cb !== callback);
    };
  }

  _apply()  { document.body.dataset.theme = this.theme; }
  _save()   { localStorage.setItem("theme", this.theme); }
  _notify() { this.subscribers.forEach(cb => cb(this.theme)); }
  _loadTheme() {
    return localStorage.getItem("theme")
      ?? (window.matchMedia("(prefers-color-scheme: dark)").matches
          ? "dark" : "light");
  }
}

Quick Takeaways

✅

When Singleton Works Well

Use it for truly global, stateless-ish resources: logging, configuration, hardware interfaces, or — as shown here — a lightweight theme manager where only one authoritative state should ever exist.

⚠️

When Singleton Hurts

Avoid it for business logic services, data repositories, or anything you need to mock in tests. Hidden global state makes code harder to reason about and unit-test in isolation.

💡

A Pragmatic Middle Ground

ES6 modules are singletons by nature — the module cache ensures one instance. Exporting a plain object or a frozen config from a module gives you singleton semantics without a class, and it's far easier to mock in tests.

Theme-Aware Form

📚 In-Depth Discussion

The Singleton pattern is one of the most recognised — and most debated — patterns in software design. Here is an honest look at both sides, and guidance on when to reach for something better.

✅ Advantages

🎯
Controlled Instance Count

Guarantees exactly one instance exists, preventing accidental duplication of expensive resources like DB connection pools or hardware drivers.

💾
Lazy Initialisation

The object is created only when first needed, avoiding wasteful startup costs if the resource is never actually used.

🌍
Global Accessibility

Any part of the codebase can retrieve the instance without threading it through constructors or function arguments — convenient for cross-cutting concerns.

📦
Single Source of Truth

For genuinely global state (app config, feature flags, active theme) having one authoritative object eliminates synchronisation bugs.

❌ Disadvantages

🧪
Poor Testability

Global state persists between tests, causing hidden coupling. You cannot easily inject a mock or reset state, making unit tests fragile and order-dependent.

🔗
Hidden Dependencies

Callers implicitly depend on the global instance rather than declaring dependencies explicitly. This violates the Dependency Inversion Principle and obscures the call graph.

🧵
Concurrency Hazards

In multi-threaded environments (e.g., Node.js workers, Java) naive implementations suffer race conditions during initialisation without careful double-checked locking.

📐
Violates SRP

The class manages both its business logic and its own lifecycle, giving it two reasons to change and making it harder to extend without modification.

🔄 When Dependency Injection (or Alternatives) Are More Appropriate

🏭 Use DI When Testing Matters

If you need to swap implementations (real DB vs. in-memory mock), inject the dependency through the constructor. Your test can pass a fake without touching global state.

📦 Use ES6 Modules as Singletons

The module cache already ensures one instance. Export a plain frozen object or an instance created at module level — simpler, no class boilerplate, and still mockable via jest.mock().

⚛️ Use Context / Stores in UI Frameworks

React Context, Vue's provide/inject, or Pinia/Redux give you scoped global state with first-class tooling, devtools, and time-travel debugging that a hand-rolled Singleton can't match.

🏗️ Use DI Containers for Large Apps

Frameworks like InversifyJS, NestJS, or Angular's injector manage object lifecycles (singleton, transient, scoped) declaratively, giving you Singleton semantics without the anti-patterns.

🎭 Use the Monostate Pattern

Instead of restricting instantiation, make all instance fields static so every instance shares state. Callers can create objects normally, preserving testability while keeping shared state.

🔒 Keep It When Truly Justified

Hardware abstractions, loggers, feature-flag clients, and — as here — a lightweight theme manager are legitimate Singleton use cases where the simplicity genuinely outweighs the trade-offs.

Bottom line: The Singleton pattern is a tool, not a rule. Ask yourself: "Would this be hard to test?" and "Does a consumer need to know about this dependency?" If either answer is yes, reach for dependency injection or a framework-managed store instead. If the resource is genuinely global, stateless, and never needs mocking, a Singleton (or better yet, a simple ES6 module export) is perfectly reasonable.
