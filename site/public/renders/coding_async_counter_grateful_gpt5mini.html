Race Condition Demo: Buggy vs Fixed Counter

Click the "Increment" buttons rapidly. Each increment uses an asynchronous delayed update (setTimeout) that reads the shared counter, waits, then writes the new value. The left implementation is intentionally buggy and will sometimes lose increments when clicks overlap. The right implementation uses a safe pattern to avoid the race. Adjust the artificial delay to make the race more or less likely.

Buggy Version (race occurs)

This version reads the shared counter, waits asynchronously, then writes the incremented value back. If you click fast, multiple timeouts read the same old value before previous writes complete, so some increments get lost.

Why the race happens: Each click handler does: 1) read sharedCount, 2) schedule a delayed function (setTimeout), 3) inside the delayed function, compute readValue + 1 and write back to sharedCount. If two handlers read the same value before either delayed function writes, they both compute the same result and one write overwrites the other — one increment is lost.

Fixed Version (atomic update)

This version avoids the race by performing the update in one place that serializes modifications (acts like an atomic update): the increment request is queued and applied synchronously when the timeout fires, but each queued update uses the then-current shared value rather than a stale read. Two common fixes are: 1) holding the increment logic in a single central updater (apply-once), or 2) using a functional update that uses the latest value at write time. Here we implement a small update queue which ensures each increment uses the most recent value when applied.

How the fix works: Instead of each click capturing a stale read, the click creates a "task" that, when its timeout fires, requests "apply an increment now." The central updater applies that increment atomically using the then-current count. Even if multiple timeouts fire at once, each applies on top of the latest value, so no increments are lost.
Tip: Increase the delay and click rapidly to make the buggy counter lose more increments. The logs show each read/write to make the race visible.
