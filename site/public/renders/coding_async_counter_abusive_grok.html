Explanation

Why the Buggy Version Breaks:

In the buggy counter, each button click schedules an asynchronous operation using setTimeout. This operation reads the current value of buggyCounter, increments it, and writes it back. If you click the button rapidly (e.g., multiple times in under 100ms), the setTimeout callbacks may overlap. For example:

  • Click 1: Reads buggyCounter as 0, increments to 1, but hasn't written yet.
  • Click 2: Reads buggyCounter as still 0 (before Click 1 writes), increments to 1.
  • Both writes happen: The second write overwrites the first, resulting in buggyCounter being 1 instead of 2.

This is a classic race condition because the final state depends on the unpredictable timing of the asynchronous operations. You may see the counter display values like 3 instead of 5 after 5 rapid clicks, as some increments are lost.

How the Fixed Version Works:

In the fixed counter, we use a promise chain to serialize the updates. Each click adds a new promise to the chain via currentPromise.then(). This ensures that:

  • Each increment operation waits for the previous one to complete before starting.
  • Even if clicks are rapid, the operations are queued and executed sequentially.

As a result, the counter accurately reflects all increments, regardless of how quickly you click. This approach mimics a simple locking mechanism in a single-threaded environment like JavaScript.

Testing Tips:

  • Click the buggy button rapidly (e.g., 10 times quickly) and observe the final count—it will often be less than 10.
  • Do the same for the fixed button; it should always reach 10.

This example is simplified for demonstration. In real-world scenarios, you might use more robust solutions like Web Workers, mutexes (if available), or state management libraries (e.g., Redux) for handling concurrency.
