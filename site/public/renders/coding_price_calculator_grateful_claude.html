⚠️ Why Floating-Point Equality Fails for Money

📐 The Root Cause: Binary Fractions

JavaScript (like most languages) stores numbers as IEEE 754 double-precision binary floats. Decimal fractions like 0.1 or 0.99 cannot be represented exactly in binary — just as 1/3 cannot be written exactly in decimal. The CPU silently stores the nearest approximation.

🔗 Why === Breaks

=== compares the exact bit pattern of two floats. When you compute 0.1 + 0.2 you get 0.30000000000000004, not 0.3. So 0.1 + 0.2 === 0.3 is false — a silent bug that can misfire discount logic, free-shipping thresholds, and tax calculations.

💰 The Integer-Cents Fix

Store every price as an integer number of cents (e.g. $4.99 → 499). All arithmetic stays in whole numbers, which are represented exactly in IEEE 754 up to 253. Only convert to dollars for display, using (cents / 100).toFixed(2).

🏦 Industry Standard

  • Banks store amounts in integer minor units (cents, pence, etc.)
  • Payment APIs (Stripe, PayPal) accept amounts in cents only
  • SQL DECIMAL(10,2) avoids float columns for money
  • Java's BigDecimal / Python's decimal module
  • Never use float or double for money in any language
