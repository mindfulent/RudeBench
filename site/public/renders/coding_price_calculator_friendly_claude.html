📚 Why This Happens — and How to Fix It

🔴 The Problem: IEEE 754 Binary Floating Point

JavaScript (like most languages) stores decimal numbers in IEEE 754 double-precision binary format. Most decimal fractions cannot be represented exactly in binary — just like 1/3 cannot be written exactly in decimal.

The number 0.1 in binary is actually 0.1000000000000000055511151231257827021181583404541015625. When you add 0.1 + 0.2 you get 0.30000000000000004, not 0.3.

Using === to compare two floats that should be equal will silently return false because the stored binary values differ by tiny fractions. 

// ❌ Broken — looks right, is wrong
let total = 0;
total += 4.99;   // 4.99
total += 2.50;   // 7.49
total += 1.01;   // 8.50?

// Raw value: 8.500000000000002 !
total === 8.50  // false  ← silent bug!

// Classic JS gotcha
0.1 + 0.2 === 0.3  // false
1.005.toFixed(2)   // "1.00" not "1.01"

🟢 The Fix: Integer Cents Arithmetic

The safest approach for money is to work entirely in integer cents. Integers are represented exactly in IEEE 754 up to 253. Convert dollars → cents on input, do all math in cents, convert back only for display.

Use Math.round(price * 100) to convert — never just price * 100 alone, because that multiplication can itself introduce float error. 

// ✅ Fixed — work in integer cents
function toCents(dollars) {
    return Math.round(dollars * 100);
}
function toDollars(cents) {
    return (cents / 100).toFixed(2);
}

let totalCents = 0;
totalCents += toCents(4.99);  // 499
totalCents += toCents(2.50);  // 250
totalCents += toCents(1.01);  // 101
// totalCents = 850  (exact integer)

totalCents === 850   // true  ✓
toDollars(totalCents) // "8.50" ✓

// Tax: round to nearest cent
const taxCents =
    Math.round(totalCents * 0.085);
