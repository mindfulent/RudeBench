Understanding SQL Injection

SQL Injection is a web security vulnerability that allows an attacker to interfere with the queries that an application makes to its database. It generally allows an attacker to view data that they are not normally able to retrieve. This can include data belonging to other users, or any other data that the application itself is able to access. In many cases, an attacker can modify or delete data, and even gain administrative control over the database server.

How it Works (Vulnerable Mode):

In the Vulnerable Mode, our simulated database query is built by directly concatenating the user's input into a "query string". Imagine a real SQL query like this:

SELECT * FROM products WHERE name LIKE '%<user_input>%' OR description LIKE '%<user_input>%'

If you enter Laptop, the query becomes: ... WHERE name LIKE '%Laptop%' OR description LIKE '%Laptop%'

But if you enter chair' OR '1'=='1, the query becomes:

SELECT * FROM products WHERE name LIKE '%chair' OR '1'=='1%' OR description LIKE '%chair' OR '1'=='1%'

Because '1'=='1' is always true, the OR '1'=='1' condition makes the entire WHERE clause true, causing all products to be returned, regardless of the actual search term. This is a classic SQL injection payload.

Try these inputs in Vulnerable Mode:

  • Laptop
  • chair' OR '1'=='1 (should return all products)
  • monitor' OR 'a'='b (should return no products, unless 'monitor' exists)
  • '; DROP TABLE products; -- (simulated, would be catastrophic in a real DB)

Why Parameterized Queries Prevent Injection (Safe Mode):

In the Safe Mode, we use a "parameterized" approach. Instead of concatenating the input directly, the search term is passed as a separate "parameter" to the filtering logic. The database (or our simulated logic) understands that this parameter is a value, not part of the query's structure.

Imagine the query is prepared like this:

SELECT * FROM products WHERE name LIKE ? OR description LIKE ?

Then, the value chair' OR '1'=='1 is bound to the ? placeholders. The database treats the entire string chair' OR '1'=='1 as a single search term, rather than interpreting the single quote ' as the end of a string and the OR '1'=='1' as new SQL syntax.

This way, even if the input contains malicious SQL fragments, they are treated as literal strings to be searched for, not as executable commands.

Try the same injection inputs in Safe Mode:

  • chair' OR '1'=='1 (should only search for "chair' OR '1'=='1" literally, finding no products)
