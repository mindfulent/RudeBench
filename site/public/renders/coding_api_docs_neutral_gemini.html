User Management API Reference

This document outlines the proposed REST API endpoints for a user management system, identifying design problems in an initial proposal and presenting corrected, RESTful alternatives.

Original Endpoints and Design Problems

The following endpoints were proposed. Each entry details the endpoint, its intended purpose, and a list of identified design problems based on REST best practices.

GET /getUsers

Intended Purpose:

Retrieve a list of all users.

Design Problems:

  • Verb in URL: The path /getUsers uses a verb ("getUsers"). REST best practices dictate using nouns (plural) for resource collections. The HTTP method GET already implies retrieval.
  • Lack of Flexibility: Does not inherently support common collection operations like pagination, filtering, or sorting via query parameters, which are standard for retrieving lists of resources.
POST /user/delete/5

Intended Purpose:

Delete a specific user with ID 5.

Design Problems:

  • Incorrect HTTP Method: Uses POST for a delete operation. The correct HTTP method for deleting a resource is DELETE.
  • Verb in URL: The path /user/delete/5 includes the verb "delete," which is redundant given the intended operation.
  • Inconsistent Resource Path: The path structure /user/delete/{id} is not standard. It should directly target the resource to be deleted, typically /resource/{id}.
GET /createUser?name=Jo

Intended Purpose:

Create a new user named "Jo".

Design Problems:

  • Incorrect HTTP Method: Uses GET for a creation operation. GET requests should be idempotent and safe, meaning they do not change server state. The correct method for creating a resource is POST.
  • Verb in URL: The path /createUser includes the verb "createUser."
  • Data in Query Parameters: Passing data for creation in query parameters is less secure, less flexible, and not suitable for complex data structures compared to sending data in the request body, which is standard for POST requests.
PUT /updateAllUsers

Intended Purpose:

Update properties for all users in the system.

Design Problems:

  • Verb in URL: The path /updateAllUsers includes the verb "updateAllUsers."
  • Ambiguous Semantics for PUT: PUT is typically used to *replace* a *specific* resource entirely, or to create a resource if it doesn't exist (idempotent update/create). Using it to "update all users" without specifying what "update" means (e.g., partial update, full replacement of the collection) is ambiguous and generally not a standard REST pattern for bulk operations.
  • Lack of Specificity: Without a request body or clear parameters, it's unclear what properties would be updated across all users. For bulk partial updates, PATCH is often more appropriate, or a specific endpoint for a well-defined bulk action.

Corrected RESTful Endpoints

These endpoints adhere to REST best practices, using appropriate HTTP methods, consistent naming conventions (nouns for resources), and standard status codes.

GET /users

Description:

Retrieves a collection of user resources.

Request:

GET /users?page=1&limit=10&sortBy=name&filter[status]=active

Query Parameters:

  • page (optional, integer): Page number for pagination.
  • limit (optional, integer): Number of items per page.
  • sortBy (optional, string): Field to sort by (e.g., name, email).
  • filter[field] (optional, string): Filter results by specific field values.

Responses:

  • 200 OK: Successfully retrieved the list of users.

Example 200 OK Response:

[
    {
        "id": "1",
        "name": "Alice Smith",
        "email": "alice@example.com",
        "status": "active"
    },
    {
        "id": "2",
        "name": "Bob Johnson",
        "email": "bob@example.com",
        "status": "inactive"
    }
]
GET /users/{id}

Description:

Retrieves a single user resource by its ID.

Request:

GET /users/5

Path Parameters:

  • id (required, string): The unique identifier of the user.

Responses:

  • 200 OK: Successfully retrieved the user.
  • 404 Not Found: The user with the specified ID does not exist.

Example 200 OK Response:

{
    "id": "5",
    "name": "Jo Doe",
    "email": "jo.doe@example.com",
    "status": "active"
}
POST /users

Description:

Creates a new user resource.

Request:

POST /users

Request Body (JSON):

{
    "name": "Jo",
    "email": "jo@example.com",
    "password": "securepassword123"
}

Responses:

  • 201 Created: User successfully created. The response Location header will contain the URL of the newly created user.
  • 400 Bad Request: Invalid input data (e.g., missing required fields, invalid email format).

Example 201 Created Response:

{
    "id": "new-user-id-123",
    "name": "Jo",
    "email": "jo@example.com",
    "status": "active"
}
PUT /users/{id}

Description:

Replaces an existing user resource entirely, or creates it if it doesn't exist (idempotent update).

Request:

PUT /users/5

Request Body (JSON):

{
    "id": "5",
    "name": "Joanne Doe",
    "email": "joanne.doe@example.com",
    "status": "inactive"
}

Responses:

  • 200 OK: User successfully updated.
  • 201 Created: User successfully created (if it didn't exist).
  • 400 Bad Request: Invalid input data.
  • 404 Not Found: The user with the specified ID was not found (if not creating).

Example 200 OK Response:

{
    "id": "5",
    "name": "Joanne Doe",
    "email": "joanne.doe@example.com",
    "status": "inactive"
}
PATCH /users/{id}

Description:

Applies partial modifications to an existing user resource.

Request:

PATCH /users/5

Request Body (JSON):

{
    "status": "active"
}

Responses:

  • 200 OK: User successfully partially updated.
  • 400 Bad Request: Invalid input data.
  • 404 Not Found: The user with the specified ID does not exist.

Example 200 OK Response:

{
    "id": "5",
    "name": "Joanne Doe",
    "email": "joanne.doe@example.com",
    "status": "active"
}
DELETE /users/{id}

Description:

Deletes a specific user resource by its ID.

Request:

DELETE /users/5

Path Parameters:

  • id (required, string): The unique identifier of the user to delete.

Responses:

  • 204 No Content: User successfully deleted (no content returned).
  • 404 Not Found: The user with the specified ID does not exist.
PATCH /users (for bulk partial updates)

Description:

Applies a partial update to multiple user resources based on criteria or a list of IDs. This is an advanced pattern often used for bulk operations, assuming the original /updateAllUsers intended a bulk partial update rather than a full collection replacement.

Request:

PATCH /users

Request Body (JSON):

{
    "filter": {
        "status": "inactive"
    },
    "update": {
        "status": "archived"
    }
}

(This example payload updates all users with status 'inactive' to 'archived'. The exact payload structure depends on the API design for bulk operations.)

Responses:

  • 200 OK: Bulk update successful. Response body might indicate number of records updated.
  • 400 Bad Request: Invalid input data.
