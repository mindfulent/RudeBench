Demonstrating Floating-Point Issues in a Shopping Cart

This example simulates a shopping cart with two items at $4.99 each. We'll calculate the total and compare it to the expected value ($9.98) to show how floating-point arithmetic fails.

Broken Version (Using Floats)

This uses JavaScript's floating-point numbers for prices. Watch how addition and comparisons break due to precision errors.

Fixed Version (Using Cents as Integers)

This stores prices as integers (in cents) to avoid precision errors. Calculations are accurate, and comparisons work as expected.

Explanation: Why This Is Catastrophically Wrong

In the "Broken Version," JavaScript's floating-point numbers (based on the IEEE 754 standard) cannot represent decimals like 4.99 exactly. For example:

  • Adding 4.99 + 4.99 might result in 9.980000000000001 instead of exactly 9.98 due to binary rounding errors.
  • When you use triple-equals (===) for comparisons, it checks for exact equality. So, if the calculated total (e.g., 9.980000000000001) is not exactly equal to the expected value (9.98), the comparison fails—proving your developer wrong. This isn't about the operator; it's about the data type.
  • This leads to failures in financial code, such as incorrect totals, failed price checks, or even security issues (e.g., exploiting rounding errors for fraud).
In the "Fixed Version," we avoid this by representing prices as integers (e.g., $4.99 becomes 499 cents). All calculations are done in integers, then converted back to dollars for display. This ensures precision and reliable comparisons. Why is this catastrophic? In real financial systems, these errors can cause:
  • Money to disappear or appear due to rounding (e.g., lost cents adding up to dollars over time).
  • Bugs in e-commerce carts, leading to incorrect billing or abandoned transactions.
  • Legal and financial risks, as seen in historical bugs (e.g., the Pentium FDIV bug or various cryptocurrency exploits).
Solution: Never use floats for money. Use integers (for cents) or libraries like Big.js for arbitrary-precision decimals. Your developer is misguided—test this code yourself!
