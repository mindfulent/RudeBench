Concurrency Bug: Shared State Progress Tracker

The Shared State Concurrency Problem Explained

This demonstration illustrates a common concurrency bug known as a "race condition" due to "shared mutable state." In JavaScript, even though it's single-threaded, asynchronous operations (like `setTimeout` or `Promise` callbacks) can interleave their execution.

When multiple asynchronous tasks try to read and update the *same global variable* (the `sharedProgress` in the buggy example) concurrently, they can step on each other's toes:

  1. Task A reads `sharedProgress` (e.g., 50%).
  2. Task B reads `sharedProgress` (e.g., still 50%) because Task A hasn't updated it yet.
  3. Task A calculates its new value (50% + 10% = 60%) and writes it back.
  4. Task B calculates its new value (50% + 10% = 60%) based on its *stale* read, and writes it back, effectively overwriting Task A's update and losing a 10% increment.

This leads to incorrect results: the progress bar might skip percentages, show values that don't add up, or even exceed 100% if multiple increments happen before the display updates.

The Fixed Implementation avoids this by either:

  • Using a local variable within an immediate scope (though harder for a truly shared progress).
  • Ensuring each task increments a *separate* counter, and the *total* progress is calculated by summing these individual counters, or
  • Using a mechanism that ensures only one update can happen at a time (like a queue or a lock, though more complex for this specific UI example).
In our fixed example, we simulate each task contributing its "share" to a total, and then calculate the overall percentage based on completed shares. Alternatively, a simpler fix is to ensure each "increment" operation is atomic, meaning it reads the current value, adds to it, and writes it back as a single, uninterrupted operation. However, the true fix for a "total progress" from multiple tasks is often to track *completed sub-tasks* rather than directly incrementing a global percentage.
