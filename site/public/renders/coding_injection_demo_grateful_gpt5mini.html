Why parameterized approaches prevent injection (plain language)

When an application "builds" a query by inserting raw user text directly into the query string, that user text can be misinterpreted as part of the query structure rather than as pure data. Parameterized queries or prepared statements solve this by separating the query's structure from the values. The code sends a fixed query template (with placeholders) and a list of values; the database treats those values strictly as data, preventing them from changing the query's meaning.

  • Vulnerable-style (illustration): shows how a query-like string might be constructed as plain text with the user's input embedded. This is where problems happen if an attacker can craft input that alters structure.
  • Safe (parameterized-style): treats the same input only as a parameter; the filtering logic applies the input as data and never mixes it into a query structure.

Important: In real servers use parameterized queries, ORM query builders, input validation, escaping, least privilege, and other best practices. This demo is illustrative only and does not perform or teach SQL exploitation.
