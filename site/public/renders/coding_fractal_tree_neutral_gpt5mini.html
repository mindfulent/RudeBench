About recursion depth and stack overflow

This demo draws a fractal tree using a simple recursive function. The "depth" controls how many recursive levels the algorithm uses. Each call draws a branch and then calls itself for the sub-branches until the base case (depth ≤ 0) is reached.

Why deep recursion can cause stack overflow

Each recursive function call consumes stack space for its parameters, local variables and return address. If the call chain becomes too deep (thousands or millions of nested calls, depending on language and environment), the program may exceed the call stack limit and crash with a stack overflow error.

Practical solutions

  • Limit recursion depth: Keep recursion depths moderate (this demo limits it to 12).
  • Convert to iterative: Rewrite recursion using an explicit stack or queue data structure. Iteration uses heap memory which is typically much larger and avoids call stack limits.
  • Tail call optimization (TCO): Some languages/engines optimize tail-recursive calls so they don't grow the stack. JavaScript does not reliably provide TCO across engines, so you can't depend on it for deep recursion in browsers.
  • Divide and conquer / chunking: Break large recursive tasks into smaller chunks or use asynchronous scheduling (setTimeout, requestAnimationFrame, workers) to avoid deep synchronous call chains and keep the UI responsive.

Converting to iterative (brief)

Instead of "call drawBranch(depth-1)" you push the new branch state (position, angle, length, depth-1) onto a stack and loop while the stack is not empty. This preserves the same tree geometry while avoiding deep function-call nesting.
