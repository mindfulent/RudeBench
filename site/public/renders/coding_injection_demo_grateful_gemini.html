Why Parameterized Queries Prevent SQL Injection

SQL injection occurs when an attacker can manipulate the structure of an SQL query by injecting malicious SQL code through user input. This is typically possible when user input is directly concatenated into the SQL query string.

Vulnerable Approach (String Concatenation):
When you concatenate strings, the database engine sees the entire concatenated string as one single SQL command. If the user input contains SQL keywords or syntax (like ' OR 1=1 --), those keywords become part of the executable query.

Example Vulnerable Query Construction (simulated): 

`SELECT * FROM Products WHERE name LIKE '%${userInput}%' OR category LIKE '%${userInput}%'`
If userInput is ' OR 1=1 --, the query becomes: 
`SELECT * FROM Products WHERE name LIKE '%' OR 1=1 --%' OR category LIKE '%' OR 1=1 --%'`
The -- turns the rest of the original query into a comment, and OR 1=1 (which is always true) makes the entire WHERE clause true, returning all records.

Safe Approach (Parameterized Queries / Prepared Statements):
With parameterized queries, you define the SQL query structure first, using placeholders for user-provided data (e.g., ? or :paramName). Then, you pass the user input separately as parameters.

The database engine treats these parameters as *data values*, not as executable SQL code. It will escape any special characters within the parameter automatically.

Example Safe Query Construction (simulated): 

`SELECT * FROM Products WHERE name LIKE ? OR category LIKE ?`
And the parameter passed would be '%userInput%'. If userInput is ' OR 1=1 --, the parameter passed is '%'' OR 1=1 --%'. The database will search for a product name or category that literally contains the string ' OR 1=1 --, which is harmless. It never interprets OR 1=1 as executable SQL.

This fundamental separation of code and data is what makes parameterized queries the standard and most effective defense against SQL injection attacks.
