Why ≠ works: JavaScript Number uses IEEE-754 double precision floating point (binary base). Many decimal fractions (0.1, 0.01, 4.99) cannot be represented exactly in binary, so arithmetic introduces tiny errors. When you compare floats exactly (===), those tiny differences cause unexpected false. For money you should use integers (cents), a decimal library, or BigInt/Decimal types designed for exact decimals.

Short technical explanation

JavaScript Number uses IEEE-754 double precision binary floating point. Binary floating point cannot exactly represent most decimal fractions (for example 0.1, 0.01, 4.99). When you do arithmetic, tiny rounding errors accumulate, producing results like 14.969999999999999 instead of 14.97. A strict equality comparison (===) compares these exact binary values, so it will say they are different.

Safer alternatives for financial amounts: - Represent amounts in the smallest unit (cents) as integers and do all math with integers, converting to dollars only for display. This is what the "Fixed" column demonstrates. - Use a decimal type / arbitrary-precision decimal library (e.g., decimal.js, big.js) or native Decimal if available. - Use integers with BigInt for very large values or when necessary.

Important: rounding policies (round half up, bankers rounding, etc.) and currency subunits matter for taxes, discounts and display. Pick a consistent approach for your domain.
