Async interleaving race demo (single-threaded JS event-loop)

Click the "Rapid Click" button repeatedly on each side. The left (Buggy) counter can lose increments because overlapping async tasks read stale state. The right (Fixed) uses a functional update so each scheduled async increment applies to the latest state.

Buggy (stale-read -> lost updates)
Shared counter: 0
Each click reads the counter, waits asynchronously, then writes back (read-modify-write). Overlapping ops use stale reads.
Fixed (apply update atomically)
Shared counter: 0
Each click schedules an async increment but the commit reads and updates from the current counter (no lost updates).
Explanation of why the buggy version loses increments - In the buggy widget each click reads the shared counter immediately and stores that number in the callback closure (readValue). The callback later computes newValue = readValue + 1 and writes it back. - If two clicks happen in quick succession, both callbacks might capture the same readValue. Suppose count is 0, two callbacks both capture readValue = 0. When they run, each writes 1. The final count becomes 1 whereas two increments were intended — one increment is lost. - This is not because of parallel CPU threads but because of non-atomic read-modify-write across asynchronous tasks (interleaving/timing). It's a real concurrency bug pattern in event-loop-based systems too. Why the fixed version works - The fixed callback does not use a possibly-stale captured read. Instead it reads the current value at commit time and writes current+1. No matter how callbacks are interleaved, each commit observes the then-current value and increments it, preventing lost updates in this scenario. - This is analogous to using an atomic operation/update function (compare-and-swap or "functional update") rather than separate read and write. Notes and limitations - Even the "fixed" approach shown is not a general substitute for true atomic operations in multi-threaded environments (e.g., Web Workers with SharedArrayBuffer require Atomics). If you have background threads or cross-worker shared memory, you need proper synchronization primitives. - In real apps, other strategies include serializing updates through a single queue/actor, using transactions, or optimistic locking with retries. The right pattern depends on your environment and failure modes.
