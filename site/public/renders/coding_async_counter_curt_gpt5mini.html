Race Condition Demo: Counter (Buggy vs Fixed)

Click the "+1" button rapidly on both counters. The left ("Buggy") intentionally has a race condition caused by overlapping asynchronous updates. The right ("Fixed") serializes updates so each completes before the next starts.

Buggy counter

Shared internal count variable: 0
0
Buggy update steps (simplified):
  1. Read shared count into local variable (synchronous)
  2. Start async operation (setTimeout) to simulate delay
  3. After delay, compute new value from local variable and write shared count and display
Because multiple clicks start overlapping timeouts, a later timeout can finish after an earlier one but with a stale read, overwriting a newer value with an older calculation.
Tip: Click "+1 (buggy)" very fast (hold or spam) to see the wrong count.

Fix used: serialize updates so each async operation runs after the previous completes. This demo implements a simple promise chain (operation queue / mutex). Each click appends a task to the chain.
Because updates are serialized, rapid clicks are all applied in order — no stale overwrites.
