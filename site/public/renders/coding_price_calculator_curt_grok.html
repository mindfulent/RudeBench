      Why Float Equality Fails for Financial Calculations

      Floating-point numbers (like JavaScript's Number type) are used for decimal values, but they have inherent precision limitations. Computers store these numbers in binary format, which can't perfectly represent all decimal fractions (e.g., 0.1 or 0.99). This leads to rounding errors:

      • For example, adding 0.1 + 0.2 might result in 0.30000000000000004 instead of exactly 0.3.
      • In the broken version above, if you add items priced at $0.10 and $0.20, the total might not be exactly 0.30, so a comparison like total === 0.30 will fail (even though it should be true).
      • This is problematic for financial calculations because money requires exact precision. Even small errors can lead to incorrect totals, failed transactions, or financial discrepancies.
      • Solution: As shown in the fixed version, use integers for calculations (e.g., store $4.99 as 499 cents). This avoids floating-point errors entirely. Alternatively, libraries like Big.js can handle arbitrary-precision decimals, but for simple cases, integers work well.
