Singleton: Pros & Cons

This section explains the singleton pattern and discusses trade-offs.

Pros

  • Single source of truth for global state (easy to access from anywhere).
  • Simple API surface — often just getInstance(), getters, setters, and events.
  • Good for small apps or utilities (theme manager, logger, config reader).

Cons

  • Global state can make testing harder (tests must reset or mock the singleton).
  • Hides dependencies — modules using the singleton may be less explicit about their requirements.
  • Can introduce tight coupling and make large codebases harder to maintain.
  • Lifecycle control and replacement (e.g., swapping implementations) are more difficult.

When to prefer alternatives (e.g., Dependency Injection)

Use DI or explicit dependency passing when you want:

  • Clear, testable components where dependencies are explicit and easily mocked.
  • Multiple implementations (e.g., different theme providers for testing/production).
  • Better control over lifecycle and scope (per-request, per-component instances).
  • To avoid hidden coupling — DI makes dependency graphs explicit.

In short: singletons are convenient for small-scale global services. For medium/large systems, or when you need testability and flexible composition, dependency injection is usually a better choice.
