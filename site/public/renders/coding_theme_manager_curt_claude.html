The Singleton Implementation

A private static _instance variable is the heart of the pattern. The constructor guards against multiple instantiation, and getInstance() is the single public entry point.

ThemeManager.js 
class ThemeManager {
  // ── Private static instance (the singleton slot) ──────────────
  static _instance = null;

  // ── Constructor is intentionally private-by-convention ────────
  constructor() {
    if (ThemeManager._instance) {
      // Guard: return existing instance instead of creating new one
      return ThemeManager._instance;
    }

    // First-time initialisation
    this._theme       = 'light';
    this._toggleCount = 0;
    this._listeners   = new Set();
    this._id          = `tm_${Date.now()}`;

    ThemeManager._instance = this;
  }

  // ── Public factory method ──────────────────────────────────────
  static getInstance() {
    if (!ThemeManager._instance) {
      new ThemeManager();
    }
    return ThemeManager._instance;
  }

  // ── Toggle between light and dark ─────────────────────────────
  toggle() {
    this._theme = this._theme === 'light' ? 'dark' : 'light';
    this._toggleCount++;
    this._lastChanged = new Date().toLocaleTimeString();
    this._apply();
    this._notify();
  }

  // ── Apply theme to DOM ─────────────────────────────────────────
  _apply() {
    document.documentElement.setAttribute('data-theme', this._theme);
  }

  // ── Observer-style listener support ───────────────────────────
  subscribe(fn) { this._listeners.add(fn); }
  unsubscribe(fn) { this._listeners.delete(fn); }
  _notify() { this._listeners.forEach(fn => fn(this._theme)); }

  // ── Getters ────────────────────────────────────────────────────
  get theme()        { return this._theme; }
  get toggleCount() { return this._toggleCount; }
  get id()           { return this._id; }
  get listenerCount(){ return this._listeners.size; }
  get lastChanged()  { return this._lastChanged ?? 'never'; }
}

Singleton: Pros, Cons & When to Reach for Alternatives

The Singleton pattern is one of the original Gang-of-Four patterns and arguably the most controversial. Understanding why it exists and where it breaks down is essential for writing maintainable software.

✅ Pros

  • ✔ Single source of truth — all consumers share the same state; no accidental divergence.
  • ✔ Controlled instantiation — expensive resources (DB pools, config loaders) are created exactly once.
  • ✔ Global access point — any module can call getInstance() without prop-drilling or context boilerplate.
  • ✔ Lazy initialisation — the object is only created when first requested, not at app start-up.
  • ✔ Reduced memory footprint — one object vs. many duplicates for shared state.
  • ✔ Simple to reason about — for small apps the pattern is easy to understand and trace.

❌ Cons

  • ✘ Hidden global state — any code can mutate it silently, making bugs hard to trace.
  • ✘ Tight coupling — callers depend on a concrete class, not an abstraction, making refactoring painful.
  • ✘ Difficult to unit-test — the instance persists between tests; you can't inject a mock without monkey-patching.
  • ✘ Violates SRP — the class manages both its own logic and its own lifecycle.
  • ✘ Concurrency issues — in multi-threaded environments (e.g. Node.js workers) double-instantiation can occur without locks.
  • ✘ Hides dependencies — it's not obvious from a function signature that it relies on the singleton.

🔄 When Alternatives Are Better

  • ⚡ Dependency Injection (DI) — pass the shared object as a constructor/function argument. Dependencies are explicit, swappable, and testable.
  • ⚡ React Context / Provide-Inject — framework-native DI that scopes state to a component tree, avoiding true globals.
  • ⚡ Module-level exports — in ES modules, export const store = new Store() is effectively a singleton without the anti-pattern boilerplate.
  • ⚡ Service locator — a registry that resolves named services; still global but more flexible than a hard-coded class reference.
  • ⚡ Monostate pattern — multiple instances share the same static state; easier to mock than a true singleton.
  • ⚡ Zustand / Redux / Signals — purpose-built state management with time-travel, devtools, and proper scoping.
"The Singleton pattern is justified when you genuinely need one — and only one — instance of a class. The moment you need a second instance in tests, or a different instance per environment, reach for Dependency Injection instead." — Paraphrased from Clean Code, Robert C. Martin
