Summary of REST Principles and Recommendations

Key problems in the original design:

  • Inconsistent naming: Endpoints mix singular/plural forms and include verbs. Recommendation: Use consistent plurals for collections (e.g., /users) and include IDs for specific resources (e.g., /users/5).
  • Wrong HTTP methods: Methods don't match actions (e.g., GET for creation). Recommendation: Strictly follow HTTP semantics—GET for reading, POST for creating, PUT/PATCH for updating, DELETE for removing.
  • Verbs in paths: Endpoints like /getUsers and /createUser use actions. Recommendation: Keep paths as nouns only.
  • Improper data handling: Using query parameters for actions. Recommendation: Use request bodies for POST/PUT/PATCH requests.
  • Bulk operations: Endpoints like /updateAllUsers are risky. Recommendation: Handle operations at the resource level; if bulk is needed, design carefully with safeguards.

Corrected design ensures:

  • Idempotent methods where appropriate.
  • Hierarchical and consistent URLs.
  • No side effects on safe methods like GET.
