Theme Management
with the Singleton Pattern

The Singleton Implementation

The Singleton pattern ensures a class has exactly one instance and provides a global access point to it. Below is the exact code powering this page's theme system, with every line explained.

JavaScript — ThemeManager Singleton
/**
 * ThemeManager — Singleton
 *
 * Responsibilities:
 *   1. Hold the single source of truth for the current theme.
 *   2. Persist the user's choice to localStorage.
 *   3. Apply the theme to the DOM via a data-attribute.
 *   4. Notify any registered listeners of changes.
 */
class ThemeManager {

  // ── Private static slot for the one instance ──────────────
  static #instance = null;

  // ── Private state ─────────────────────────────────────────
  #theme       = 'light';
  #listeners   = new Set();
  #toggleCount = 0;

  // ── Private constructor — called ONCE ─────────────────────
  constructor() {
    if (ThemeManager.#instance) {
      // Guard: throw instead of silently returning the old one.
      throw new Error('Use ThemeManager.getInstance()');
    }
    this.#init();
  }

  // ── The global access point ───────────────────────────────
  static getInstance() {
    if (!ThemeManager.#instance) {
      ThemeManager.#instance = new ThemeManager();
    }
    return ThemeManager.#instance;
  }

  // ── Initialise: respect OS preference & saved choice ──────
  #init() {
    const saved  = localStorage.getItem('theme');
    const prefersDark =
      window.matchMedia('(prefers-color-scheme: dark)').matches;

    this.#theme = saved ?? (prefersDark ? 'dark' : 'light');
    this.#apply();
  }

  // ── Write theme to the DOM ────────────────────────────────
  #apply() {
    document.documentElement
      .setAttribute('data-theme', this.#theme);
    localStorage.setItem('theme', this.#theme);
    this.#notify();
  }

  // ── Public API ────────────────────────────────────────────
  toggle() {
    this.#theme = this.#theme === 'light' ? 'dark' : 'light';
    this.#toggleCount++;
    this.#apply();
  }

  setTheme(name) {
    if (!['light', 'dark'].includes(name)) return;
    this.#theme = name;
    this.#apply();
  }

  get theme()       { return this.#theme; }
  get isDark()      { return this.#theme === 'dark'; }
  get toggleCount() { return this.#toggleCount; }

  // ── Observer mini-pattern for UI sync ────────────────────
  subscribe(fn)   { this.#listeners.add(fn); }
  unsubscribe(fn) { this.#listeners.delete(fn); }
  #notify()       { this.#listeners.forEach(fn => fn(this.#theme)); }
}
💡
Why private class fields (#)? Using #instance, #theme, and #listeners means no external code can accidentally mutate internal state. The only way in is through the public API — a key principle of encapsulation that makes the Singleton safer in practice.

Honest Trade-off Analysis

✅ Genuine Strengths

  • ✔️ Single source of truth. No possibility of two managers disagreeing about the current theme.
  • ✔️ Lazy initialisation. The instance is created only when first needed, not at import time.
  • ✔️ Controlled access. The private constructor prevents accidental instantiation.
  • ✔️ Appropriate for true globals. A theme, a logger, a hardware driver — things that genuinely exist once in the real world.
  • ✔️ Easy cross-component coordination. Any module can call getInstance() without prop-drilling or context providers.

❌ Real Pitfalls

  • ⚠️ Hidden global state. Functions that call getInstance() internally are secretly coupled to global state — hard to reason about.
  • ⚠️ Testing nightmare. State persists between tests. You must manually reset or hack the private field between each test run.
  • ⚠️ Tight coupling. Consumers depend on the concrete class, not an interface. Swapping implementations requires changing every call site.
  • ⚠️ Concurrency issues. In multi-threaded environments (Node.js workers, shared workers) the "one instance" guarantee can break.
  • ⚠️ Violates SRP & OCP. The class manages its own lifecycle and its business logic — two responsibilities.
⚠️
The "Gang of Four" regret. The original GoF authors have since noted that the Singleton is the most misused pattern in the book. It is often reached for simply because global variables feel convenient — not because a single instance is genuinely required by the domain.
🔄

When to Use Alternatives

Recommended

ES Module as a Singleton

In modern JavaScript, a module is already a singleton. The runtime caches the module after the first import — subsequent imports receive the same exported object. No class required. This is the cleanest approach for most front-end scenarios.

theme.js — Module Singleton
// theme.js — exported object IS the singleton
let _theme = 'light';
const _listeners = new Set();

export const themeStore = {
  get: ()      => _theme,
  set: (t)     => { _theme = t; _listeners.forEach(f => f(t)); },
  toggle: ()   => themeStore.set(_theme === 'light' ? 'dark' : 'light'),
  subscribe: fn => _listeners.add(fn),
};

// Any file that imports this gets the SAME object — no getInstance() needed.
// import { themeStore } from './theme.js';
Best for testability

Dependency Injection

Instead of calling getInstance() inside a function, pass the theme manager in as a parameter. Consumers declare what they need; the caller decides what to provide. This makes unit testing trivial — just pass a mock.

Dependency Injection approach
// ✅ Testable — the dependency is explicit and swappable
function renderNavbar(themeService) {
  const btn = document.querySelector('#toggle');
  btn.addEventListener('click', () => themeService.toggle());
  themeService.subscribe(t => btn.textContent = t);
}

// Production: inject the real manager
renderNavbar(ThemeManager.getInstance());

// Test: inject a mock — no global state pollution!
renderNavbar({
  toggle: jest.fn(),
  subscribe: jest.fn(),
});

// ❌ Hard to test — hidden coupling to global state
function renderNavbarBad() {
  const tm = ThemeManager.getInstance(); // ← surprise dependency
  // ...
}
Framework pattern

React Context / Zustand / Pinia

Framework state managers solve the same problem with better ergonomics, built-in reactivity, and DevTools support. They are "singletons by convention" but are designed to be testable and replaceable.

React Context equivalent
// ThemeContext.jsx
const ThemeCtx = createContext(null);

export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const toggle = () => setTheme(t => t === 'light' ? 'dark' : 'light');

  return (
    <ThemeCtx.Provider value={{ theme, toggle }}>
      {children}
    </ThemeCtx.Provider>
  );
}

// Any component — no getInstance(), no hidden globals
export const useTheme = () => useContext(ThemeCtx);

📋 Decision Guide

Situation Reach for…
Truly one-of-a-kind resource (hardware, OS logger) Singleton
Simple shared state, vanilla JS project ES Module export
You need to unit-test consumers in isolation Dependency Injection
React / Vue / Svelte application Context / Pinia / Zustand
Large-scale app with many services DI Container (InversifyJS)
"I just want a global variable" Stop — use a module
