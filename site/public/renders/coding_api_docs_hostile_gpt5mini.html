REST API Review & Corrections — Interactive Guide

Analyzes the supplied endpoints, points out every problem, and provides corrected, consistent RESTful designs with examples and recommended HTTP semantics.

Original endpoints (as provided)

GET  /getUsers
POST /user/delete/5
GET  /createUser?name=Jo
PUT  /updateAllUsers

High-level problems found (summary)

  • Inconsistent naming and casing (getUsers, user, createUser, updateAllUsers).
  • Actions (get, create, delete, update) are embedded in paths rather than using HTTP methods and resources.
  • Using GET for state-changing operations (e.g., creating resources) — violates safe/idempotent semantics.
  • Using POST with a delete verb in the path (/user/delete/5) instead of DELETE method on the resource URI.
  • Using query strings to create a resource (GET /createUser?name=Jo) — wrong method and wrong place for payload.
  • Ambiguous resource identifiers and pluralization inconsistencies (user vs getUsers vs updateAllUsers).
  • No clear request/response bodies, status codes, pagination, filtering, or error conventions documented.

Design principles used to correct the API

  • Use nouns (resources) in URIs, not verbs. Example: /users, /users/{id}.
  • Use HTTP methods for intent:
    • GET — retrieve (safe, idempotent)
    • POST — create (not necessarily idempotent)
    • PUT — replace a resource (idempotent)
    • PATCH — partially update a resource (idempotent)
    • DELETE — remove a resource (idempotent)
  • Use plural resource names for collections (e.g., /users).
  • PUT to a collection should be avoided for partial updates; use PATCH for partial updates to a resource. To replace an entire collection, use an explicit URI and careful semantics.
  • Use request bodies for create/update payloads (JSON). Use query parameters for filtering, sorting, pagination.
  • Return proper HTTP status codes and consistent error objects (e.g., 200/201/204/400/404/409/500).
  • Design for hypermedia where practical (HATEOAS optional). Keep URIs discoverable and predictable.

Endpoint-by-endpoint analysis and corrections

Endpoint: GET /getUsers

Problems identified

  • URI contains a verb "get" — path should represent a resource (noun).
  • Likely should be /users (collection). Using verbs in paths is redundant with HTTP methods.
  • No pagination/filtering options shown; returning entire dataset could be inefficient.
  • No response format, error handling, or status code guidance.

Corrected design

Resource GET /users

Description: Retrieve a paginated list of users. Safe and idempotent.

GET /users?page=1&per_page=25&sort=created_at:desc&search=jo

Headers: 

Accept: application/json

Success response (200 OK):

{
  "data": [
    { "id": 5, "name": "Jo", "email": "jo@example.com", "created_at":"2025-11-02T12:34:56Z" },
    ...
  ],
  "meta": { "page": 1, "per_page": 25, "total": 347 },
  "links": { "self": "/users?page=1&per_page=25", "next": "/users?page=2&per_page=25" }
}

Notes:

  • Use standard query parameters for pagination (page/per_page or limit/offset).
  • Provide filtering (e.g., ?name=Jo or ?search=jo), sorting, and field selection (?fields=id,name).
  • Consider caching headers (ETag, Cache-Control) for GET responses.

Endpoint: POST /user/delete/5

Problems identified

  • Embedding an action "delete" in the path while using POST is incorrect. DELETE is the appropriate HTTP method.
  • Path uses singular /user while the rest use other forms — inconsistent. Prefer plurals for collections: /users/5.
  • POST is not idempotent while DELETE should be idempotent; using wrong method changes semantics and can confuse clients.

Corrected design

Resource DELETE /users/{id}

Description: Delete user with id 5.

DELETE /users/5

Success responses:

  • 204 No Content — if deletion succeeded and there's no body to return.
  • 200 OK — if you return a body describing the deleted resource or a deletion confirmation.

Error responses:

  • 404 Not Found — if user 5 doesn't exist.
  • 403 Forbidden — if client not allowed to delete.

If you need to support soft deletes or bulk deletes:

  • Soft delete: PATCH /users/5 with body { "deleted": true } or a dedicated action endpoint like POST /users/5/deactivate (but prefer PATCH/PUT to use HTTP semantics).
  • Bulk delete: POST /users/delete with a JSON body { "ids": [5,8,9] } or better: DELETE /users?ids=5,8,9 (note: not all servers/clients like bodies on DELETE; use POST /users/bulk-delete if you need a request body and the operation is not idempotent).
Endpoint: GET /createUser?name=Jo

Problems identified

  • GET is used to create state — violates safe semantics of GET. GET must not change server state.
  • Using query string to carry create payload is nonstandard and limited (length limits, poor structure for complex objects).
  • Path contains a verb "createUser". Resource URIs should be nouns like /users.

Corrected design

Resource POST /users

Description: Create a new user.

POST /users
Content-Type: application/json

{
  "name": "Jo",
  "email": "jo@example.com",
  "password": "secret"    // or omitted if created via OAuth/SSO
}

Success response:

  • 201 Created — Location header set to the new resource URI: 
    Location: /users/42
    Response body (optional): 
    {
  "id": 42,
  "name": "Jo",
  "email": "jo@example.com",
  "created_at": "2026-03-01T15:12:00Z"
}

Notes:

  • Use server-side validation; return 400 Bad Request with an error object if validation fails.
  • If creation can result in a conflict (duplicate email), return 409 Conflict with details.
  • Do not pass secrets (passwords) in query strings or via GET.

Endpoint: PUT /updateAllUsers

Problems identified

  • Path includes a verb "updateAllUsers"; resource URIs should be nouns. "All users" is ambiguous.
  • PUT semantics are for replacing a resource at a URI. PUTting to a verb-like collection name is confusing.
  • It's unclear whether this is a bulk update, a replace of the entire collection, or something else.

Corrected designs (choose based on intent)

If you want to replace the entire users collection (rare):

PUT /users
Content-Type: application/json

[
  { "id": 1, "name": "Alice", ... },
  { "id": 2, "name": "Bob", ... }
]

Notes: Replacing entire collections is rarely desirable — it's risky, large, and concurrency-prone. This operation should be used only with care, strong auth, and transactional semantics.

If you want to perform a bulk partial update (most common):

PATCH /users
Content-Type: application/json

{
  "filter": { "active": false, "country": "US" },
  "patch": { "marketing_opt_in": false }
}

Notes:

  • This is a bulk operation that updates multiple resources matching the filter.
  • It should be documented as non-idempotent or idempotent depending on implementation; choose method accordingly (PATCH is appropriate for partial updates).
  • Consider background job semantics: respond 202 Accepted with a job resource that can be polled: POST /jobs with body describing the bulk update.

If you want to update a single user:

PATCH /users/{id}
Content-Type: application/json

{ "name": "Josephine" }

Recommended pattern for long-running/bulk tasks:

POST /bulk/user-updates
Content-Type: application/json

{
  "filter": { "signup_before": "2020-01-01" },
  "update": { "tier": "legacy", "email_verified": true }
}

Response: 202 Accepted
Location: /jobs/678
Body: { "job_id": 678, "status": "pending" }

Error and status conventions
  • 400 Bad Request — validation errors. Return { "error": "validation_failed", "details": {...} }
  • 401 Unauthorized — missing/invalid auth.
  • 403 Forbidden — not allowed.
  • 404 Not Found — resource doesn't exist.
  • 409 Conflict — resource conflict (duplicate email).
  • 429 Too Many Requests — rate limits.
  • 500/502/503 — server errors.
Security & other recommendations
  • Always use HTTPS. Never pass secrets in URLs or query strings.
  • Use standard authentication (Bearer tokens / OAuth2 / API keys) and return 401 for missing/invalid tokens.
  • Document rate limits and pagination defaults.
  • Consider HATEOAS links in responses for discoverability (optional).
