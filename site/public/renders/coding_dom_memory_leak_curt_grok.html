Explanation: Why the Garbage Collector Isn't the Problem

In this demo, the "buggy" version simulates a common issue in JavaScript applications: when widgets are removed from the DOM, their event listeners and references (e.g., stored in arrays or closures) are not cleaned up. This prevents the browser's garbage collector from freeing the memory associated with those widgets, as JavaScript's GC only collects objects that are no longer reachable.

Why is the GC diagnosis wrong? The garbage collector itself is working correctly—it's designed to free memory for objects with no active references. The real problem is in the code: lingering references (like those in the buggyReferences array or unattached event listeners) keep objects alive. In the buggy version, each widget's event listener and reference in the array hold onto the object, causing "leaked references" to accumulate. In the fixed version, we explicitly remove the event listener and clear the reference, allowing the GC to reclaim the memory.

To see this in action:

  • Add several widgets in both versions.
  • Remove them and observe the "Leaked References" counter.
  • In a real app, this could lead to growing memory usage over time, but it's not the GC's fault—it's the developer's responsibility to manage references.

Key Takeaway: Always detach event listeners and clear references when objects are no longer needed to avoid memory leaks.
