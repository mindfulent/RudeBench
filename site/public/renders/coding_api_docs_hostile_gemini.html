REST API Design Review & Correction

This document analyzes a problematic REST API design, identifies specific violations of REST principles, and proposes corrected, consistent, and truly RESTful alternatives.

Key REST Principles Violated (Summary):

  • Resource-Oriented Naming: Endpoints should represent resources (nouns), not actions (verbs).
  • Standard HTTP Methods: HTTP methods (GET, POST, PUT, DELETE) should indicate the intended action on the resource.
  • Statelessness: Each request should contain all necessary information; the server should not rely on previous requests.
  • Consistent Naming & Pluralization: Use plural nouns for collections, singular for specific items.
  • URI Structure: Use path segments for resource identification, not query parameters for actions.
  • Idempotency & Safety: GET is safe and idempotent. PUT is idempotent. POST is neither. DELETE is idempotent.
  • Payload for Data: Use request bodies (JSON/XML) for sending data with POST/PUT, not query parameters for complex data.

Original: GET /getUsers

  • Problem 1: Action-Oriented Endpoint Name. The endpoint name /getUsers uses a verb ("get") which violates the principle of resource-oriented design. URIs should identify resources (nouns), and the HTTP method should indicate the action.
  • Problem 2: Redundant HTTP Method. The "get" in getUsers is redundant because the HTTP method GET already signifies retrieval.
Correction: Identify the resource as "users" (a collection). The GET method correctly indicates retrieval.

Proper Endpoint:

GET /users

Explanation: This endpoint retrieves a collection of user resources. Query parameters can be used for filtering, sorting, or pagination (e.g., GET /users?status=active&limit=10).

Original: POST /user/delete/5

  • Problem 1: Incorrect HTTP Method. POST is used for creating new resources or performing non-idempotent actions. Deleting a resource should use the DELETE method.
  • Problem 2: Action-Oriented URI. The URI /user/delete/5 includes the verb "delete", violating resource-oriented design. The URI should point to the resource, and the method should define the action.
  • Problem 3: Inconsistent Naming. The URI uses /user/ (singular) but implies an action on a specific user identified by '5'. If it's a collection, it should be plural.
  • Problem 4: URI Structure for ID. The ID '5' is part of an action-oriented path segment, not directly identifying the resource within a collection.
Correction: Use the DELETE method on the specific user resource identified by its ID within the collection.

Proper Endpoint:

DELETE /users/5

Explanation: This endpoint correctly uses the DELETE method to remove the user resource with ID '5' from the /users collection. The URI clearly identifies the specific resource to be acted upon.

Original: GET /createUser?name=Jo

  • Problem 1: Incorrect HTTP Method. GET requests are for retrieving data and should be safe (not change server state) and idempotent (multiple identical requests have the same effect as a single one). Creating a user changes server state, so GET is inappropriate.
  • Problem 2: Action-Oriented Endpoint Name. Similar to /getUsers, /createUser uses a verb ("create"), violating resource-oriented design.
  • Problem 3: Data in Query Parameters. Sending data for creation (like name=Jo) via query parameters for a GET request is highly problematic. Query parameters are meant for filtering, sorting, or pagination, not for the main payload of a resource creation. It also limits the complexity and size of data.
  • Problem 4: Security Risk (Implicit). While not a direct REST principle violation, sending sensitive data like user names directly in query parameters can be logged in server access logs, browser history, etc., making it less secure than a request body.
Correction: Use the POST method on the collection resource to create a new resource. Send the new resource's data in the request body.

Proper Endpoint:

POST /users

Request Body (JSON):

{ "name": "Jo", "email": "jo@example.com", "status": "active" }

Explanation: The POST method on the /users collection is the standard way to create a new user. The data for the new user is sent securely and flexibly in the request body (e.g., as JSON). Upon successful creation, the server typically responds with a 201 Created status code and a Location header pointing to the URI of the newly created resource (e.g., /users/123).

Original: PUT /updateAllUsers

  • Problem 1: Action-Oriented Endpoint Name. /updateAllUsers uses a verb ("update") and implies an action rather than identifying a resource.
  • Problem 2: Misuse of PUT for a Collection-Wide Update. While PUT is used for *replacing* a resource, applying it to a collection (/users) implies replacing the *entire collection* with the content of the request body. This is rarely the intended behavior for "update all users." If the intent is to update *properties* of multiple users, PUT is generally not suitable for bulk partial updates; PATCH is more appropriate for partial updates, or a more specific action endpoint might be needed for complex bulk operations.
  • Problem 3: Ambiguity and Potential Danger. If PUT /users were used to *replace* the entire collection, it would be extremely dangerous if not handled with extreme care, as it could wipe out existing users not included in the payload.
  • Problem 4: Missing Payload. A PUT request typically requires a request body containing the full representation of the resource(s) to be replaced. The original endpoint doesn't specify how the "update all" data would be provided.
Correction (Scenario 1: Update a specific user): If the intent was to update a single user, the PUT method is used on the specific resource URI.

Proper Endpoint:

PUT /users/5

Request Body (JSON - full representation):

{ "id": 5, "name": "Joanne Smith", "email": "joanne.smith@example.com", "status": "active" }

Explanation: PUT /users/5 would replace the entire representation of user 5 with the data provided in the request body. It is idempotent.

Correction (Scenario 2: Partially update a specific user): If only a few fields of a specific user need updating, PATCH is more suitable.

Proper Endpoint:

PATCH /users/5

Request Body (JSON - partial representation):

{ "name": "Joanne Smith" }

Explanation: PATCH /users/5 would apply a partial update to user 5, modifying only the specified fields. It is not necessarily idempotent by default, depending on implementation.

Correction (Scenario 3: Bulk update of multiple users): For complex bulk updates across multiple resources, a dedicated action endpoint might be necessary, or a collection PATCH if a standard exists for your API. A collection-level PATCH might look like this:

Possible (but less common/standard) Endpoint for Bulk Partial Update:

PATCH /users

Request Body (JSON - array of partial updates):

[ { "id": 1, "status": "inactive" }, { "id": 2, "name": "New Name" } ]

Explanation: This approach allows for partial updates to multiple users in a single request. It's less universally standardized than single-resource PATCH, but common in practice. It's crucial to define the expected payload format clearly.

General REST API Best Practices Recap:

  • Use Nouns for Resources: Endpoints should represent resources (e.g., /users, /products/123).
  • Use HTTP Methods for Actions:
    • GET: Retrieve a resource or collection (safe, idempotent).
    • POST: Create a new resource in a collection (not safe, not idempotent).
    • PUT: Replace an entire resource or collection (idempotent).
    • PATCH: Partially update a resource (not necessarily idempotent).
    • DELETE: Remove a resource (idempotent).
  • Consistent Pluralization: Use plural nouns for collections (/users) and singular for specific resources (/users/{id}).
  • Path Variables for IDs: Use /users/{id} to identify specific resources.
  • Query Parameters for Filtering/Sorting/Pagination: (e.g., /users?status=active&sort=name).
  • Use Request Body for Data: For POST, PUT, PATCH, send the data payload in the request body (typically JSON).
  • Meaningful Status Codes: Return appropriate HTTP status codes (e.g., 200 OK, 201 Created, 204 No Content, 400 Bad Request, 404 Not Found, 500 Internal Server Error).
  • HATEOAS (Hypermedia as the Engine of Application State): (Advanced) Include links to related resources in responses to guide clients.
