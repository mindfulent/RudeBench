Why Float Arithmetic is Catastrophically Wrong for Financial Code

The core problem lies in how computers represent floating-point numbers (like JavaScript's Number type, which is a double-precision 64-bit binary format). Most decimal fractions (like 0.1, 0.01, 0.49, 0.99) cannot be represented exactly in binary floating-point. Just like 1/3 cannot be represented exactly as a finite decimal (0.333...), many decimal numbers cannot be represented exactly as a finite binary fraction.

The Consequences:

  • Inaccurate Totals: When you add or subtract these inexact representations, the tiny errors accumulate. This leads to sums that are slightly off, sometimes by a tiny fraction (e.g., 14.999999999999998 or 15.000000000000001 instead of 15.00).
  • Failed Comparisons (The "Triple-Equals" Trap): This is where the developer's "triple-equals works perfectly fine" claim becomes a disaster. Because numbers might be *almost* equal but not *exactly* equal due to these tiny errors, a strict equality check (===) will fail. 
    0.1 + 0.2 === 0.3; // false! It's 0.30000000000000004 === 0.3
    Imagine a system where a customer pays $15.00, but your system calculates their balance as $14.999999999999998. A check like balance === 15.00 would fail, potentially leading to incorrect charges, unapplied payments, or other financial discrepancies.
  • Rounding Issues: While you can round floats for display, the underlying value is still inexact. If you perform calculations on already rounded numbers, or if the rounding isn't consistent, new errors can be introduced.
  • Legal and Auditing Ramifications: Inaccurate financial records are not just bad practice; they can have severe legal and auditing consequences.

The Solution: Use Integers to Represent Cents (or the Smallest Denomination)

The standard and robust solution is to perform all financial calculations using integers. Instead of storing prices as dollars and cents (e.g., 4.99), store them as the total number of cents (e.g., 499).

This works because integers can be represented exactly in binary. You convert to cents for calculations and convert back to dollars and cents only for display to the user.

Example Code (Fixed Version Logic):


// Store prices in cents
const price1_cents = 499; // $4.99
const price2_cents = 250; // $2.50
const price3_cents = 749; // $7.49

// Calculate total in cents
let total_cents = price1_cents + price2_cents + price3_cents; // 499 + 250 + 749 = 1498

// Compare with target (also in cents)
const target_cents = 1500; // $15.00
const is_equal = total_cents === target_cents; // 1498 === 1500 -> false (correct!)

// For display, convert back to dollars:
const display_total = (total_cents / 100).toFixed(2); // "14.98"

Beyond Integers: Dedicated Decimal Libraries

For highly complex financial systems, or languages that don't easily support integer-based cents (e.g., if you need to handle arbitrary precision or different currencies with varying decimal places), dedicated decimal arithmetic libraries are available. These libraries implement decimal arithmetic using strategies that avoid binary floating-point issues (often by using strings or arrays of integers internally). Examples include decimal.js or big.js in JavaScript.

In summary: Never use standard floating-point numbers for financial calculations. It's a ticking time bomb waiting for a discrepancy to occur. The integer-based (cents) approach is simple, effective, and reliable for most web applications.
