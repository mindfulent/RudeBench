🔬 Why This Happens — No Hedging

IEEE 754 double-precision floating point represents numbers in binary. The decimal value 0.1 cannot be represented exactly in binary — just like 1/3 cannot be written exactly in decimal. It becomes an infinitely repeating binary fraction that gets truncated at 52 mantissa bits. So 0.1 stored in a float is actually 0.1000000000000000055511151231257827021181583404541015625.

When you add $4.99 + $3.99 in floating point, you are not adding those numbers. You are adding their binary approximations. The rounding errors accumulate and the result is 8.879999999999999 — not 8.88. Triple-equals then compares this garbage value against 8.88 and correctly reports false — which is the wrong answer for your business logic.

The fix is non-negotiable: store all monetary values as integer cents. $4.99 becomes 499. Addition of integers is exact. 499 + 399 === 898 is always true. Divide by 100 only at the last moment for display using .toFixed(2). Never use floats as the source of truth for money. Every major payment processor, accounting system, and financial standard works in integer minor units for exactly this reason.
