Buggy Counter (Race Condition)

Current Count: 0

Rapid clicks may cause the count to be incorrect due to overlapping updates.

Corrected Counter (No Race Condition)

Current Count: 0

Rapid clicks will always show the correct count.

Explanation of the Race Condition and Fix

Why the race condition occurs in the buggy version:

In the buggy counter, each button click schedules an asynchronous update using setTimeout. This update reads the current value of the shared count variable, increments it, and writes it back. If you click the button rapidly, multiple setTimeout calls can overlap. For example:

  • Click 1: Reads count (e.g., 0), increments to 1, but before writing, another click happens.
  • Click 2: Reads the still-unchanged count (e.g., 0), increments to 1, and writes 1.
  • Click 1 finally writes its value (1), overwriting the previous write.

This results in lost updates, making the displayed count fall behind or show incorrect values because the operations are not atomic—they don't happen as a single, uninterruptible step.

How the fix resolves it in the corrected version:

The corrected counter uses a promise chain to serialize the updates. Each increment operation waits for the previous one to complete before starting. This ensures that even if multiple button clicks are made rapidly, the updates are processed one at a time in the order they were requested. Specifically:

  • We maintain a chain of promises where each setTimeout call returns a promise that resolves after the update is done.
  • This prevents overlapping by queuing the operations, making the increments effectively atomic in sequence.

As a result, the corrected counter always reflects the accurate total number of clicks, regardless of how fast you click.
