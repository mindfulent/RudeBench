Understanding the Race Condition and Its Fix

The Buggy Counter: Why it Fails

  • Shared Resource: Both counters operate on a shared variable (buggyCount and fixedCount respectively).
  • Asynchronous Operation: When you click the button, instead of incrementing immediately, we simulate an asynchronous operation using setTimeout(..., 100). This delay is crucial.
  • Read-Modify-Write Cycle:
    1. A click event handler reads the current value of the counter (e.g., 0).
    2. It then schedules a task (via setTimeout) to modify that value (add 1).
    3. Finally, the scheduled task writes the new value back to the display.
  • The Race: If you click the button rapidly, say twice, within the 100ms delay:
    • Click 1: Reads buggyCount = 0. Schedules to update to 0 + 1 = 1.
    • Click 2 (happens before Click 1's update finishes): Reads buggyCount = 0 (it's still 0 because Click 1 hasn't written its result yet!). Schedules to update to 0 + 1 = 1.
  • The Result: Both operations end up calculating 1 and writing 1 to the display. You clicked twice, but the counter only shows 1, missing one increment. This is a race condition because the outcome depends on the unpredictable timing or "race" of multiple operations trying to access and modify the shared resource.

The Fixed Counter: How it Works

  • The Problem: The core issue in the buggy version is that the "read" operation happens *before* the asynchronous operation completes, leading to outdated information.
  • The Fix (Queuing/Synchronization): The corrected version introduces a simple form of synchronization by using a **promise chain** (or more simply, by processing one update at a time).
    • Instead of directly incrementing fixedCount, each click adds a task to a queue.
    • A mechanism ensures that only one increment operation (the setTimeout) is active at any given time.
    • The isUpdating flag acts as a **mutex** (mutual exclusion lock). When an update starts, isUpdating is set to true, preventing new updates from starting until the current one finishes.
    • Subsequent clicks just increment pendingIncrements, which acts as a queue. Once the current update finishes, it checks pendingIncrements and processes the next one if available.
  • Guaranteed Order: This ensures that each increment operation reads the *latest* value of fixedCount after the previous operation has completed and written its result. No increments are "lost."
  • Alternative Fix (Simpler for this case): A simpler fix for *just this specific counter* would be to simply remove the `setTimeout` and increment immediately, but the purpose here is to demonstrate how to handle asynchronous operations that *must* read and write a shared state without race conditions. The `isUpdating` flag and `pendingIncrements` demonstrate a basic queuing mechanism.
