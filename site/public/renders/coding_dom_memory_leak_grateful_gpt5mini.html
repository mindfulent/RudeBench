Explanation (short):
If removed widgets appear to release from the DOM but memory (as shown by a dashboard or devtools heap) grows, it typically means something in your JS code is still holding references to those widgets (event listeners, arrays, maps, timers, closures). The garbage collector works when there are no reachable references to an object. This demo shows a common mistake: keeping arrays of widget objects or closures that close over DOM nodes without clearing them when the widget is removed — this prevents GC, so memory appears to "grow" even if DOM nodes are gone.
Why "GC isn't working" is usually the wrong diagnosis
- The JavaScript garbage collector reclaims objects that are unreachable from roots (global variables, currently executing stack frames, DOM/node roots, closures, etc). If an object remains reachable, GC will not collect it. - Common causes of apparent leaks: arrays/maps that keep references to removed items, failing to call removeEventListener (or using inline handlers kept in other scopes), timers (setInterval/setTimeout), or closures that accidentally capture DOM nodes or large data. - The correct fix is to find the live references and break them (remove listeners, clear arrays, null out properties) so objects become unreachable. Use DevTools memory snapshots (allocation/retainer graphs) to find what is holding objects alive.
