📖 Understanding the Race Condition

🐛 The Problem: Read-Modify-Write Race

When multiple async tasks share a variable, each one performs three steps:

  • Read the current value of sharedProgress
  • Compute the new value (add their increment)
  • Write the result back

Because JavaScript is single-threaded but interleaved via the event loop, two tasks can both read the same stale value before either writes back. The second write overwrites the first, causing lost updates — or both add their full increment to the same base, causing overflow past 100%.

Time →→→→→→→→→→→→→→→→→→→→→→→→→→→→ Task A: READ(30) → +25 → WRITE(55) Task B: READ(30) → +30 → WRITE(60) Result: 60 ← Task A's write was lost! OR both add to same base → 85+ → OVERFLOW
// ❌ BUGGY: no synchronization let sharedProgress = 0; async function buggyTask(name, increment, delay) { await sleep(delay); // Both tasks read 30 at the same time! const current = sharedProgress; // stale read await sleep(50); // simulate work sharedProgress = current + increment; // overwrites! }

✅ The Fix: Serialized Promise Queue

The solution is to ensure that read-modify-write operations are atomic — no other task can interleave between the read and the write.

  • A promise chain (taskQueue) acts as a mutex
  • Each task appends itself to the chain with .then()
  • Tasks execute one at a time, in order — no interleaving
  • A clamp guard caps the total at 100% as a safety net
Time →→→→→→→→→→→→→→→→→→→→→→→→→→→→ Task A: READ(30) → +25 → WRITE(55) ✓ Task B: READ(55) → +30 → WRITE(85) ✓ Result: 85 ← Both updates preserved! Sequential, no lost writes, no overflow
// ✅ FIXED: serialized via promise queue let sharedProgress = 0; let taskQueue = Promise.resolve(); // mutex function fixedTask(name, increment, delay) { taskQueue = taskQueue.then(async () => { await sleep(delay); // Safe: only one task runs at a time const next = sharedProgress + increment; sharedProgress = Math.min(next, 100); // clamp updateUI(name, sharedProgress); }); return taskQueue; }
