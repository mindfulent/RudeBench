Race Condition Counter Demonstration

Buggy Counter (Race Condition)

0

Click rapidly to observe the count falling behind!

Why the Race Condition Occurs:

This counter demonstrates a race condition. Each click triggers an asynchronous update using setTimeout. The update process involves these steps:

  1. READ: The function reads the current value of buggyCount.
  2. MODIFY: It calculates the new value (current + 1).
  3. WRITE: After a short delay (simulating network latency or complex computation), it writes the new value back.

If you click rapidly, multiple "read" operations can happen before the first "write" operation completes. For example:

  • Click 1: Reads 0. Schedules to write 1.
  • Click 2: Reads 0 (because Click 1 hasn't written yet). Schedules to write 1.
  • Click 1's timeout fires, writes 1.
  • Click 2's timeout fires, writes 1 (overwriting Click 1's result).

The result is that some increments are lost because they all started with an outdated value.

Corrected Counter (No Race Condition)

0

This counter will always be accurate, even with rapid clicks.

How the Race Condition is Fixed:

This counter prevents the race condition by using a queueing mechanism. Instead of directly updating the count, each click adds an "increment" operation to a queue.

A separate function then processes this queue one item at a time. It waits for the current update to fully complete (read, modify, write) before taking the next item from the queue.

This ensures that each increment operation starts with the most up-to-date count, preventing any "reads" from stale data. This is a common pattern for managing asynchronous operations that modify shared state, often implemented with techniques like promises, async/await with locks (in environments that support them), or simple queues like this.
