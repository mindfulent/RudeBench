How to Trigger the Bug:

Click the "Increment (Buggy)" button *very rapidly* multiple times. You'll likely see the counter value jump or miss some increments, especially if you click 5-10 times quickly.

This version simulates an asynchronous operation (like a network request or heavy computation) by using setTimeout to delay the actual update. Rapid clicks expose the race condition.

Fixed Counter (No Race)

0

How it's Fixed:

Click the "Increment (Fixed)" button rapidly. The counter should always increment correctly by one for each click, even if you click very fast.

This version prevents the race condition by disabling the button (or using a flag) during the asynchronous update, ensuring only one increment operation is "in flight" at a time, making the read-modify-write effectively atomic.

Understanding the Race Condition

The Core Problem: Non-Atomic Read-Modify-Write

A race condition occurs when multiple asynchronous operations try to access and modify shared state (in this case, our counter variable) concurrently, and the final outcome depends on the non-deterministic order in which these operations complete.

In our "Buggy Counter" example, the increment operation is broken down into three logical steps:

  1. Read: Get the current value of the counter.
  2. Modify: Calculate the new value (current value + 1).
  3. Write: Update the counter with the new value.

When these steps are performed synchronously, they happen one after another, and no other operation can interfere. However, by introducing setTimeout, we simulate an asynchronous delay between the "read" and the "write" steps.

Scenario for the Buggy Counter:

Imagine the counter is at 0 and you click the button twice very quickly:

  • Click 1:
    • Reads counter = 0.
    • Starts a setTimeout to perform the write after 100ms, calculating 0 + 1 = 1.
  • Click 2 (happens almost immediately, before Click 1's setTimeout fires):
    • Reads counter = 0 (it's still 0 because Click 1 hasn't written its result yet!).
    • Starts another setTimeout to perform the write after 100ms, calculating 0 + 1 = 1.
  • After 100ms (approximately):
    • Click 1's setTimeout fires: It writes counter = 1.
    • Click 2's setTimeout fires (shortly after or around the same time): It also writes counter = 1.

The expected result after two clicks should be 2, but because both operations read the *same initial value* before any writes occurred, they both ended up writing 1, causing one increment to be "lost". This is a classic "lost update" problem.

Why the Fixed Counter Works:

The fixed counter prevents this by ensuring that the read-modify-write cycle is effectively atomic from the perspective of external triggers. It does this by:

  • Disabling the button: When an increment operation starts, the button is disabled. This prevents new clicks from initiating another read-modify-write cycle until the current one is complete.
  • Re-enabling the button: Once the setTimeout completes and the counter is updated, the button is re-enabled, allowing the next increment operation to begin.

This approach acts as a simple mutual exclusion mechanism, ensuring that only one asynchronous increment operation can be active at any given time, thus preventing concurrent reads and writes to the shared state. Other solutions could involve using a queue for operations, or more complex state management patterns, but for a simple counter, disabling the UI element is effective.

Key Takeaway:

Even in single-threaded JavaScript, asynchronous operations (like those introduced by setTimeout, fetch, Promises, etc.) can create race conditions if not properly managed, especially when dealing with shared state. The "atomic" nature of an operation can be broken by asynchronous delays, requiring careful synchronization.
