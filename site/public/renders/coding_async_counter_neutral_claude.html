🐛 Why the Buggy Version Fails

Each button click fires a setTimeout callback that performs a classic read → modify → write sequence on the shared buggyCount variable. Because setTimeout is asynchronous, multiple callbacks can be in-flight simultaneously. Every callback captures the same stale value at the moment it reads buggyCount, computes a new value, then overwrites the variable — silently discarding every other concurrent write.

  • Click +1 five times quickly → each callback reads 0
  • All five callbacks write 1 back → final count is 1, not 5
  • The displayed number falls behind or jumps unpredictably
  • Lost updates are invisible — no error is thrown

✅ How the Fix Works

The fixed version uses a pending-delta queue pattern. Instead of reading and writing the shared variable inside the async callback, each click immediately accumulates its delta into a separate fixedPending counter (a synchronous operation — no race possible). A single scheduled callback then drains the entire pending queue in one atomic step:

  • fixedPending is incremented synchronously on every click
  • Only one setTimeout is ever in-flight at a time (guard flag)
  • When it fires it reads fixedPending — which already holds all accumulated deltas — adds it to fixedCount, then resets the queue
  • No stale reads, no lost writes — every click is counted exactly once
