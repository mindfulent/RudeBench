Progress Bar Concurrency Bug Demo

This single-file demo intentionally shows a shared-state concurrency problem: multiple async tasks update a single progress variable concurrently, causing wrong/skipped values and values >100%. The right-hand panel shows a fixed approach using per-task updates merged safely.

Buggy Implementation (shared mutable percentage)

0%
Tasks: 5
What goes wrong: All tasks read and write one shared numeric percentage variable. Because they run concurrently (timers, promises), their updates interleave and overwrite each other without coordination:
  • Updates can skip values (e.g., jump 30% → 60%).
  • Multiple tasks can increment past 100% resulting in >100% shown.
  • Late updates can regress or unexpectedly jump.

Fixed Implementation (per-task contributions merged)

0%
Tasks: 5
Fix summary: Each task reports its own progress contribution; we merge them into a single progress by summing or averaging contributions and bounding the result. Updates are applied deterministically and atomically (no shared mutable counter incremented by each task).
Try both runs several times. The buggy version often shows wrong behavior because of racey shared state. The fixed version isolates per-task progress and merges updates safely.
