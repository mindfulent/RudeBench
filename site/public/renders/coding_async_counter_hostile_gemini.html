The Problem (Race Condition)

This counter demonstrates a race condition. When you click rapidly, multiple "increment" operations start almost simultaneously.

Each click handler:

  1. Reads the current value (e.g., 0).
  2. Starts an asynchronous delay (simulated by setTimeout).
  3. After the delay, increments its *own* read value (e.g., 0 + 1 = 1).
  4. Writes this new value back to the display.

If click A reads 0, and then before A writes its result, click B also reads 0, both will calculate 1. The one that finishes last overwrites the other, effectively "losing" an increment.

Visual Clue: You'll often see the counter increment, then sometimes "jump" back down to a lower value, or simply not reflect the true number of clicks you made.

The Solution (Synchronization)

This counter prevents the race condition by ensuring that only one update operation can be "active" at a time.

This version uses a simple isUpdating flag:

  1. When a click occurs, it checks if an update is already in progress.
  2. If not, it sets isUpdating = true and proceeds.
  3. It reads the value and increments it synchronously, storing it in a variable.
  4. Then, it starts the asynchronous delay.
  5. After the delay, it updates the display with the *already incremented* value.
  6. Finally, it sets isUpdating = false, allowing the next pending update (if any) to proceed.

Alternative Fixes:

  • Queuing: Add pending operations to a queue and process them sequentially.
  • Atomic Updates: Some systems provide "compare-and-swap" operations.
  • State Management: Use a central state management pattern (like Redux or Vuex) where updates are dispatched actions that modify state in a controlled, synchronous manner.

Visual Clue: This counter should always accurately reflect the number of times you clicked, even with rapid clicks.
