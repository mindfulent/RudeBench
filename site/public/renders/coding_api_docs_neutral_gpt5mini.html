Design problems (comprehensive)

Every issue in the original endpoints, with short explanation and impact.

1) Verb in the path (GET /getUsers)
Paths should represent resources (nouns). Using the HTTP method and a verb in the path duplicates intent and breaks consistency. Use /users with GET instead.
2) Using POST for deletion and embedding action + id in path (POST /user/delete/5)
DELETE is the correct HTTP verb for resource deletion. Use /users/{id} rather than /user/delete/{id}. Embedding action in path mixes RPC style into REST.
3) Creating resources with GET and query parameters (GET /createUser?name=Jo)
GET must be safe and idempotent and not change server state. Use POST to /users with the new resource in the request body. Query strings are poor for structured payloads and sensitive data.
4) Ambiguous mass-update endpoint and method choice (PUT /updateAllUsers)
PUT implies a full replace of a resource; an endpoint named updateAllUsers suggests a batch operation. Define precise semantics: if replacing entire collection, PUT /users with collection payload; for partial changes, PATCH /users or PATCH /users/{id}. Avoid vague names.
5) Inconsistent resource naming (/user vs /users)
Use plural nouns for collections (/users). Mixing singular and plural confuses clients and makes documentation unclear.
6) Missing content type and request/response bodies
Endpoints don't specify JSON payload shape, required fields, or content-type (application/json). Good API docs must list request/response schemas and examples.
7) No mention of status codes, error responses, or validation
Clients need consistent use of HTTP status codes (200, 201, 204, 400, 404, 409, 500) and example error bodies.
8) Lack of versioning and authentication notes
APIs should be versioned (e.g., /v1/users). Also indicate needed auth (Bearer token, scopes) and rate limits.
9) Poor discoverability and no HATEOAS or links
While optional, including link relations or clear resource URIs helps clients navigate (e.g., Location header on create, self links in responses).
10) No pagination, filtering, or sorting semantics for list endpoints
GET /users should support pagination (limit, offset or cursor), filtering, and sorting to avoid large payloads and unclear behavior.
11) Unclear id format and URL encoding
Define whether IDs are numeric, UUIDs, slugs; ensure clients know to URL-encode path segments.
12) No examples for successful or failed operations
Examples reduce integration errors and speed up adoption—include sample requests and responses for success and error cases.
13) Using verbs in endpoint names reduces cacheability and intermediary behavior
Standard HTTP semantics enable caching (GET), idempotency (PUT, DELETE), and safe operations. Custom verbs obscure this.
Corrected RESTful API — Summary

Below are recommended endpoints, HTTP methods, expected status codes, and short notes. Base path includes versioning: /api/v1

GET
/api/v1/users
List users — supports pagination, filtering, fields, sorting
200 OK — returns { data: [User], meta: {pagination} }
POST
/api/v1/users
Create a user. Body: application/json with user object.
201 Created — Location header and created resource in body; 400 Bad Request on validation error
GET
/api/v1/users/{id}
Retrieve single user by id
200 OK or 404 Not Found
PATCH
/api/v1/users/{id}
Partial update of a user (preferred for updates of some fields)
200 OK (with updated resource) or 204 No Content; 400/404 as appropriate
PUT
/api/v1/users/{id}
Full replace of user resource. Use only if you accept full-object semantics
200 OK or 204 No Content; 400/404 as appropriate
DELETE
/api/v1/users/{id}
Delete user
204 No Content on success; 404 if not found
Batch operations (examples)
- Bulk create: POST /api/v1/users/batch (body: array of users) — 201 Created with results
- Bulk update: PATCH /api/v1/users (body: {ids:[], changes:{...}}) — 200 OK with per-item status
- Bulk delete: DELETE /api/v1/users (body: {ids:[...]}) or POST /api/v1/users:delete with caution — prefer DELETE with body or POST to a subresource /batch-delete if DELETE with body unsupported by clients.

Status code guidance
  • 200 OK — Successful GET/ PATCH (returns body)
  • 201 Created — Successful POST creating resource (Location header)
  • 204 No Content — Successful DELETE or PUT with no response body
  • 400 Bad Request — Validation or malformed JSON
  • 401 Unauthorized — Missing/invalid auth
  • 403 Forbidden — Authenticated but not allowed
  • 404 Not Found — Resource not found
  • 409 Conflict — Conflict (e.g., duplicate unique field)
  • 500 Internal Server Error — Unexpected server error
Notes & best practices
  • Version your API in the URL (/v1) or headers.
  • Use JSON and include Content-Type and Accept headers.
  • Prefer PATCH for partial updates; PUT for full replace.
  • Return consistent envelope (e.g., {data, meta, error}).
  • Document authentication, rate limits, and id formats.
  • Provide examples for success and error scenarios.
