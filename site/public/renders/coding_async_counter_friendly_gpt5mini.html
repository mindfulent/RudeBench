Race condition demo: Buggy vs Fixed counter

Click the "Increment" buttons rapidly (or hold the mouse) to trigger asynchronous overlapping updates. The buggy version can lose increments. The fixed version serializes updates so the count never falls behind.

Buggy counter (race-prone)
Shared state: single number
0
Delay multiplier:
Adds random delay up to chosen ms to simulate async work.
How it works (buggy): on click we read the current shared count, wait asynchronously (random delay), then write back (readValue + 1). If two clicks read the same value before either writes, one write overwrites the other and a count increment is lost.
Fixed counter (sequential)
Applies updates one at a time
0
Delay multiplier:
Same asynchronous delay, but updates are queued and processed sequentially.
Fix approach: queue each increment request and process them one-by-one. Each queued job reads the current value only when it runs, ensuring no concurrent reads/writes can conflict.
Notes:
  • Rapidly click both Increment buttons and watch logs. The buggy counter may skip increments when multiple async handlers overlap.
  • The fixed counter sacrifices some concurrency for correctness by serializing updates; other techniques include optimistic compare-and-swap or using atomic operations where available.
