How the Recursive Tree Works

Each call to drawBranch(x, y, length, angle, depth) draws one line segment, then spawns two child calls — one angled left, one right — with length × ratio and depth - 1. The base case fires when depth === 0, stopping the chain. 

function drawBranch(x, y, len, angle, depth) {
  if (depth === 0) return;          // ← base case: MUST exist

  const x2 = x + len * Math.cos(angle);
  const y2 = y + len * Math.sin(angle);
  drawLine(x, y, x2, y2);

  drawBranch(x2, y2, len*ratio, angle-spread, depth-1);
  drawBranch(x2, y2, len*ratio, angle+spread, depth-1);
}

With depth d, this produces 2d+1 − 2 total recursive calls and draws 2d − 1 segments. At depth 12 that's 8,190 segments from 8,191 calls — exponential growth is the defining characteristic of naive binary recursion.

Stack Overflow: The Real Mechanics
⚠ WHY IT CRASHES

Every function call pushes a stack frame onto the call stack. Each frame stores: return address, local variables, parameters, and saved registers. In V8 (Chrome's JS engine) a frame costs roughly 200–500 bytes. The default stack is ~10,000–15,000 frames deep before RangeError: Maximum call stack size exceeded fires.

The call stack during a depth-4 tree looks like this — notice how the left subtree must fully resolve before the right subtree even starts:

drawBranch(depth=4) ← currently executing
drawBranch(depth=3) ← waiting for left child
drawBranch(depth=2) ← waiting for left child
drawBranch(depth=1) ← waiting for left child
drawBranch(depth=0) ← base case, now unwinding

Maximum stack depth at any moment equals the tree depth, not the total call count. Depth 12 = max 12 frames live simultaneously. This tree is actually safe. The danger comes from unbounded depth — e.g. traversing a linked list of 100,000 nodes recursively will overflow because depth = list length.

Hidden multiplier: each frame in this tree also holds x, y, len, angle, depth plus the canvas context reference. In memory-heavy recursive algorithms (e.g. recursive JSON parsing, graph DFS on dense graphs), frames can be kilobytes each, cutting your safe depth dramatically.

Real Solutions — Not Just "Use a Loop"

Solution 1: Explicit Stack (True Iterative Conversion)

Replace the call stack with a heap-allocated array. You control memory, no OS limit, and you can inspect/pause/serialize the work queue mid-execution. 

function drawTreeIterative(startX, startY, len, angle, maxDepth) {
  // Explicit stack holds the same data a call frame would hold
  const stack = [{ x: startX, y: startY, len, angle, depth: maxDepth }];

  while (stack.length > 0) {
    const { x, y, len, angle, depth } = stack.pop();
    if (depth === 0) continue;

    const x2 = x + len * Math.cos(angle);
    const y2 = y + len * Math.sin(angle);
    drawLine(x, y, x2, y2);

    // Push children — order determines traversal direction
    stack.push({ x: x2, y: y2, len: len*ratio, angle: angle+spread, depth: depth-1 });
    stack.push({ x: x2, y: y2, len: len*ratio, angle: angle-spread, depth: depth-1 });
  }
}
// Stack never exceeds O(d) frames in memory — identical to recursive version
// but now lives on the heap with no OS-imposed limit

Solution 2: Tail-Call Optimization (TCO) — and Why JS Lies About It

TCO reuses the current stack frame instead of pushing a new one — but only when the recursive call is the very last operation (a true tail position). This tree cannot be TCO'd because after the left child returns, we still need to call the right child. The frame must survive.

⚠ JS TCO REALITY

ES6 mandated TCO in strict mode. Only Safari actually implements it. V8 (Chrome, Node) intentionally removed their TCO implementation in 2017, citing debuggability concerns. Never rely on TCO in production JS/Node unless you control the runtime.

Solution 3: Trampolining

Instead of calling recursively, return a thunk (a function representing the next step). A trampoline loop calls thunks one at a time — constant stack depth, works in any JS engine, enables lazy/infinite recursion. 

// Thunk: a zero-arg function deferring computation
const thunk = (fn, ...args) => () => fn(...args);

function trampoline(fn) {
  return function(...args) {
    let result = fn(...args);
    while (typeof result === 'function') {
      result = result();   // unwrap one thunk at a time
    }
    return result;
  };
}

// Factorial rewritten for trampolining (single recursive path = works cleanly)
const factT = trampoline((n, acc = 1) =>
  n <= 1 ? acc : thunk(factT, n - 1, n * acc)
);
// factT(100000) — no stack overflow, O(1) stack depth

Solution 4: Async Chunking for UI-Blocking Recursion

For extremely deep trees or slow operations, yield control back to the browser periodically using setTimeout(0) or scheduler.yield(). This prevents "page unresponsive" warnings and allows progress UI updates. 

async function drawTreeAsync(nodes) {
  let i = 0;
  for (const node of nodes) {
    drawSegment(node);
    if (++i % 500 === 0) {
      // Yield to browser every 500 ops — keeps UI responsive
      await new Promise(r => setTimeout(r, 0));
    }
  }
}
✓ DECISION GUIDE

Depth < ~5,000 and not hot path → keep recursion (readable, correct).
Depth unbounded or >10k → explicit stack (safest, fastest).
Functional style, single recursive path → trampolining.
UI thread, massive data → async chunking + Web Workers.

Why This Tree Looks Good
  • Branch width scales with depth. Trunk uses lineWidth = depth × 0.9. Twigs are 1px. This matches real tree biomechanics (Murray's Law: r³ conservation).
  • Color shifts from brown to green. HSL interpolation — deep branches are warm brown, leaf-level branches are desaturated green. Depth ratio drives the hue linearly.
  • Asymmetry slider. Adding a constant angular offset to one branch produces wind-blown or leaning trees. Real trees are never perfectly symmetric.
  • Length ratio controls fractal dimension. At 0.5 the tree is sparse; at 0.85 branches nearly overlap, simulating dense canopy. The Hausdorff dimension of this fractal is log(2) / log(1/ratio).
