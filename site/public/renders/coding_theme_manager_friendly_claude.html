Section 01

Feature Cards

🏗️

Single Instance

Only one ThemeManager object ever exists. Every call to getInstance() returns the exact same reference.

📡

Observer Pattern

Components subscribe to theme changes. The singleton notifies all listeners whenever the theme switches.

💾

Persistence

Preference is saved to localStorage so the chosen theme survives page reloads and browser restarts.

⚡

Zero Dependencies

Pure vanilla JavaScript — no frameworks, no libraries. Just the pattern in its clearest possible form.

🎨

CSS Custom Properties

Theme tokens cascade through the entire document via :root variables, enabling smooth transitions.

♿

Accessible

ARIA labels, keyboard-navigable controls, and sufficient colour contrast in both themes.

Singleton: Pros, Cons & When to Use DI


      

        
        

          
✅ Advantages

          
    
            
  • →Single source of truth. All consumers read from one consistent state, eliminating sync bugs.
    • 
            
  • →Lazy initialisation. The instance is created only when first requested, saving resources.
    • 
            
  • →Global access point. Any module can call getInstance() without prop-drilling or context wiring.
    • 
            
  • →Reduced memory footprint. One object instead of many duplicated instances.
    • 
            
  • →Ideal for shared resources. Config, loggers, caches, connection pools — all benefit from a single instance.
    • 
          

        


        
        

          
❌ Disadvantages

          
    
            
  • →Hard to unit-test. Global state leaks between tests; mocking the singleton is awkward.
    • 
            
  • →Hidden coupling. Consumers depend on a global implicitly, making dependencies invisible in signatures.
    • 
            
  • →Violates SRP. The class manages both its own logic and its own lifecycle (instance control).
    • 
            
  • →Concurrency risks. In multi-threaded environments, race conditions can create multiple instances without careful locking.
    • 
            
  • →Lifetime management. The instance lives for the entire app lifetime, which may waste memory if rarely used.
    • 
          

        

      


      
      

        
💉 When Dependency Injection Is the Better Choice

        

          Dependency Injection (DI) passes a service into a consumer rather than having the consumer
          reach out for a global. Prefer DI over Singleton when:
        

        

          🧪 Writing unit tests that need mock implementations
          🔄 Different consumers need different configurations
          📦 Building a library or reusable package
          🏗️ Working in a framework with a DI container (Angular, Spring)
          🔀 Running parallel test suites in isolation
          📋 Wanting explicit, readable dependency signatures
          ♻️ Needing to swap implementations at runtime
          🧩 Composing complex object graphs cleanly
        

        

          Rule of thumb: use a Singleton for truly app-wide, single-configuration
          resources (theme, logger, feature flags). Use DI everywhere you want testability,
          flexibility, or explicit contracts between modules.
