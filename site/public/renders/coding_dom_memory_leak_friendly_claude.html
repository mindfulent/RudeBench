🧠 Memory Leak vs. Garbage Collector Demo

Why your colleague is wrong about the GC — and what's actually leaking.

🔍 The Diagnosis Problem

Your colleague sees memory growing and blames the garbage collector for "not cleaning up". But the GC can only collect objects that have no remaining references. The real culprit is application code that holds onto references — event listeners registered on a global EventEmitter, closures that capture DOM nodes, and arrays that accumulate removed widget data — all preventing the GC from doing its job. The GC is innocent. The bug is in the cleanup logic (or lack thereof).

📚 What's Actually Happening

❌ Buggy Pattern — Three Leak Sources

  • Global event bus listeners — each widget subscribes to a shared EventEmitter but never unsubscribes. The emitter holds a reference to the handler closure, which in turn captures the widget's DOM node. Even after the element is removed from the DOM, neither the handler nor the node can be GC'd.
  • Static registry array — removed widgets are never spliced out of WidgetRegistry.all[]. The array keeps every widget object alive indefinitely.
  • Interval not cleared — each widget starts a setInterval that updates a counter. Removing the DOM element does not stop the interval. The timer callback holds a closure over the element reference, preventing GC.
// ❌ Leak: listener never removed
function createWidget(id) {
  const el = buildElement(id);
  WidgetRegistry.all.push(el);          // ❌ never removed

  const handler = () => updateEl(el);   // closure captures el
  globalBus.on('tick', handler);        // ❌ never off()'d

  el._interval = setInterval(() => {
    el.dataset.count++;               // ❌ interval keeps el alive
  }, 1000);

  return el;
}

function removeWidget(id) {
  getEl(id).remove();                  // ❌ that's it — nothing cleaned up
}

✅ Fixed Pattern — Proper Cleanup

  • Unsubscribe on removal — the handler reference is stored so globalBus.off('tick', handler) can be called during cleanup, releasing the closure and the captured DOM reference.
  • Registry splice — WidgetRegistry.all is spliced on removal so the array no longer holds a reference to the widget object.
  • clearInterval — the interval ID is stored on the widget and explicitly cleared during removal, allowing both the timer and its closure to be garbage collected.
// ✅ Clean: store references for teardown
function createWidget(id) {
  const el = buildElement(id);
  const idx = WidgetRegistry.all.push({ id, el }) - 1;

  const handler = () => updateEl(el);
  globalBus.on('tick', handler);

  const intervalId = setInterval(() => {
    el.dataset.count++;
  }, 1000);

  // ✅ store everything needed for cleanup
  el._cleanup = { handler, intervalId, idx };
  return el;
}

function removeWidget(id) {
  const el = getEl(id);
  const { handler, intervalId, idx } = el._cleanup;

  globalBus.off('tick', handler);       // ✅ unsubscribe
  clearInterval(intervalId);            // ✅ stop timer
  WidgetRegistry.all.splice(idx, 1);   // ✅ remove from registry
  el.remove();                           // ✅ remove from DOM
}
