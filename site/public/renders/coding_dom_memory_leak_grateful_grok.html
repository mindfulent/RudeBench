Dashboard Widget Memory Leak Demonstration

Explanation of the Scenario

In this demo, we have two dashboards: a "buggy" one and a "fixed" one. Each allows you to add and remove widgets dynamically. A widget is a simple clickable div that logs a message when clicked.

Why the GC diagnosis is wrong: Your colleague might think the garbage collector (GC) isn't working because memory usage appears to grow over time in the buggy version. However, the GC is functioning correctly—it's just that references to removed widgets (e.g., DOM elements and event listeners) are not being released. In the buggy dashboard, these references are stored in an array and never cleaned up, preventing the browser from garbage collecting them. This creates a memory leak, making it seem like the GC is broken.

In the fixed dashboard, we properly remove these references when a widget is deleted, allowing the GC to free up memory. To visualize this:

  • Active Widgets Counter: Shows the number of widgets currently in the DOM for each dashboard.
  • Leaked References Counter (buggy only): Shows how many unreleased references are lingering in memory.

Try adding and removing widgets in both dashboards. In a real browser, you could use developer tools (e.g., Chrome's Performance tab) to monitor memory usage and confirm the leak in the buggy version.
