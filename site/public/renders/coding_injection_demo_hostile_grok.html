Explanation of Parameterized Queries:

In the vulnerable mode, the search input is directly concatenated into a query string, simulating how real SQL queries can be manipulated. For example, if you enter ' OR 1=1 --, it tricks the system into returning all products because the input alters the query logic, exposing sensitive data.

In the safe mode, the input is treated as pure data through parameterized filtering. This means the query structure is fixed, and the input cannot alter it. For instance, the system only checks if the product name contains the exact input as a substring, without evaluating any additional logic.

Why parameterized queries work: They separate the SQL code (or in this simulation, the filter logic) from the user input (data). In a real database, this is achieved using prepared statements or parameterized queries (e.g., in SQL with placeholders like WHERE name LIKE ?). The database engine treats the input as literal values, not as part of the command, preventing attackers from injecting malicious code. This is a fundamental best practice to avoid vulnerabilities like SQL injection, as it ensures that even if an attacker crafts a special input, it won't execute unintended commands.
