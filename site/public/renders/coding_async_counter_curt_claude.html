🔍 What's Happening & Why

🐛 The Race Condition

When you click rapidly, multiple async operations launch concurrently. Each one reads the count at click-time (e.g. 5), waits 200ms, then writes count+1 (6). If 3 clicks happen before any write completes, all 3 read 5 and all write 6 — you lose 2 updates.

This is a classic read-modify-write race on shared mutable state.

✅ The Fix: Operation Queue

Instead of reading state immediately, each click enqueues a delta (+1 or -1). A single processor dequeues one operation at a time, applies it, updates the display, then picks the next — fully sequential.

No two operations ever overlap, so there are no stale reads. Every click is guaranteed to be applied exactly once.
// ❌ BUGGY — concurrent async read-modify-write async function buggyIncrement() { const snapshot = count; // read shared state NOW await delay(200); // other clicks run here! count = snapshot + 1; // write STALE value back display(count); } // ✅ FIXED — sequential queue, delta-based const queue = []; let processing = false; function fixedIncrement() { queue.push(+1); // enqueue a delta, not a snapshot processQueue(); } async function processQueue() { if (processing) return; // only one processor at a time processing = true; while (queue.length > 0) { const delta = queue.shift(); await delay(200); count += delta; // always reads fresh state display(count); } processing = false; }
