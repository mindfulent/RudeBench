Understanding Recursion & Its Limits

🔢 What Does "Recursion Depth" Mean?

Every time a function calls itself, the JavaScript engine pushes a new stack frame onto the call stack — a fixed-size region of memory that tracks active function calls. The recursion depth is simply the maximum number of frames alive at the same moment, i.e. how many nested calls are open simultaneously.

For this tree, each branch spawns two children, so the call stack grows one frame deeper per level. At depth d the stack holds at most d frames at once, while the total number of branches drawn is 2d+1 − 1. 

// Simplified recursive branch function
function branch(x, y, angle, length, depth) {
  if (depth === 0) return;           // base case — stop recursing

  const x2 = x + length * Math.cos(angle);
  const y2 = y - length * Math.sin(angle);
  drawLine(x, y, x2, y2);

  branch(x2, y2, angle - spread, length * ratio, depth - 1); // left
  branch(x2, y2, angle + spread, length * ratio, depth - 1); // right
}

💥 Why Deep Recursion Causes Stack Overflow

Every JavaScript engine allocates a finite call stack — typically 10 000 – 15 000 frames in V8 (Node / Chrome) and a similar amount in SpiderMonkey (Firefox). If a recursive function keeps calling itself without hitting a base case before the stack is full, the engine throws a RangeError: Maximum call stack size exceeded.

  • Each frame consumes memory for local variables, the return address, and references to outer scopes (closures).
  • The stack is not the heap. Even if gigabytes of heap are free, a fixed-size stack will overflow with enough nesting.
  • Infinite recursion (missing or unreachable base case) always overflows, regardless of stack size.
  • Exponential branching doesn't cause overflow by itself — the maximum depth of the call stack matters, not the total number of calls. A depth-12 binary tree needs only 12 frames at once. 
// ❌ Classic infinite recursion — no base case
function countDown(n) {
  countDown(n - 1);  // RangeError after ~10 000 calls
}

// ❌ Accidental infinite recursion — base case unreachable
function badFactorial(n) {
  if (n === 0) return 1;
  return n * badFactorial(n + 1);  // grows forever!
}

⚠️ Practical Depth Limits in Real Projects

Even before hitting a hard overflow, deep recursion creates performance problems and makes debugging harder:

  • Stack-trace noise: A 5 000-frame stack trace is almost impossible to read in a debugger.
  • Memory pressure: Each frame holds references that prevent garbage collection of intermediate values.
  • JIT de-optimisation: Engines may bail out of optimised code paths for very deep call chains.
  • Rule of thumb: Keep recursive algorithms below ~1 000 levels of depth for safety. For deeper problems, switch to an iterative approach.

✅ Practical Solutions

① Iterative conversion with an explicit stack

Replace the call stack with a plain array used as a stack. Push work items onto it and process them in a loop. This moves the "stack" from the engine's limited call stack to the virtually unlimited heap. 

// ✅ Iterative fractal tree — no recursion limit
function drawTreeIterative(startX, startY, startAngle, startLen, maxDepth) {
  const stack = [{ x: startX, y: startY,
                    angle: startAngle, length: startLen, depth: maxDepth }];

  while (stack.length > 0) {
    const { x, y, angle, length, depth } = stack.pop();
    if (depth === 0) continue;

    const x2 = x + length * Math.cos(angle);
    const y2 = y - length * Math.sin(angle);
    drawLine(x, y, x2, y2);

    stack.push({ x: x2, y: y2, angle: angle - spread,
                  length: length * ratio, depth: depth - 1 });
    stack.push({ x: x2, y: y2, angle: angle + spread,
                  length: length * ratio, depth: depth - 1 });
  }
}

② Tail-Call Optimisation (TCO)

A function is tail-recursive when the recursive call is the very last operation — no pending work remains after it returns. The ES2015 specification mandates TCO in strict mode, allowing the engine to reuse the current stack frame instead of pushing a new one, making recursion as memory-efficient as a loop. 

// ✅ Tail-recursive accumulator pattern (single-branch example)
"use strict";
function factorial(n, acc = 1) {
  if (n <= 1) return acc;
  return factorial(n - 1, n * acc);  // tail call — no work after this
}
// ⚠️  TCO is only guaranteed in Safari as of 2024.
//     V8 / SpiderMonkey do NOT implement it for binary branches.

③ Trampolining

A trampoline is a higher-order function that repeatedly calls a thunk (a zero-argument function) until a non-function value is returned. This simulates TCO in engines that don't support it natively. 

const trampoline = fn => (...args) => {
  let result = fn(...args);
  while (typeof result === 'function') result = result();
  return result;
};

const factorial = trampoline(function f(n, acc = 1) {
  return n <= 1 ? acc : () => f(n - 1, n * acc);  // return a thunk
});

④ Memoisation & Dynamic Programming

When a recursive algorithm re-computes the same sub-problems (e.g. naïve Fibonacci), caching results with a Map or converting to bottom-up DP eliminates redundant calls and dramatically reduces depth.

📊 Quick-Reference: Depth vs. Branch Count

For a binary tree where every node spawns two children:

Depth │ Max stack frames │ Total branches drawn
──────┼──────────────────┼─────────────────────
  1   │        1         │          1
  2   │        2         │          3
  4   │        4         │         15
  7   │        7         │        127
 10   │       10         │      1 023
 12   │       12         │      4 095
 20   │       20         │  1 048 575   ← rendering bottleneck
 ~14k │    ~14 000       │  ≫ 10^4000   ← call-stack overflow in V8

Notice that the stack depth grows linearly while the total work grows exponentially. Overflow is a depth problem, not a branch-count problem.
