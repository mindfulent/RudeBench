🔬 What's Actually Happening — The Bugs Explained

The Setup

Five async "tasks" simulate work by firing progress updates via setTimeout. Each task has a random duration and tick rate. They all run concurrently. The goal: a master progress bar showing overall completion (0–100%).

Bug #1 — Read-Modify-Write Race (the classic)

The buggy version stores a single shared sharedProgress number. Every task callback does: 

// Task A fires. Reads stale value, adds its delta, writes back.
sharedProgress += myDelta;   // Task B may have ALSO read the old value
updateBar(sharedProgress);   // one write silently stomps the other

In a single-threaded JS engine this still causes logical races because setTimeout callbacks interleave. Task B reads sharedProgress = 40, Task A also reads 40, both add their delta, both write — one update is silently lost. The bar jumps backward or stalls.

Bug #2 — Stale Closure Capture

let taskProgress = 0;          // captured by closure at schedule time
setTimeout(function tick() {
  taskProgress += increment;   // fine locally...
  sharedProgress += increment; // but sharedProgress was already mutated
                               // by OTHER tasks between ticks — stale!
  setTimeout(tick, delay);
}, delay);

Bug #3 — No Completion Guard (phantom updates)

// Task finishes at 100% but its setTimeout chain keeps firing
if (taskProgress >= 100) return;  // too late — already scheduled next tick
sharedProgress += delta;           // pushes master bar PAST 100%

The master bar can exceed 100% or oscillate near the end.

The Fix — Derived State, No Shared Mutation

// Each task owns ONLY its own slice. Never touches shared state.
taskState[id].progress = clamp(newVal, 0, 100);

// Master recomputes from source of truth on every tick.
function getMasterProgress() {
  const total = taskState.reduce((sum, t) => sum + t.progress, 0);
  return total / taskState.length;   // pure derivation, no mutation
}
  • BUGGY: shared mutable accumulator — multiple writers, no lock
  • FIXED: each task writes only its own isolated slot
  • BUGGY: master progress is state — can be corrupted
  • FIXED: master progress is derived — always consistent
  • BUGGY: no cancellation — phantom ticks after completion
  • FIXED: each task tracks its own timer ID and clears it
  • BUGGY: delta accumulation drifts — tasks double-count increments
  • FIXED: absolute values only — progress = newAbsoluteValue

Why This Kills Production Systems

Replace "progress bar" with "order quantity", "account balance", "retry counter", or "connection pool size". The same read-modify-write race on shared mutable state causes inventory overselling, double charges, infinite retry storms, and pool exhaustion. The fix is always the same: isolate writes, derive reads.
