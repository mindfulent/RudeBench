🔍 What's actually happening and why it matters

JavaScript is single-threaded, but setTimeout callbacks are scheduled asynchronously. When multiple callbacks fire in rapid succession they all read the same stale snapshot of shared state before any of them has had a chance to write their update back — a classic read-modify-write race condition.

Buggy pattern: a single completedTasks counter is incremented by every callback. Because JS is cooperative (not preemptive), the race doesn't corrupt a single increment — but the overall percentage is recomputed from a shared total that tasks overwrite without coordination. The real-world version of this bug shows up when you add debouncing, batching, or network calls: two responses arrive, both read total = 3, both write total = 4, and you silently lose one unit of work. The bar jumps, stalls, or never reaches 100%.

Fixed pattern: each task owns its own isolated progress slot in an array. The overall percentage is derived by reducing the array — a pure read with no shared write target. No task can corrupt another task's slot. The aggregate is always computed fresh from the ground truth, so the bar is always consistent regardless of scheduling order.

🐛 Bug Step 1 Tasks A, B, C all start. Each reads completedTasks = 0.
🐛 Bug Step 2 Task A finishes, writes completedTasks = 1. Bar → 33%.
🐛 Bug Step 3 Tasks B & C finish almost simultaneously. Both read stale value, both write completedTasks = 2. Bar → 67% — one task lost!
✅ Fix Each task writes only to progress[i]. Overall = sum(progress) / total. No shared write target. Always correct.
