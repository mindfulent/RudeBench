Understanding the Concurrency Bug

This page demonstrates a classic "race condition" concurrency bug using a progress bar as an example. Both sections run simulated "tasks" that update a shared progress percentage.

The problem occurs when multiple tasks try to update the same shared variable (e.g., `progressPercentage`) at roughly the same time. If a task reads the current value, another task also reads the *same old value* before the first task has written its new value, they will both base their calculation on outdated information. This leads to lost updates, skipped percentages, or even exceeding 100%.

Buggy Implementation: Updates the shared percentage directly, allowing race conditions.

// Buggy update: currentProgress = sharedProgressPercentage; newProgress = currentProgress + updateAmount; // Both tasks might read the same currentProgress sharedProgressPercentage = newProgress; // One task overwrites the other's update

Corrected Implementation: Uses a promise chain (acting like a simple mutex or critical section) to ensure only one update operation can proceed at a time. Each update waits for the previous one to complete, guaranteeing that it reads the *most current* value.

// Corrected update: // The 'currentUpdatePromise' ensures that this new update function // only starts *after* the previous update function has finished. currentUpdatePromise = currentUpdatePromise.then(() => { const currentProgress = sharedProgressPercentage; const newProgress = Math.min(100, currentProgress + updateAmount); sharedProgressPercentage = newProgress; return newProgress; // Pass the new value to the next promise in the chain });
