⚠️ The Developer's Claim vs. Reality

A developer says: "I use === for all price comparisons and it works perfectly." This is dangerously wrong. JavaScript (like all IEEE 754 languages) stores decimals in binary floating-point, which cannot exactly represent most decimal fractions. The number 0.1 in binary is an infinitely repeating fraction — like 1/3 in decimal — so it gets rounded. When you add several rounded numbers together, the rounding errors accumulate unpredictably.

// What every JS developer needs to see in their console:
0.1 + 0.2                    → 0.30000000000000004  // NOT 0.3 !
0.1 + 0.2 === 0.3           → false         // === fails silently
4.99 + 4.99 + 0.02         → 10.000000000000002
(4.99 + 4.99 + 0.02) === 10   → false         // discount check FAILS
// ─────────────────────────────────────────────────────
// Fix: work in integer cents, convert only for display:
Math.round(4.99 * 100) + Math.round(4.99 * 100)   → 998  // exact cents
998 / 100                   → 9.98  // safe display

📚 Deep Dive: Why Floating-Point Equality Fails for Financial Calculations

🔴 The Problem

IEEE 754 double-precision floats use 64 bits to represent numbers. Most decimal fractions like 0.1, 0.01, 4.99 have no exact binary representation — they are stored as the closest approximation. Each arithmetic operation can introduce a tiny rounding error. After several additions, these errors compound into visible discrepancies. Using === then compares the error-laden result against a "clean" expected value, and they will often not match even when mathematically they should.

// Proof of representation error:
0.1.toPrecision(20)
// → "0.10000000000000000555"
0.2.toPrecision(20)
// → "0.20000000000000001110"
0.1 + 0.2
// → 0.30000000000000004
(0.1 + 0.2) === 0.3 // false !

🟡 Why It Happens

Binary fractions work like decimal fractions: just as 1/3 cannot be written exactly in base-10 (it becomes 0.333…), the fraction 1/10 cannot be written exactly in base-2. The CPU stores the nearest representable value. toFixed(2) hides this by rounding for display, but the underlying float is still wrong. Discount checks, tax calculations, and running totals all operate on the raw float — so the bug is real even if the display looks correct.

// toFixed hides but doesn't fix:
let t = 0.1 + 0.2;
t.toFixed(2) // → "0.30" looks fine
t === 0.3 // → false still broken
// And it gets worse with more items:
let s = 0;
for(let i=0;i<10;i++) s += 0.1;
s === 1.0 // → false
s        // → 0.9999999999999999

🟢 The Correct Solutions

Option 1 — Integer cents: Multiply all prices by 100 immediately, work entirely in integers (which ARE exact in float64 up to 2^53), then divide by 100 only for final display. Integer === is always exact.

Option 2 — Epsilon comparison: Instead of a === b, use Math.abs(a - b) < 0.001 (a tolerance / epsilon).

Option 3 — Decimal library: Use decimal.js or big.js for arbitrary-precision decimal arithmetic.

// ✅ Integer cents approach:
const toCents = p => Math.round(p * 100);
const toDisplay = c => (c / 100).toFixed(2);
let total = 0; // integer cents
total += toCents(4.99); // 499
total += toCents(4.99); // 998
total === 998 // → true ✅
// ✅ Epsilon approach:
Math.abs(total-998) < 0.001 // true
