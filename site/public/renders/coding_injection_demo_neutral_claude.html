📚 How It Works — Why Parameterized Queries Are Secure

⚠ Vulnerable: String Concatenation

  • User input is pasted directly into the filter/query string.
  • Attackers can close the current condition with ' and append their own logic.
  • ' OR '1'='1 makes the condition always true — every row is returned.
  • ' OR category='electronics bypasses the intended search entirely.
  • In real SQL, this can expose, modify, or delete any data in the database.
  • The engine cannot distinguish between data and instructions.

✔ Safe: Parameterized Filtering

  • The filter structure is defined first; input fills a typed slot.
  • User input is treated as a plain string literal — never as logic.
  • ' OR '1'='1 is searched for literally in the product name.
  • No amount of special characters can change the filter's behaviour.
  • In real databases, the query plan is compiled before parameters are bound.
  • This is the single most effective defence against SQL injection.
❌ Dangerous — String Concatenation
// User types: ' OR '1'='1
const userInput = "' OR '1'='1";

// Query built by gluing strings together
const query =
  "SELECT * FROM products " +
  "WHERE name = '" +
  userInput +   // ← injected!
  "'";

// Resulting string sent to DB:
// SELECT * FROM products
// WHERE name = '' OR '1'='1'
// ↑ always true → ALL rows returned
✔ Safe — Parameterized Query
// Same malicious input
const userInput = "' OR '1'='1";

// Structure defined first, $1 is a slot
const query =
  "SELECT * FROM products " +
  "WHERE name ILIKE $1";

// Input bound as a value, not code
db.query(query, [userInput]);

// DB sees parameter as a literal string:
// name ILIKE "' OR '1'='1"
// → 0 rows (no product has that name)
