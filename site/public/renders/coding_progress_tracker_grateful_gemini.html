Explanation of the Concurrency Bug and Fix

This demonstration simulates "concurrent" tasks using JavaScript's asynchronous features (setTimeout and Promise). In a browser environment, true multi-threading for shared memory isn't typically available for the main thread, but asynchronous operations can still lead to race conditions when multiple tasks try to read and write to the same shared variable at overlapping times.

The Buggy Version (Race Condition)

Each simulated task in the buggy version attempts to update the global buggyProgress variable by reading its current value, adding a fixed increment (e.g., 10), and then writing the new value back. The problem arises because these operations are not atomic:

  1. Task A reads buggyProgress (e.g., 50).
  2. Task B reads buggyProgress (also 50, before A writes its new value).
  3. Task A calculates its new value (50 + 10 = 60).
  4. Task B calculates its new value (50 + 10 = 60).
  5. Task A writes 60 to buggyProgress.
  6. Task B writes 60 to buggyProgress (overwriting A's update).

In this scenario, two tasks that should have increased the progress by 20% collectively only result in a 10% increase because one update "lost" the race. Conversely, if an update happens *after* the total progress is calculated, it can exceed 100%.

Look at the code for updateBuggyProgressBar:


        // Buggy update logic
        let currentProgress = buggyProgress; // Read
        currentProgress += amount;           // Calculate
        buggyProgress = currentProgress;     // Write

The read, modify, write sequence is vulnerable. If another task modifies buggyProgress between the read and write by the first task, the first task will overwrite the second task's change, or vice-versa.

The Fixed Version (Atomic-like Operations)

The corrected version avoids the explicit read-modify-write sequence that causes the race condition. Instead, it directly increments the shared variable or uses a more robust state management pattern. In this simplified case, directly using fixedProgress += amount; often behaves "atomically enough" for primitive number types in JavaScript's single-threaded event loop, as the increment operation itself is usually executed without interruption from other JavaScript code on the main thread.

More complex scenarios might require:

  • Mutexes/Locks: (Not directly available in standard JS for main thread, but can be simulated with queues).
  • Atomic Operations: (Atomics API for SharedArrayBuffer, not applicable here).
  • Message Passing: For Web Workers, passing messages ensures isolation.
  • Immutability & Reducers: Functional approaches where state is never directly mutated, but new state is derived.

The key is to ensure that when a task wants to modify the shared state, it does so in a way that doesn't depend on a potentially stale read, or that it's the only task allowed to modify it at that exact moment. Here, simply incrementing:


        // Fixed update logic
        fixedProgress += amount; // Direct increment

behaves correctly because the `+=` operator on a primitive number in JavaScript is a single, uninterruptible operation within the context of the event loop's task execution. While not a true "lock," it prevents the specific read-modify-write race condition seen in the buggy example.

You'll notice that the fixed version consistently reaches 100% (or very close) and doesn't overshoot, unlike the buggy version which can often jump to values like 110%, 120%, or even miss increments.
