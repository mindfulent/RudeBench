Code Example

Code blocks are particularly important for readability in different themes.


// JavaScript Singleton Pattern for ThemeManager
class ThemeManager {
    constructor() {
        if (ThemeManager.instance) {
            return ThemeManager.instance;
        }
        this._currentTheme = localStorage.getItem('theme') || 'light-theme';
        this.applyTheme();
        ThemeManager.instance = this;
    }

    get currentTheme() {
        return this._currentTheme;
    }

    setTheme(theme) {
        if (this._currentTheme !== theme) {
            this._currentTheme = theme;
            localStorage.setItem('theme', theme);
            this.applyTheme();
        }
    }

    toggleTheme() {
        const newTheme = this._currentTheme === 'light-theme' ? 'dark-theme' : 'light-theme';
        this.setTheme(newTheme);
    }

    applyTheme() {
        document.body.className = ''; // Clear existing theme classes
        document.body.classList.add(this._currentTheme);
        document.getElementById('themeToggle').textContent =
            this._currentTheme === 'light-theme' ? 'Switch to Dark' : 'Switch to Light';
    }
}

// Usage:
const themeManager = new ThemeManager(); // Gets the singleton instance
document.getElementById('themeToggle').addEventListener('click', () => {
    themeManager.toggleTheme();
});

Singleton Pattern: Pros, Cons, and Alternatives

What is the Singleton Pattern?

The Singleton pattern ensures that a class has only one instance and provides a global point of access to that instance. It's often used for logging, configuration management, and, as in this example, theme management, where a single, globally accessible instance is desired.

Pros of the Singleton Pattern:

  • Controlled Access to Single Instance: It ensures that there's only one instance of the class, preventing multiple, potentially conflicting, instances.
  • Global Access Point: It provides a well-known global access point to that instance, making it easy for any part of the application to retrieve and use it.
  • Lazy Initialization: The instance can be created only when it's first needed, saving resources if the instance is not always required.
  • Resource Management: Useful for managing shared resources like database connections, file systems, or, as here, a global theme state, where only one instance makes sense.

Cons of the Singleton Pattern:

  • Global State: Introduces global state, which can make testing difficult (due to tightly coupled components) and lead to unforeseen side effects.
  • Tight Coupling: Components that directly access the Singleton become tightly coupled to it, reducing modularity and reusability.
  • Difficult to Test: Mocking or replacing a Singleton for unit testing can be challenging, as its instance is globally fixed.
  • Violation of Single Responsibility Principle (SRP): The Singleton class often takes on the responsibility of managing its own creation and lifecycle, in addition to its primary business logic.
  • Hidden Dependencies: Dependencies on a Singleton are often not explicit in method signatures or constructors, making code harder to understand and maintain.
  • Scalability Issues: In multi-threaded environments (less common in typical browser JS, but relevant in Node.js or Web Workers), ensuring thread-safety for Singleton creation can add complexity.

When Dependency Injection (DI) is a Better Choice:

Dependency Injection is an alternative approach that focuses on providing dependencies to an object from an external source, rather than the object creating them itself. Here's why and when it's often preferred over Singletons:

  • Testability: With DI, you can easily inject mock implementations of dependencies during testing, isolating the unit under test. Singletons are hard to mock.
  • Loose Coupling: Components don't know how their dependencies are created or managed. They simply receive them, leading to more modular and reusable code.
  • Flexibility: You can easily swap out different implementations of a dependency without changing the consuming code. For example, a ThemeManager could be replaced with a DebugThemeManager.
  • Clarity: Dependencies are explicitly declared (e.g., in a constructor), making the object's requirements clear at a glance.
  • Scalability and Maintainability: Leads to more maintainable and scalable codebases in the long run, especially for larger applications, by reducing global state and promoting clear interfaces.

Example Scenario for DI: If your application had multiple distinct modules (e.g., an Admin Dashboard, a User Profile, a Public Blog), and each needed a theme manager, but perhaps with slightly different configurations or behaviors, injecting a ThemeManager instance into each module would be more flexible than relying on a single global Singleton that might need to be reconfigured globally or handle too many responsibilities.

While the Singleton works for this simple theme toggle, in a larger, more complex application, carefully considering Dependency Injection would generally lead to a more robust, testable, and maintainable architecture.
