📚 What's happening under the hood?

🐛 The Bug — Shared Mutable State

JavaScript is single-threaded, but async tasks interleave between await / setTimeout boundaries. Each task reads the current value of sharedProgress, adds its chunk, and writes back — but another task may have already changed it between the read and the write:

// BUGGY — classic read-modify-write race
let sharedProgress = 0;   // shared by ALL tasks

async function buggyTask(name, steps) {
  for (let i = 0; i < steps; i++) {
    await delay(randomMs());
    sharedProgress += 10;  // ← RACE: read then write
    // Task B may write between our read & write!
    updateBar(sharedProgress); // may show 110%, 130%…
  }
}
  • Two tasks read 50 simultaneously
  • Both add 10 → both write 60
  • One increment is lost, or both arrive late and total exceeds 100%

✅ The Fix — Isolated Slices + Atomic Read

Give each task its own private progress slot. The displayed value is always the sum of all slots, computed fresh each render tick. No two tasks ever touch the same memory location:

// FIXED — each task owns its own slice
const slices = {};          // isolated per-task state

function totalProgress() {
    return Object.values(slices).reduce((a,b) => a+b, 0);
}

async function fixedTask(name, share, steps) {
  slices[name] = 0;         // private slot, no sharing
  for (let i = 0; i < steps; i++) {
    await delay(randomMs());
        slices[name] += share / steps; // only writes own slot
    updateBar(totalProgress()); // always accurate sum
  }
}
  • Each task writes only its own slot — zero contention
  • The sum is computed atomically in a single expression
  • Total is mathematically guaranteed to stay ≤ 100%
