⚠️ Why Singletons Are Often Terrible

  • Global mutable state — anything, anywhere can change it with zero traceability
  • Hidden dependencies — consumers don't declare what they need, they just reach in and grab it
  • Untestable — you can't inject a mock; the singleton IS the dependency
  • Tight coupling — every consumer is married to one concrete implementation
  • Concurrency nightmares — shared mutable state across threads is a race condition waiting to happen
  • Violates SRP — the class manages its own lifecycle AND its own logic
  • Order-of-initialization bugs — which singleton initialized first? Nobody knows until it crashes
  • Impossible to reset between tests — state leaks from one test into the next

✅ Better Alternatives

  • Dependency Injection — pass dependencies explicitly; swap them freely; test with mocks
  • React Context / Provide-Inject — scoped, declarative, framework-native state sharing
  • Module-level constants — if the value truly never changes, a plain exported const is simpler and honest
  • Service Locator (with caution) — at least centralises the registry; still has hidden-dependency smell
  • Event Bus / Pub-Sub — decoupled communication without shared state ownership
  • State management libraries — Redux, Zustand, Pinia — explicit, traceable, devtools-friendly
  • Composition Root — wire everything together at startup; nothing self-instantiates
