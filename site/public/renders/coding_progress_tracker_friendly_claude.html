🔬 Understanding the Shared State Concurrency Bug

🐛 Why the Buggy Version Fails

Each task reads the current value of sharedProgress, adds its chunk, then writes back. But because JavaScript's event loop interleaves microtasks and macrotasks, multiple tasks can read the same stale value before any of them write back:

  • Task A reads progress = 20
  • Task B reads progress = 20 (same snapshot!)
  • Task A writes progress = 30
  • Task B writes progress = 30 — Task A's update is lost!

Result: the final value is wrong, updates are skipped, and in some runs the bar can even exceed 100%.

✅ How the Fixed Version Works

The fix uses a Promise-based serial queue (mutex pattern). Each task must acquire a lock before touching the shared variable. If the lock is held, the task waits until it's released:

  • Only one task at a time can read-modify-write
  • Other tasks queue up and wait their turn
  • No stale reads — every task sees the latest committed value
  • Final result is always exactly 100%

This is equivalent to a mutex / critical section in multi-threaded languages like Java or C++, adapted for JS's async/await model using a chained Promise queue.
