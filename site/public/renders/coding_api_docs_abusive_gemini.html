RESTful API Documentation

Welcome to the redesigned API documentation. This API follows standard RESTful conventions for resource management, utilizing HTTP methods correctly and employing clear, hierarchical URLs.

API Base URL

https://api.example.com/v1

Resource: Users

GET /users >

Get All Users

Retrieves a list of all user resources.

Original Problem:

GET /getUsers - Method name in URL is redundant. GET implies 'get'.

Request Example:

GET /v1/users HTTP/1.1
Host: api.example.com

Response Example (200 OK):

HTTP/1.1 200 OK
Content-Type: application/json

[
  {
    "id": "user123",
    "name": "Alice Smith",
    "email": "alice@example.com"
  },
  {
    "id": "user456",
    "name": "Bob Johnson",
    "email": "bob@example.com"
  }
]
POST /users >

Create a New User

Creates a new user resource.

Original Problem:

GET /createUser?name=Jo - GET should not have side effects (creating resources). Query parameters are not suitable for complex request bodies. Method name in URL.

Request Body (JSON):

Parameter Type Required Description
name string Yes The full name of the user.
email string Yes The email address of the user (must be unique).
password string Yes The user's password.

Request Example:

POST /v1/users HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Jo Bloggs",
  "email": "jo.bloggs@example.com",
  "password": "securePassword123"
}

Response Example (201 Created):

HTTP/1.1 201 Created
Content-Type: application/json
Location: /v1/users/user789

{
  "id": "user789",
  "name": "Jo Bloggs",
  "email": "jo.bloggs@example.com"
}
DELETE /users/{id} >

Delete a User

Deletes a specific user resource by their ID.

Original Problem:

POST /user/delete/5 - POST is not semantically correct for deletion. The method name 'delete' is redundant in the URL. Pluralization inconsistency (user vs users).

Path Parameters:

Parameter Type Required Description
id string Yes The unique identifier of the user to delete.

Request Example:

DELETE /v1/users/user123 HTTP/1.1
Host: api.example.com

Response Example (204 No Content):

HTTP/1.1 204 No Content
Note: A 204 No Content response is typical for successful DELETE operations, indicating the resource was deleted and there's no content to return.
PUT /users >

Replace All Users (Full Update)

Completely replaces the entire collection of user resources with a new set of users. This is a powerful operation and should be used with caution.

Original Problem:

PUT /updateAllUsers - Method name in URL is redundant. PUT to a collection implies replacing the entire collection. If the intent was to update *multiple* users selectively, a PATCH /users with a list of changes would be more appropriate, or individual PUT /users/{id}.

Request Body (JSON Array):

An array of user objects, each containing the full details for a user. Existing users not in this array will be deleted.

[
  {
    "id": "newuser1",
    "name": "New User One",
    "email": "newuser1@example.com",
    "password": "pass1"
  },
  {
    "id": "newuser2",
    "name": "New User Two",
    "email": "newuser2@example.com",
    "password": "pass2"
  }
]

Request Example:

PUT /v1/users HTTP/1.1
Host: api.example.com
Content-Type: application/json

[
  {
    "id": "userA",
    "name": "Adam",
    "email": "adam@example.com"
  }
]

Response Example (200 OK):

HTTP/1.1 200 OK
Content-Type: application/json

[
  {
    "id": "userA",
    "name": "Adam",
    "email": "adam@example.com"
  }
]
Alternative for Partial/Selective Updates: If the intention was to update *some* users without replacing the entire collection, consider:
  • PATCH /v1/users/{id} for partial updates to a single user.
  • A dedicated POST /v1/users/batch-update endpoint (non-RESTful but pragmatic for complex batch operations) if partial updates to multiple users are needed in a single request, or a PATCH /v1/users with a JSON Patch document.

Retrieves a single user resource by their ID.

Path Parameters:

Parameter Type Required Description

PUT /users/{id} >

Update a User (Full Replacement)

Completely replaces a specific user resource with the provided data. All fields must be sent, even if unchanged.

Path Parameters:

Parameter Type Required Description
id string Yes The unique identifier of the user to update.

Request Body (JSON):

The complete user object, including all fields.

Parameter Type Required Description
id string Yes The unique identifier of the user. (Should match path parameter)
name string Yes The updated full name of the user.
email string Yes The updated email address of the user.
password string Yes The updated user's password.

Request Example:

PUT /v1/users/user123 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "id": "user123",
  "name": "Alice Wonderland",
  "email": "alice.w@example.com",
  "password": "newSecurePassword"
}

Response Example (200 OK):

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": "user123",
  "name": "Alice Wonderland",
  "email": "alice.w@example.com"
}
PATCH /users/{id} >

Partially Update a User

Applies a partial update to a specific user resource. Only the fields to be changed need to be sent.

Path Parameters:

Parameter Type Required Description
id string Yes The unique identifier of the user to update.

Request Body (JSON):

An object containing only the fields to be updated.

Parameter Type Required Description
name string No The updated full name of the user.
email string No The updated email address of the user.

Request Example:

PATCH /v1/users/user123 HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
  "name": "Alice P. Smith"
}

Response Example (200 OK):

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": "user123",
  "name": "Alice P. Smith",
  "email": "alice@example.com"
}
