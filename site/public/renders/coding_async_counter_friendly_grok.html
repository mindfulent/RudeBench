Race Condition Demonstration

This page shows a simple counter widget with two versions: a buggy one that suffers from a race condition and a corrected one that avoids it. A race condition occurs when multiple asynchronous operations access and modify a shared variable at the same time, leading to unpredictable results.

Buggy Version

How it works: Each button click schedules an asynchronous update using setTimeout to increment the counter after a short delay. If you click rapidly, multiple updates can overlap, causing the counter to read the same value multiple times before writing, which results in lost increments (e.g., if you click 5 times, the counter might only go up by 3).

Why the race condition happens: In asynchronous code like this, each setTimeout callback runs independently. When clicks happen quickly, they might all read the current counter value (e.g., 0) at nearly the same time, increment it individually, and then write back the same new value (e.g., 1), overwriting each other. This leads to the counter falling behind or showing incorrect values.

Corrected Version

How it works: This version also uses setTimeout for asynchronous updates, but it processes increments sequentially using a promise chain. This ensures that each update waits for the previous one to finish, preventing overlaps and maintaining accuracy even with rapid clicks.

How it fixes the race condition: By chaining promises, we create a queue for updates. Each click adds to the end of the queue, so operations execute one at a time, avoiding concurrent access to the shared variable. This makes the counter reliable.
