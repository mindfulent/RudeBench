Explanation (short)

  • Bug: multiple asynchronous workers perform a read-modify-write on a shared variable without synchronization. If two workers read the same base value, both compute a new value, and the slower one overwrites the faster one's much larger update, progress can be lost or even appear to go backward when values are computed from stale reads.
  • Result in demo: run the buggy panel several times and you may see progress occasionally jump backward or stagnate because a stale computed value overwrote a more advanced state.
  • Fix approaches shown:
    • Serialized update queue (shown): each proposed update is enqueued and runs one at a time; each sees the latest committed value before computing the next committed state.
    • Alternative fixes: mutex/lock, atomics in SharedArrayBuffer (in multi-threaded contexts like Web Workers), or designing updates to be commutative/mergeable (use max() for monotonic counters) so lost updates are harmless.
