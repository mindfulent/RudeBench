REST API Design Review

Endpoints Analysis

1. GET /getUsers

Problem: The endpoint should not have a verb ("get") in its path. RESTful APIs use HTTP methods to indicate the action.
Correction: Use GET /users to retrieve a list of users.

2. POST /user/delete/5

Problems:
  • The HTTP method is incorrect. DELETE should be used for deleting resources.
  • The endpoint should not have a verb ("delete") in its path.
Correction: Use DELETE /users/5 to delete a user with ID 5.

3. GET /createUser?name=Jo

Problems:
  • The HTTP method is incorrect. POST or PUT should be used for creating resources.
  • The endpoint should not have a verb ("create") in its path.
Correction: Use POST /users with a JSON body containing the user data (e.g., {"name": "Jo"}) to create a new user.

4. PUT /updateAllUsers

Problems:
  • The endpoint is too broad and implies updating all users at once.
  • The HTTP method is correct (PUT), but the endpoint could be improved.
Correction: Use PATCH /users or PUT /users/{id} to update a specific user or a subset of users. For example, PUT /users/5 with a JSON body containing the updated user data.

Summary of Corrections

  • Use resource-based endpoint paths (e.g., /users instead of /getUsers)
  • Use correct HTTP methods:
    • GET: retrieve resources
    • POST: create resources
    • PUT/PATCH: update resources
    • DELETE: delete resources
  • Avoid verbs in endpoint paths
