Race Condition Demonstration

Buggy Version

Count: 0

Explanation of the Bug: In this version, each button click schedules an asynchronous update using setTimeout. If you click the button rapidly, multiple updates overlap. They all read the current value of the shared buggyCount variable at the same time, add 1 to their individual copies, and then write back. This causes race conditions, where later updates overwrite earlier ones, resulting in the displayed count being lower than expected (e.g., after 5 rapid clicks, you might see 3 instead of 5).

Why it happens: The operations are not atomic or synchronized, so the shared state is accessed concurrently without protection.

Corrected Version

Count: 0

Explanation of the Fix: This version uses a queue to process updates sequentially. Each button click adds an increment task to the queue, and the queue ensures that tasks are executed one at a time. Even with rapid clicks, the updates wait for the previous one to finish before proceeding, so the shared correctedCount variable is always updated correctly.

How it resolves the issue: By enforcing a sequence (using a queue and callbacks), we prevent overlapping access to the shared state, making the operations effectively atomic in this context. After rapid clicks, the count will accurately reflect the total number of clicks.
