Why regex is fundamentally broken for email validation

RFC 5321 / 5322 allow quoted local parts

"john doe"@example.com is a perfectly legal email address. Spaces, @ signs, and most special characters are valid inside double quotes. Nearly every regex rejects this outright.

Comments are part of the spec

john(comment)@example.com and john@(comment)example.com are valid per RFC 5322. No practical regex handles parenthetical comments in either the local part or the domain.

IP address literals are valid domains

user@[192.168.1.1] and user@[IPv6:::1] are legitimate. Most regexes only accept dot-separated domain names and silently reject these valid forms.

Internationalised domains (IDN) and Unicode local parts

用户@例子.广告 is a real email address. RFC 6531 extends SMTP for UTF-8. Any ASCII-only regex rejects millions of valid addresses used across Asia, the Middle East, and beyond.

TLD length is not capped at 4 or 6 characters

Regexes like /\.[a-z]{2,4}$/ reject user@example.photography or user@example.international. ICANN new gTLDs can be up to 63 characters long.

Regex cannot verify the domain actually exists

fake@definitelynotreal99999.com passes every regex perfectly. The only way to know a domain is real is a DNS MX/A lookup — something regex is physically incapable of doing.

The RFC 5322 compliant regex is 6,000+ characters long

The theoretically correct regex for RFC 5322 is so large it is practically unmaintainable, still doesn't handle all edge cases, and causes catastrophic backtracking on malicious input — a ReDoS attack vector.

Syntactic validity ≠ deliverability

Even a perfectly valid, correctly formatted email address can bounce. The mailbox might not exist, the server might be down, or the domain may have no MX record. Regex tells you none of this.

Practical alternatives that actually work

📨 Send a verification email

The only proof an address is real and owned by the user. Every serious auth system does this. No exceptions.

Gold standard

🌐 DNS MX lookup

Check that the domain has a mail exchanger record. Catches fake domains instantly. Do this server-side.

Server-side

📦 email-validator (npm)

Handles RFC edge cases, does optional DNS checks, actively maintained. Far better than rolling your own regex.

🐍 validate_email (Python)

python-email-validator on PyPI. RFC-aware, Unicode-aware, optional DNS checking built in.
