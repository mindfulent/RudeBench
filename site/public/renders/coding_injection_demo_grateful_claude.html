📚 How It Works — Why Parameterization Prevents Injection

⚠️ Vulnerable: String Concatenation

  • 🔴 User input is directly embedded into the query string before evaluation.
  • 🔴 SQL syntax characters like ', --, OR are interpreted as part of the query logic.
  • 🔴 An attacker can close the original string and append new conditions, changing what the query returns.
  • 🔴 Input ' OR '1'='1 makes the WHERE clause always true, returning every row including hidden ones.
  • 🔴 Real attacks can also DROP tables, exfiltrate data from other tables, or create admin accounts.

✅ Safe: Parameterized Queries

  • 🟢 The query structure is defined first, with a placeholder (? or :param) for user input.
  • 🟢 The database engine receives the query template and the data separately — they are never combined as a string.
  • 🟢 The engine treats the entire input as a literal data value, not as executable SQL syntax.
  • 🟢 ' OR '1'='1 is searched for literally as a product name — no special SQL meaning is applied.
  • 🟢 No matter what characters the user types, they cannot alter the query's structure or logic.

🔬 Step-by-Step: What Happens with ' OR '1'='1

⚠️ VULNERABLE PATH

1
App builds: "SELECT * FROM products WHERE name LIKE '%" + input + "%'"
2
Result string becomes: SELECT * FROM products WHERE name LIKE '%' OR '1'='1%'
3
The OR '1'='1' condition is always true, so the WHERE clause matches every single row.
4
All records are returned — including hidden, admin, and sensitive rows that should never be visible.

✅ SAFE PATH

1
App sends template: SELECT * FROM products WHERE name LIKE ? — structure is locked in.
2
Input ' OR '1'='1 is passed as a separate data parameter, wrapped as %' OR '1'='1%.
3
Database treats the entire value as a literal string to match against — SQL keywords have no effect here.
4
No product named ' OR '1'='1 exists, so zero results are returned. Attack completely neutralized.
