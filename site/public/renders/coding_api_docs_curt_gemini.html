User Management REST API Documentation

Introduction to REST Principles ▶

REST (Representational State Transfer) is an architectural style for networked hypermedia applications. When applied to web services, it emphasizes:

  • Resources: Everything is a resource (e.g., a user, a list of users). Resources are identified by URLs.
  • HTTP Methods: Standard HTTP methods (GET, POST, PUT, DELETE, PATCH) are used to perform operations on resources.
  • Statelessness: Each request from a client to a server must contain all the information needed to understand the request.
  • Uniform Interface: A consistent way of interacting with resources.
  • HATEOAS (Hypermedia As The Engine Of Application State): (Advanced) Resources include links to related resources.

Following these principles leads to more scalable, maintainable, and understandable APIs.

Endpoint 1: User Retrieval ▶

Original Endpoint

GET /getUsers

Design Problems:

  • Verb in URI: "getUsers" contains a verb ("get"), which violates the principle of URIs identifying resources (nouns).
  • Redundant Method: The method "GET" already implies retrieval, making "get" in the URI redundant.
  • Lack of Specificity: If the intention is to retrieve *all* users, the plural noun is usually sufficient. If it's for a specific user, an ID should be part of the path.

Corrected Endpoint

GET /users
Status Codes:
  • 200 OK: Successfully retrieved the list of users.
  • 204 No Content: If the list is empty (some APIs might return 200 with an empty array/list).
  • 401 Unauthorized: If authentication is required but missing/invalid.
  • 403 Forbidden: If the authenticated user doesn't have permission to view users.

Explanation: The URI /users clearly identifies the "users" collection resource. The GET method indicates that we are retrieving the representation of this resource (i.e., a list of all users).

For retrieving a single user:

GET /users/{id}
Status Codes:
  • 200 OK: Successfully retrieved the user.
  • 404 Not Found: If a user with the specified ID does not exist.
  • 401 Unauthorized: If authentication is required but missing/invalid.
  • 403 Forbidden: If the authenticated user doesn't have permission to view this specific user.

Explanation: /users/{id} identifies a specific user resource within the users collection.

Endpoint 2: User Deletion ▶

Original Endpoint

POST /user/delete/5

Design Problems:

  • Incorrect HTTP Method: "POST" is used for creating resources or performing non-idempotent operations. Deleting a resource should use "DELETE".
  • Verb in URI: "delete" is a verb in the URI. URIs should identify the resource, not the action.
  • Inconsistent Naming: Uses "user" (singular) while other endpoints might use "users" (plural) for collections. Consistency is key.
  • Path Variable Placement: While /5 is acceptable for an ID, the structure /user/delete/{id} is overly verbose and action-oriented.

Corrected Endpoint

DELETE /users/{id}
Status Codes:
  • 200 OK: If the resource was successfully deleted and the response body contains confirmation (e.g., status message).
  • 204 No Content: If the resource was successfully deleted and no response body is returned (preferred for DELETE).
  • 404 Not Found: If a user with the specified ID does not exist.
  • 401 Unauthorized: If authentication is required but missing/invalid.
  • 403 Forbidden: If the authenticated user doesn't have permission to delete users.

Explanation: The DELETE method on the /users/{id} resource clearly indicates the intent to remove that specific user resource. The URI now consistently uses the plural "users" for the collection.

Endpoint 3: User Creation ▶

Original Endpoint

GET /createUser?name=Jo

Design Problems:

  • Incorrect HTTP Method: "GET" is for retrieving data and should be idempotent and safe. Creating a resource changes state and should use "POST".
  • Verb in URI: "createUser" is a verb. URIs identify resources.
  • Query Parameters for Creation: Sending creation data via query parameters is generally discouraged for several reasons:
    • Limited data size.
    • Security concerns (data might appear in server logs, browser history).
    • Less suitable for complex data structures.
    Data for creation should be in the request body.

Corrected Endpoint

POST /users
Status Codes:
  • 201 Created: Successfully created the user. The response body should contain the newly created user resource, and the Location header should point to its URI (e.g., /users/123).
  • 400 Bad Request: If the request body is missing required fields or data is invalid (e.g., name is too short).
  • 401 Unauthorized: If authentication is required but missing/invalid.
  • 403 Forbidden: If the authenticated user doesn't have permission to create users.
  • 409 Conflict: If a user with the same unique identifier (e.g., email) already exists.

Explanation: The POST method on the /users collection resource indicates that a new resource should be added to that collection. The user data (e.g., {"name": "Jo", "email": "jo@example.com"}) would be sent in the request body as JSON.

Endpoint 4: Bulk User Update ▶

Original Endpoint

PUT /updateAllUsers

Design Problems:

  • Verb in URI: "updateAllUsers" contains a verb ("update").
  • Ambiguous Scope: "AllUsers" is vague. Is it truly *all* users in the system, or a specific subset?
  • PUT Semantics: PUT is typically used to *replace* an entire resource at a known URI. If this endpoint is meant to modify *parts* of multiple users or perform a complex bulk operation, PUT might not be the most appropriate. If it's replacing the *entire collection* of users with a new one, then PUT on /users could be valid, but this is rare.
  • Lack of Idempotence (Potentially): If "updateAllUsers" implies applying a change to existing users without fully replacing the entire collection, then repeated calls might have different effects, violating PUT's idempotence principle.

Corrected Endpoints (Multiple Scenarios)

Scenario 1: Replacing the entire collection of users (less common but valid for PUT)

PUT /users
Status Codes:
  • 200 OK: Successfully replaced the entire collection.
  • 204 No Content: If the replacement was successful and no content is returned.
  • 400 Bad Request: If the request body (the new collection) is invalid.
  • 401 Unauthorized, 403 Forbidden.

Explanation: This assumes you are sending a complete, new list of users in the request body, which will *replace* the existing /users collection. This is a very powerful and potentially destructive operation, usually not what "updateAllUsers" implies.

Scenario 2: Partially updating multiple users (Batch Update)

PATCH /users
Status Codes:
  • 200 OK: Successfully applied partial updates to users. Response might contain details of updated users or success/failure for each.
  • 207 Multi-Status: (WebDAV extension, sometimes used) If individual updates within the batch can have different outcomes.
  • 400 Bad Request: If the batch request format is invalid.
  • 401 Unauthorized, 403 Forbidden.
  • 404 Not Found: If some users in the batch do not exist.

Explanation: A PATCH request to the /users collection implies applying a set of partial modifications to multiple resources within that collection. The request body would contain instructions on which users to update and what changes to apply (e.g., an array of objects, each specifying a user ID and the fields to change).

Scenario 3: Updating a single user (most common "update" scenario)

PUT /users/{id}

Explanation: To *replace* an existing user resource entirely with new data. The request body would contain the complete, updated representation of that user.

PATCH /users/{id}
Status Codes:
  • 200 OK: Successfully updated the user.
  • 400 Bad Request: If the request body is invalid.
  • 404 Not Found: If the user with the specified ID does not exist.
  • 401 Unauthorized, 403 Forbidden.

Explanation: To *partially modify* an existing user resource. The request body would contain only the fields to be changed.
