Buggy Version: Shared Mutable State (Race Condition)

Problem: Multiple asynchronous tasks update a single `progress` variable directly and simultaneously. When two tasks read the current `progress` value (e.g., 50%), then each adds its own increment (e.g., +10), they both calculate 60% and write 60% back. One update is lost, leading to an incorrect final progress percentage (it will be less than 100%).

This is a classic Race Condition.

Fixed Version: Atomic Operations / Synchronization (Implicit)

Solution: Each asynchronous task signals its completion, and a central "coordinator" (or the task itself, but carefully) updates the progress based on completed tasks, ensuring each task's contribution is counted exactly once.

Here, we track `completedTasks` and derive the `progress` from that, avoiding direct concurrent modification of the `progress` percentage itself. Each task increments `completedTasks` atomically.
