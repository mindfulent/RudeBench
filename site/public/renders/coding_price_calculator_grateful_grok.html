Shopping Cart Demo: Floating-Point Arithmetic Problems

This demo shows how floating-point numbers in JavaScript can cause issues in financial calculations, like comparing prices in a shopping cart. We'll build two versions:

  • Broken Version: Uses regular floating-point numbers, leading to precision errors and failed comparisons (e.g., with triple-equals, ===).
  • Corrected Version: Uses integer-based handling (storing prices in cents) to avoid precision errors.
Why floating-point equality fails for financial calculations: JavaScript uses floating-point numbers (based on the IEEE 754 standard) to represent decimals. These numbers are stored in binary, and not all decimal values (like 0.1 or 0.99) can be represented exactly. For example: - Adding 0.1 + 0.2 doesn't equal exactly 0.3; it might be something like 0.30000000000000004. - When you use triple-equals (===) for comparison, these tiny errors cause the comparison to fail, even if the values "look" equal. - This is problematic for prices because financial accuracy is critical—e.g., you don't want $0.10 + $0.10 to fail a check against $0.20, which could lead to incorrect billing or failed validations. In the broken version, you'll see these errors in action. The corrected version fixes this by avoiding floats altogether, using integers for cents.
