Counter race condition demo — Buggy vs Fixed

Click "Increment" rapidly on each panel. The buggy version uses overlapping asynchronous read/modify/write operations (setTimeout) on a shared variable and can miss or overwrite updates. The fixed version serializes updates so each increment applies to the most recent value.

Buggy counter (has race)
0
Implementation: reads a shared variable, waits asynchronously, then writes back (classic read-modify-write race).
Event log
Why this is buggy:
- Each increment handler does: read count -> async delay -> compute new value -> write count. If two handlers run concurrently, the second may read the old value before the first has written its result. When both later write, an update can be lost (overwrite).
Fixed counter (updates serialized)
0
Implementation: each async increment is queued so updates apply sequentially to the latest value.
Event log
Why this fixes it:
- The fixed version ensures each increment's asynchronous work is applied after all earlier updates have completed. This prevents overlapping reads of stale values. The queue/promise-chaining strategy effectively serializes read-modify-write.
