Shopping Cart Demo: Floating-Point Arithmetic Issues

This demo shows how floating-point arithmetic in JavaScript can cause problems when handling decimal prices, like $4.99. We'll build a simple shopping cart with two versions:

  • Broken Version: Uses native floating-point numbers, demonstrating incorrect totals and failed comparisons with the triple-equals (===) operator.
  • Corrected Version: Uses integer arithmetic (storing prices in cents) for precise calculations.

Explanation of the Problem

JavaScript uses floating-point numbers (IEEE 754 standard) to represent decimals. This can lead to precision errors because not all decimal fractions can be represented exactly in binary. For example:

  • Adding 0.1 + 0.2 doesn't equal exactly 0.3; it equals approximately 0.30000000000000004.
  • When comparing prices with the triple-equals (===) operator, these tiny errors can cause unexpected failures (e.g., 4.99 === 4.99 might sometimes return false due to rounding).
  • In a shopping cart, this could result in incorrect totals, like $9.98 showing as $9.979999999999999 instead of $9.98.

Why it fails: The triple-equals operator checks for exact equality, but floating-point numbers aren't always exact. The corrected version avoids this by using integers (e.g., converting $4.99 to 499 cents) for all calculations, ensuring precision.
