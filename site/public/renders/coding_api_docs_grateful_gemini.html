REST API Design Review & Best Practices

This document provides an interactive review of proposed REST API endpoints, identifying common design problems and offering corrections based on RESTful best practices. Each section details the issues, proposes a corrected, resource-oriented approach with appropriate HTTP methods, and explains the underlying principles.

General REST Best Practices

  • Resource-Oriented URIs: URIs should identify resources (nouns), not actions (verbs). E.g., /users instead of /getUsers.
  • HTTP Methods for Actions: Use standard HTTP methods (GET, POST, PUT, PATCH, DELETE) to define the action performed on the resource.
  • Consistent Naming: Use plural nouns for collection resources (e.g., /users) and singular nouns for specific resource instances (e.g., /users/5). Use kebab-case for multi-word paths (e.g., /user-preferences).
  • Statelessness: Each request from a client to server must contain all the information necessary to understand the request.
  • Proper Status Codes: Return appropriate HTTP status codes (2xx for success, 4xx for client errors, 5xx for server errors).
  • Hypermedia as the Engine of Application State (HATEOAS): (Advanced) Include links to related resources in responses to guide client navigation.

Endpoint Review

Review of: GET /getUsers

Original Endpoint

GET /getUsers

Identified Problems

  • Verb in URI: The URI /getUsers includes the verb "get". RESTful URIs should represent resources (nouns), not actions. The HTTP method itself (GET) indicates the action.
  • Non-Standard Naming: While "Users" is a noun, the prefix "get" violates the resource-oriented principle.

Corrected Endpoint

GET /users

This endpoint correctly identifies the collection resource "users". The GET method inherently implies retrieval of the resource.

Explanation & Status Codes

For retrieving a collection of resources, the standard practice is to use the plural noun representing the resource type. The HTTP GET method is idempotent and safe, meaning it doesn't change server state and can be called multiple times without different effects.

Status Code Meaning Description
200 OK Success The request was successful, and a list of users (possibly empty) is returned in the response body.
401 Unauthorized Client Error The client is not authenticated to access this resource.
403 Forbidden Client Error The client is authenticated but does not have permission to access this resource.
500 Internal Server Error Server Error An unexpected error occurred on the server.
Review of: POST /user/delete/5

Original Endpoint

POST /user/delete/5

Identified Problems

  • Incorrect HTTP Method: A POST method is typically used to *create* a new resource or perform a non-idempotent action. Deleting a resource should use the DELETE method.
  • Verb in URI: The URI /user/delete/5 contains the verb "delete". URIs should be resource-oriented.
  • Inconsistent Naming: Using /user/ for a single instance, while not strictly wrong, is often better as /users/{id} for consistency with collection naming.
  • Non-Idempotent Action with POST: While a POST can be used for deletion in specific, complex scenarios (e.g., bulk delete with a request body), for a single resource identified by ID, DELETE is the standard and idempotent choice.

Corrected Endpoint

DELETE /users/5

This endpoint correctly uses the DELETE method to remove the user resource identified by ID 5 from the /users collection.

Explanation & Status Codes

The DELETE method is designed for removing a resource identified by its URI. It is an idempotent operation, meaning that calling it multiple times will have the same effect as calling it once (the resource will remain deleted).

Status Code Meaning Description
204 No Content Success The request was successful, and there is no content to send in the response body. This is common for DELETE operations.
200 OK Success The request was successful, and the response body contains a message or status confirming the deletion (less common but acceptable).
404 Not Found Client Error The user with ID 5 was not found.
401 Unauthorized Client Error The client is not authenticated.
403 Forbidden Client Error The client is authenticated but lacks permission to delete this user.
Review of: GET /createUser?name=Jo

Original Endpoint

GET /createUser?name=Jo

Identified Problems

  • Incorrect HTTP Method: A GET method is for *retrieving* data and should not cause side effects (like creating a user). Creating a resource should use the POST method.
  • Verb in URI: The URI /createUser contains the verb "createUser". URIs should be resource-oriented (e.g., /users).
  • Query Parameters for Creation Data: While query parameters can carry data, sensitive or complex creation data should be sent in the request body of a POST request. Using query parameters for creation is less secure (e.g., logged in server access logs, browser history) and limited in size.
  • Non-Idempotent Operation with GET: Repeatedly calling this GET endpoint would create multiple users named "Jo", violating the idempotency principle of GET requests.

Corrected Endpoint

POST /users

This endpoint correctly uses the POST method to create a new user resource within the /users collection. The user's data (e.g., name, email) would be sent in the request body (e.g., as JSON). 

Request Body Example (JSON):
{
  "name": "Jo",
  "email": "jo@example.com"
}

Explanation & Status Codes

The POST method is used to submit an entity to the specified resource, often causing a change in state or the creation of a new resource. When creating a new resource, the request body typically contains the data for the new resource. The server then assigns an ID (or other unique identifier) to the new resource.

Status Code Meaning Description
201 Created Success The request has been fulfilled and resulted in a new resource being created. The response typically includes a Location header pointing to the URI of the new resource and the resource itself in the body.
400 Bad Request Client Error The server cannot process the request due to malformed syntax (e.g., invalid JSON body, missing required fields).
409 Conflict Client Error The request could not be completed due to a conflict with the current state of the resource (e.g., trying to create a user with an email that already exists).
401 Unauthorized Client Error The client is not authenticated.
403 Forbidden Client Error The client is authenticated but lacks permission to create users.
Review of: PUT /updateAllUsers

Original Endpoint

PUT /updateAllUsers

Identified Problems

  • Verb in URI: The URI /updateAllUsers contains the verb "update". URIs should be resource-oriented.
  • Misuse of PUT for Collection Update: While PUT /users *could* theoretically replace the *entire* collection of users with a new one provided in the request body, this is a very rare and often dangerous operation.
  • Ambiguity with "All": "All" in the URI implies an operation on the entire collection, but PUT on a collection is generally used for *replacement*, not partial updates or modifications of existing members. For partial updates to multiple resources, a more specific endpoint or a PATCH request to the collection might be considered (though PATCH on a collection is also complex).
  • Lack of Specificity: If the intent is to update attributes of *all* existing users (e.g., set a new status for all), this operation is highly destructive and requires careful design. A PATCH to a specific resource or a collection with a clear intent (e.g., PATCH /users with a body containing the update criteria and changes) would be more appropriate, but even then, it's complex.

Corrected Endpoint (with considerations for intent)

Scenario 1: Update specific attributes for *all* users (complex bulk operation)

PATCH /users

If the intention is to apply a *partial update* to all users (e.g., change a specific field for every user), PATCH /users could be used, with the request body specifying the update to be applied to all members of the collection. This is a powerful and potentially dangerous operation, requiring careful implementation. The body would define the operation. 

Request Body Example (JSON Merge Patch):
{
  "status": "active"
}
Request Body Example (JSON Patch - more explicit):
[
  { "op": "replace", "path": "/-/status", "value": "active" }
]

Scenario 2: Replace the entire collection of users (rare)

PUT /users

This would completely replace the existing collection of users with the new collection provided in the request body. This is a highly destructive operation and rarely the desired behavior for a "update all users" requirement. 

Request Body Example (JSON Array):
[
  { "id": 1, "name": "New User 1" },
  { "id": 2, "name": "New User 2" }
]

Scenario 3: Update a single user (most common "update" use case)

PUT /users/5

To update a *specific* user, PUT /users/{id} is used for full replacement of the resource, or PATCH /users/{id} for partial updates. Given "updateAllUsers", this isn't the direct intent, but it's the most common update pattern.

Explanation & Status Codes

The PUT method is used to *replace* a resource with the content provided in the request body. If the resource does not exist, PUT can also create it (upsert). When applied to a collection (/users), it implies replacing the *entire* collection, which is often not what "update all users" means.

For updating *parts* of resources within a collection, PATCH is more appropriate. However, applying a PATCH to an entire collection is an advanced pattern that requires careful definition of how the patch applies to each member.

Due to the ambiguity of "updateAllUsers", the correction depends heavily on the precise business requirement. The PATCH /users approach for bulk partial updates is often the closest RESTful interpretation, but it introduces complexity.

Status Code Meaning Description
200 OK Success The request was successful, and the resource(s) were updated. A response body might indicate the affected resources or their new state.
204 No Content Success The request was successful, but there is no content to send in the response body.
400 Bad Request Client Error The request body is malformed or invalid for the update operation.
401 Unauthorized Client Error The client is not authenticated.
403 Forbidden Client Error The client is authenticated but lacks permission to perform bulk updates.
415 Unsupported Media Type Client Error The server does not support the media type sent in the request body (e.g., expecting JSON Patch but received plain JSON for a PATCH request).

Summary of Key Takeaways

  • URIs are Nouns, Methods are Verbs: URIs identify resources (e.g., /users, /products/123), and HTTP methods (GET, POST, PUT, DELETE, PATCH) describe the action on that resource.
  • Use Plural Nouns for Collections: /users for a list of users, /products for a list of products.
  • Use IDs for Specific Resources: /users/{id}, /products/{id}.
  • GET for Retrieval: Safe and idempotent. Don't use for creating or modifying data.
  • POST for Creation: Creates a new resource. The server typically assigns its ID.
  • PUT for Full Replacement/Upsert: Replaces an entire resource. Idempotent.
  • PATCH for Partial Update: Applies partial modifications to a resource. Idempotent (if well-defined).
  • DELETE for Removal: Removes a resource. Idempotent.
  • Clear Status Codes: Always return appropriate HTTP status codes to indicate the outcome of the request.
