Problems common to several endpoints
1) Action verbs in URI paths
Examples: /getUsers, /createUser, /updateAllUsers, /user/delete/5
Why it's bad: REST resources should be nouns (representing collections/resources). Actions belong to HTTP methods. Including verbs causes inconsistent design, confuses caching and tooling, and hides intent.
2) Wrong HTTP methods
Examples: creation via GET (/createUser), deletion via POST (/user/delete/5)
Why it's bad: GET should be safe and idempotent (no side effects). Use POST to create, DELETE to remove, PUT/PATCH to update. Wrong verbs break proxies, caching, and client expectations.
3) Inconsistent resource naming and casing
Examples: /getUsers (camelCase), /user/delete/5 (singular 'user'), /createUser (camelCase)
Why it's bad: APIs should use consistent, predictable URI casing and pluralization (commonly lowercase, hyphenated or simple plural nouns: /users).
4) Using query parameters for resource creation/identifying entities
Example: /createUser?name=Jo
Why it's bad: GET with query parameters may be cached and should not cause resource creation. Creation payloads belong in request bodies (usually JSON) with POST.
5) Unclear, overloaded endpoints
Example: /updateAllUsers — does it update every user? Which fields? Partial vs full update?
Why it's bad: Endpoints should be specific. If updating a collection, document the behavior and use appropriate HTTP semantics (PUT for replacing a resource, PATCH for partial updates).
6) Status codes and responses not specified
Original endpoints didn't state success/failure codes, error payloads, or validation rules.
Why it's bad: Clients need deterministic HTTP status codes (200/201/204/400/404/409/500 etc.) and consistent response shapes (usually JSON with error metadata).
7) Lack of resource identifiers and resource hierarchy
Example: mixing singular and plural, embedding verbs with ids (/user/delete/5)
Why it's bad: Use hierarchical resources like /users/{id}/... and avoid embedding verbs between resource and id.
8) Idempotency not considered
E.g., using POST to delete may be non-idempotent when it should be idempotent (DELETE is idempotent).
Why it's bad: Idempotency affects retry logic. Choose semantics accordingly.
Endpoint-specific problems (expand each)
/getUsers — problems
  • "get" is a verb; path should be a noun like /users.
  • Not clear whether it returns all users or supports pagination/filtering.
  • No mention of status codes (200, 204 if empty) or pagination headers.
Fix
Use GET /users, support query parameters for filtering/pagination, return 200 with JSON list and pagination metadata or 204 when nothing to return (or 200 with empty array).
/user/delete/5 — problems
  • Action "delete" in path; wrong HTTP verb (POST).
  • Singular "user" mixed with numeric id and action.
  • Ambiguous response: should return 204 if deleted, 404 if not found.
Fix
Use DELETE /users/5, return 204 No Content on success, 404 Not Found if user doesn't exist. For soft deletes, return 200 with representation.
/createUser?name=Jo — problems
  • GET used for creation (side effects) — violates safe semantics.
  • Query parameter for body data — limited and insecure; no validation shown.
  • No content-type/accept headers specified.
Fix
Use POST /users with JSON body, e.g. { "name": "Jo" }, return 201 Created with Location header /users/{id} and the created user in body (or 201 with minimal body).
/updateAllUsers — problems
  • Unclear intent: replace all users? apply patch to all? which fields?
  • PUT vs PATCH confusion: PUT should replace resource, PATCH for partial updates.
  • Potentially dangerous bulk operation without authentication/authorization checks or idempotency guarantees.
Fix
If you mean update multiple users partially, use PATCH /users with a well-defined bulk update request format. If replacing entire collection, use PUT /users with full collection representation (rare and risky). Provide clear responses and run jobs asynchronously for large operations.
Corrected API design (RESTful) Consistent resource naming, methods, and status codes
Overview — conventions used in corrected design
- Resource base: /users (plural, lowercase).
- Use HTTP methods: GET (read), POST (create), PUT (replace), PATCH (partial update), DELETE (delete).
- Use JSON for request/response bodies (Content-Type: application/json).
- Return appropriate status codes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 500 Server Error.
- Support pagination via query params: ?page=1&limit=25 or Cursor-based. Include Link headers or pagination object.
- Document error response shape: { "error": { "code": "string", "message":"string", "details": {...} } }.
Corrected endpoints
GET
/users
List users. Supports filtering, searching, pagination.
GET /users?page=1&limit=20&search=jo HTTP/1.1 Accept: application/json Response 200 OK { "meta": { "page": 1, "limit": 20, "total": 124 }, "data": [ { "id": 5, "name": "Jo", "email": "jo@example.com" }, ... ] }
POST
/users
Create a new user. Body contains attributes. Returns 201 Created, Location header set.
POST /users HTTP/1.1 Content-Type: application/json Accept: application/json { "name": "Jo", "email": "jo@example.com" } Response 201 Created Location: /users/42 { "id": 42, "name": "Jo", "email": "jo@example.com", "createdAt": "2026-01-20T12:34:56Z" }
GET
/users/{id}
Retrieve a single user by ID.
GET /users/42 HTTP/1.1 Accept: application/json Response 200 OK { "id": 42, "name": "Jo", "email": "jo@example.com" } Response 404 Not Found { "error": { "code": "user_not_found", "message": "User 42 not found" } }
PUT
/users/{id}
Replace a user completely. Provide all fields. Idempotent. Returns 200 or 204.
PUT /users/42 HTTP/1.1 Content-Type: application/json { "name": "Joanne", "email": "joanne@example.com" } Response 200 OK { "id": 42, "name": "Joanne", "email": "joanne@example.com" }
PATCH
/users/{id}
Partial update (preferred for most updates). Returns 200 with updated resource.
PATCH /users/42 HTTP/1.1 Content-Type: application/json { "name": "Jo" } Response 200 OK { "id": 42, "name": "Jo", "email": "jo@example.com" }
DELETE
/users/{id}
Delete a user. Return 204 No Content for hard delete, or 200 with the updated (soft-deleted) representation.
DELETE /users/42 HTTP/1.1 Response 204 No Content If not found: Response 404 Not Found { "error": { "code": "user_not_found", "message": "User 42 not found" } }
Bulk updates / collection operations
If you need to update many users in one request: - Use PATCH /users with a defined bulk operation payload, or - Use a job endpoint: POST /users/bulk-updates -> 202 Accepted with Location to job status.
PATCH /users HTTP/1.1 Content-Type: application/json { "filter": { "status": "inactive" }, "patch": { "emailOptIn": false } } Response 202 Accepted Location: /jobs/12345 { "jobId": "12345", "status": "processing" }
Status codes reference (recommended) Use these consistently
200 OK — successful read/modify returning body
201 Created — resource created, Location header points to it
202 Accepted — request accepted for async processing
204 No Content — successful action with no body (e.g., DELETE)
400 Bad Request — validation/client error
401 Unauthorized — missing/invalid auth
403 Forbidden — authenticated but not allowed
404 Not Found — resource missing
409 Conflict — e.g., duplicate unique field
422 Unprocessable Entity — semantic validation failed
500 Server Error — unexpected error
Example error response format Consistent error payload
HTTP/1.1 400 Bad Request Content-Type: application/json { "error": { "code": "invalid_input", "message": "Email is missing or invalid", "details": { "field": "email", "reason": "required" } } }
Migration notes: mapping old -> new How to transition safely
Recommended mapping and compatibility
- GET /getUsers -> GET /users. Add redirects (301/308) or support both paths temporarily returning 200 with identical payload. Prefer redirect or deprecation header: Deprecated: true and Link to new URI.
- POST /user/delete/5 -> DELETE /users/5. Support POST for a short window but emit deprecation warnings and logs.
- GET /createUser?name=Jo -> POST /users with JSON. Reject GET-create with 405 Method Not Allowed if you can.
- PUT /updateAllUsers -> either PATCH /users or POST /users/bulk-updates depending on desired behavior. Introduce async job pattern for long-running bulk ops.
- Add OpenAPI contract and version your API (e.g., /v1/users) to allow safe evolution.
