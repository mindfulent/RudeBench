⚙️ How Each Mode Works

💀 Vulnerable — String Concatenation

1
User input is taken raw

Whatever the user types is directly inserted into the query string without any validation or escaping.

2
Query is built by concatenation

The "SQL" becomes: SELECT * FROM products WHERE name LIKE '%' + userInput + '%'

3
Attacker breaks out of the string

By injecting ' OR '1'='1, the attacker changes the query logic entirely, bypassing all filters.

4
All records exposed

Hidden admin records, passwords, sensitive data — everything becomes accessible.

// ❌ VULNERABLE CODE const query = "SELECT * FROM products " + "WHERE name LIKE '%" + userInput + // ← DANGER! "%'"; // If userInput = "' OR '1'='1" // Query becomes: // SELECT * FROM products // WHERE name LIKE '%' OR '1'='1%' // → Returns ALL rows! 💀

🛡️ Safe — Parameterized Queries

1
Query structure defined first

The SQL template is written with placeholders (? or $1) before any user input is involved.

2
Input passed as a separate parameter

User input is never concatenated into the query. The database driver handles binding safely.

3
Input treated as pure data

Even ' OR '1'='1 is treated as a literal string to search for — not as SQL code.

4
Only matching results returned

The query logic cannot be altered. The attacker's payload is harmlessly searched as plain text.

// ✅ SAFE CODE const query = "SELECT * FROM products " + "WHERE name LIKE ?"; // placeholder const params = [`%${userInput}%`]; db.execute(query, params); // DB driver escapes input automatically // If userInput = "' OR '1'='1" // Searches literally for: // "%' OR '1'='1%" → 0 results ✅

📚 Why Parameterized Queries Are Safe

💀 The Core Problem

SQL injection happens when code and data are mixed together. When user input is concatenated directly into a SQL string, the database cannot distinguish between your intended query structure and the attacker's injected commands.


The database parser sees the final string and executes whatever SQL logic it contains — including the attacker's additions.

🛡️ The Parameterized Solution

Parameterized queries (also called prepared statements) separate the query structure from the data. The query template is compiled first, then data is bound to placeholders.


The database knows exactly which parts are "code" (the template) and which are "data" (the parameters). Data can never be interpreted as SQL commands.

🔬 What Actually Happens

  • The DB compiles the query template once
  • Parameters are sent separately in a different protocol channel
  • Special characters like ', --, ; are automatically escaped
  • The query execution plan is fixed and cannot change
  • Even ' OR '1'='1 becomes just a harmless search string

⚡ Real-World Impact

  • Data theft: Dump entire databases
  • Authentication bypass: Login without passwords
  • Data destruction: DROP TABLE, DELETE records
  • Privilege escalation: Access admin accounts
  • SQL injection is #1 in OWASP Top 10 web vulnerabilities
  • Always use parameterized queries or ORMs
