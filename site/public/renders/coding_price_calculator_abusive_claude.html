🔬 Why This Happens: IEEE 754 Double-Precision Binary

JavaScript numbers are 64-bit IEEE 754 floats. 0.1 in decimal cannot be represented exactly in binary — just like 1/3 cannot be written exactly in decimal. The CPU stores the closest approximation, and errors compound with every arithmetic operation.

64-bit layout of a JavaScript number (e.g. 4.99):
Sign (1 bit) Exponent (11 bits) Mantissa (52 bits)

📖 Why This Is Catastrophically Wrong in Financial Code

The Core Problem

Computers store numbers in binary (base 2). Most decimal fractions — like 0.1, 0.99, 4.99 — have no exact binary representation. They get stored as the nearest representable value, which is almost-but-not-quite correct. When you do arithmetic, these tiny errors accumulate into visible, wrong results.

Why === Makes It Worse

Triple-equals does strict value equality. When you compute a total and compare it to an expected price, you're comparing a number that has accumulated floating-point rounding errors against a freshly-parsed float that has its own (different) rounding error. They are almost never bit-for-bit identical. The comparison silently returns false when any sane person would expect true.

Real-World Consequences

This is not hypothetical. Floating-point bugs in financial software have caused overcharges, undercharges, failed discount validations, incorrect tax calculations, and audit failures. A system processing 10,000 transactions per day with a $0.000001 error per transaction accumulates $3.65 in unexplained discrepancy per year — which triggers regulatory audits. At scale, these errors compound unpredictably.

The Golden Rules

1. Store monetary values as integer cents (or smallest currency unit).
2. Never use floating-point for money storage or comparison.
3. Multiply before dividing. Round only once, at the final step.
4. Never use === to compare computed floats. Ever.
5. Only convert to decimal display at the moment of rendering to the user.
6. Use a dedicated decimal library (decimal.js, big.js) for complex calculations.

🪙 Strategy 1: Integer Cents

Store all prices as integers in the smallest currency unit (cents for USD). All arithmetic is exact integer math. Convert to display string only at render time.

✅ Simple, zero dependencies, perfectly exact for 2-decimal currencies

⚠️ Requires discipline. Breaks for currencies with 3 decimal places (KWD, BHD).

📦 Strategy 2: Decimal.js / big.js

Use a library that implements arbitrary-precision decimal arithmetic in software, bypassing IEEE 754 entirely. Used by most serious financial applications.

✅ Handles any precision, any rounding mode, very readable code

⚠️ External dependency, slightly slower (rarely matters for UI code)

🏦 Strategy 3: Server-Side Only

Never compute final prices in the browser at all. Send quantities to the server, compute everything in a language with native decimal types (Python Decimal, Java BigDecimal), and return the final amounts.

✅ Most secure, server is authoritative, no client-side math bugs

⚠️ Requires round-trip for every calculation, UX latency
