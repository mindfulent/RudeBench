Concurrency Bug Demo: Progress Bar

The Shared State Concurrency Problem

In JavaScript, while most code runs in a single-threaded event loop, asynchronous operations (like setTimeout or Promise) can still expose concurrency issues when they interact with a shared mutable state.

The problem arises when multiple asynchronous "tasks" (which are scheduled to run at different points in the future) try to read, modify, and write back the same variable concurrently, without proper synchronization. This is often called a "race condition."

Consider a progress bar. Each task wants to increment the progress. If Task A reads progress = 50%, then Task B also reads progress = 50% before Task A has written back its incremented value. Task A calculates 50 + 10 = 60% and writes it. Task B also calculates 50 + 10 = 60% and writes it. The progress only increased by 10%, not 20%, because Task B overwrote Task A's update. This leads to incorrect values, missed updates, or even values exceeding 100%.

The Fix: The standard approach to fix this is to serialize access to the shared state. This means ensuring that only one task can modify the shared state at any given time. Common patterns include:

  • Queues: Process updates one by one from a queue.
  • Locks/Mutexes: (More common in multi-threaded languages) Prevent simultaneous access.
  • Atomic Operations: Operations that are guaranteed to complete without interruption.
In our JavaScript example, we'll use a simple queue mechanism to process progress updates sequentially.
