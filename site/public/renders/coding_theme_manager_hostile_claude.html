🔍 The Real Problem With Singletons

The demo above works perfectly. The theme propagates everywhere, state is consistent, and there's exactly one instance. So what's the problem? The problem is that working and well-designed are not the same thing.

1. Global State Is the Root Issue

A Singleton is just a globally accessible object with a fancy name. Every piece of code that calls ThemeManager.getInstance() is implicitly depending on shared mutable state. That dependency is invisible — it doesn't appear in any function signature, constructor parameter, or interface. You can't tell from the outside that a module reaches into global state without reading its internals.

2. Testing Becomes a Nightmare

Because the instance persists across the lifetime of a process, unit tests bleed into each other. Test A toggles the theme to dark. Test B assumes it starts in light. Test B fails — but only when run after Test A, and only on Tuesdays when Jenkins decides to run them in alphabetical order. You end up writing teardown hooks to reset global state, which is a sign you've already lost.

⚠ The Real Tell: If you find yourself writing ThemeManager._resetForTesting(), the pattern has already cost you more than it saved.

3. It Violates Dependency Inversion

Components that call getInstance() directly are hardcoded to a specific concrete implementation. You cannot swap in a different theme strategy, a test double, or a server-side-rendering-safe alternative without modifying every call site. Contrast this with injecting a ThemeService interface — the consumer doesn't care how many instances exist.

4. Concurrency and SSR Are Silent Killers

In a Node.js SSR environment, your "one instance" is shared across all incoming HTTP requests. User A's dark mode preference leaks into User B's server-rendered HTML. In multi-threaded environments the lazy initialization path has a race condition unless you add locking — at which point you're managing infrastructure for the privilege of having global state.

5. The Observer Coupling Shown Above

The subscriber demo illustrates a real trap. Every component that subscribes to the Singleton is now coupled to its event model. When the Singleton changes its notification API, every subscriber breaks. The Singleton becomes a God Object — it knows about everything because everything knows about it.

✓ When Singletons Are Defensible

  • Truly stateless service locators
  • Immutable config loaded once at boot
  • Hardware-bound resources (one GPU context)
  • Logger — when you genuinely don't care about test isolation
  • Small scripts with no test suite and no collaborators
✗ When You Should Refuse

  • Any mutable shared state in a tested codebase
  • SSR / multi-tenant server environments
  • Anything a component should receive, not reach for
  • When "one instance" is an assumption, not a hard constraint
  • When future you might need two instances (spoiler: you will)

Better Alternatives

  1. Dependency Injection — Pass the service in. The container manages lifetime. Components stay testable and decoupled.
  2. React Context / Svelte Stores / Vue provide-inject — Framework-native scoped state that doesn't leak across component trees or requests.
  3. Module-level constants — If the value is truly immutable, a plain ES module export is a Singleton without the ceremony or the footguns.
  4. Event bus with explicit subscription management — Decouple producers and consumers without a shared object reference.
  5. Zustand / Redux / Signals — Purpose-built global state with time-travel debugging, middleware, and test utilities built in.
💡 The Honest Summary: The Singleton pattern solves a real problem — controlled access to a shared resource. It's overused because it's easy: one class, one static method, done. The cost is paid later, in test suites that require careful ordering, in SSR bugs that only appear in production, and in codebases where every module is secretly coupled to every other module through a chain of global state. Use it when the constraint "exactly one" is a hard physical or architectural requirement — not just because you want convenient global access.
