REST Endpoint Review — Proposed vs Corrected

This interactive single-file documentation identifies design problems in the proposed endpoints and presents corrected, RESTful alternatives with recommended HTTP methods, URI structure, request/response examples, and status codes.

Original endpoints (proposed)
Four endpoints were proposed. Each card below explains issues and provides a corrected design.
Principles used: resource-oriented URIs, correct HTTP verbs, idempotence, status codes, plural nouns, consistent nesting, & proper parameter placement.
GET (bad)
/getUsers
Problem: 
Verb in URI — "/getUsers" mixes action (get) into the path. URIs should identify resources, not actions.
Problem: 
Pluralization ambiguity — good to use plural nouns consistently (e.g., /users).
Problem: 
Filtering/pagination missing — listing endpoints should support pagination, filtering, and sorting via query parameters.
Corrected design
GET /users Query parameters (optional): - ?page=1&per_page=25 - ?q=Jo (search) - ?sort=created_at&order=desc - ?status=active Description: - Retrieves a paginated list of user resources. - Uses plural noun resource path (/users) and GET to retrieve resources. Recommended response: - 200 OK - Body: JSON array plus pagination metadata Example response: 
HTTP/1.1 200 OK
Content-Type: application/json

{
  "data": [
    { "id": 5, "name": "Jo", "email": "jo@example.com" },
    { "id": 6, "name": "Alex", "email": "alex@example.com" }
  ],
  "meta": {
    "page": 1,
    "per_page": 25,
    "total": 124
  }
}
Notes: - Use query parameters for filtering/search. Avoid embedding verbs in paths. - Consider HATEOAS links or Link headers for pagination if needed.
POST (bad)
/user/delete/5
Problem: 
Action in path — "/delete" is an action in the URI; REST uses HTTP verbs to express actions.
Problem: 
Wrong HTTP verb — POST used to delete; DELETE should be used for deletions.
Problem: 
Inconsistent pluralization — path uses singular /user; prefer plural (/users/{id}).
Corrected design
DELETE /users/5 Description: - Deletes the user resource with id = 5. - Use DELETE method and resource path /users/{id}. Recommended responses: - 204 No Content — deletion successful, no body. - 200 OK with body — allowed if you return the deleted resource or a message. - 404 Not Found — if user with id 5 doesn't exist. - 401/403 — if authentication/authorization fails. 
Example success (no body):
HTTP/1.1 204 No Content

Example not found:
HTTP/1.1 404 Not Found
Content-Type: application/json

{ "error": "User not found", "code": "USER_NOT_FOUND" }
Notes: - If deletion is asynchronous, return 202 Accepted and a Location or operation resource. - Consider soft-delete: update a status field instead of actual deletion; then PATCH /users/5 with { "status": "deleted" } may be used.
GET (bad)
/createUser?name=Jo
Problem: 
Side-effect on GET — GET must be safe and not change server state; this path creates a user.
Problem: 
Action in path — "/createUser" is an operation, not a resource; prefer POST /users.
Problem: 
Using query for body data — creation data should be sent in request body (often JSON), not as query params.
Corrected design
POST /users Content-Type: application/json Request body: { "name": "Jo", "email": "jo@example.com", "password": "securePassword123" // password handling: send over TLS and store hashed } Description: - Use POST on the collection resource (/users) to create a new user. - Use a request body (JSON) for the resource attributes; do not create via GET or query string. 
Example success:
HTTP/1.1 201 Created
Location: /users/42
Content-Type: application/json

{
  "id": 42,
  "name": "Jo",
  "email": "jo@example.com",
  "created_at": "2026-03-08T12:34:56Z"
}
Notes: - 201 Created is preferred; include Location header pointing to the new resource. - For idempotent creation, consider PUT to /users/{client-generated-id} if client provides the id. - Validate input and return 400 Bad Request for malformed data, 409 Conflict for duplicate unique fields (like email).
PUT (questionable)
/updateAllUsers
Problem: 
Action in path — "/updateAllUsers" again mixes action into path.
Problem: 
Ambiguous semantics — Does it replace all users, patch many, or bulk update? PUT on a collection is allowed but must be well-defined.
Problem: 
Potentially unsafe/global operation — Bulk operations should be explicit and careful about idempotence and partial failures.
Corrected design
Options depending on intent: 1) Replace entire collection (rare) PUT /users Request body: full array of user representations Response: 200 OK or 204 No Content (idempotent) 2) Bulk update a set of users (recommended pattern) PATCH /users Content-Type: application/json Request body: { "filter": { "status": "pending" }, "patch": { "status": "active" } } Response: - 200 OK with results summary, or - 202 Accepted if processed asynchronously with a location to track the job 3) Update a single user PATCH /users/{id} Request body: { "email": "new@example.com" } Response: - 200 OK with updated resource - 204 No Content if returning no body Additional: - For long-running bulk ops, implement an operation resource: POST /operations with { "type": "bulk_user_update", ... } and return 202 with Location: /operations/{opId} 
Example bulk async acceptance:
HTTP/1.1 202 Accepted
Location: /operations/87
Content-Type: application/json

{
  "operation_id": 87,
  "status": "pending",
  "submitted_at": "2026-03-08T13:00:00Z"
}
Notes: - Use PATCH for partial updates and PUT only when replacing a full resource. - Design bulk endpoints that clearly specify idempotency, partial success handling, and security/authorization checks to avoid accidental wide changes.
Best-practice summary
Consistent naming, use of correct verbs, proper status codes, and resource modeling.
URI design
  • Use nouns (plural) for collections: /users, /orders
  • Use hierarchical URIs for relationships: /users/{id}/posts
  • Avoid verbs and RPC-style paths (e.g., /getUsers, /doThing)
HTTP methods
  • GET — safe, read-only
  • POST — create (non-idempotent) or to submit processing jobs
  • PUT — replace a resource (idempotent)
  • PATCH — partial update (recommended for updates)
  • DELETE — remove a resource
Other recommendations:
  • Use proper status codes (200, 201, 202, 204, 400, 401, 403, 404, 409, 422, 500).
  • Return useful error bodies (machine-readable codes and human messages).
  • Document pagination, filtering, sorting, and rate limits.
  • Secure endpoints with TLS; avoid passing secrets in query strings.

Tip: choose 201 for creations, 204 for successful deletions with no body, 202 for asynchronous work.
Corrected endpoints quick reference
GET /users — list users (supports paging, filtering)
POST /users — create a user (201 Created + Location)
GET /users/{id} — retrieve a single user (200 or 404)
PATCH /users/{id} — partial update a user (200 or 204)
DELETE /users/{id} — delete a user (204 or 404)
PATCH /users — bulk updates (or POST /operations for async jobs)
