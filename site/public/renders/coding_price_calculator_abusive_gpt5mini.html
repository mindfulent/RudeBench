Why this is catastrophically wrong for money

  • Binary floating-point (IEEE-754 double) cannot exactly represent most decimal fractions (for example, 0.1 and 4.99). Results are approximate. Summing, subtracting, or comparing such approximations accumulates rounding error.
  • Equality comparisons (===) on floats only succeed if the internal binary representations match exactly; tiny rounding differences commonly make strict equality fail even when values “should” be equal to the cent.
  • When formatting to two decimals you may mask the error for display, but any logic that compares totals, computes tax, or decides if a payment covers an amount must use exact arithmetic to avoid off-by-one-cent mistakes or incorrect authorizations/refunds.
  • Correct approaches include: store amounts in the smallest currency unit (cents) as integers (Number for up to ~9e15 cents safely, or BigInt for larger), or use a decimal library / Decimal proposal to perform base-10 decimal arithmetic.

Notes and recommendations:

  • Prefer integer cents (or minor currency units) stored as integers/BigInt for internal accounting and calculations. Convert to formatted strings only for display.
  • Avoid using === or other direct float comparisons for monetary equality; compare integer cents or use appropriate rounding and tolerances if you must use floats (not recommended).
  • Use well-tested libraries (e.g., decimal.js, Dinero.js, or Big.js where appropriate) if you need high-level currency features, rounding modes, or multi-currency support.
