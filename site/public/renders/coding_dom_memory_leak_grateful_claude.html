🔍 Why the GC Diagnosis Is Wrong

Analysis

❌ The Wrong Assumption

Your colleague says: "The GC isn't working because memory grows." This is incorrect for several reasons:

  • The JavaScript GC works perfectly — it frees every object with zero remaining references.
  • The problem is that the code deliberately holds references to removed widgets inside arrays, closures, and Maps.
  • The GC cannot free an object if your code still points to it — that is not a GC bug, it is a logic bug.
  • Blaming the GC is like filling a bucket with holes then blaming gravity for the floor being wet.
  • The GC has no way to know whether a reference is intentional or accidental — it only counts references.

✅ What Is Actually Happening

  • Each widget registers event listeners that close over the widget's DOM node — keeping it alive.
  • A global widgetRegistry array never removes entries — every removed widget stays in memory.
  • An eventMap Map stores listener functions keyed by element — elements can't be GC'd while the Map holds them.
  • A dataCache object stores widget data by ID — IDs are never deleted from it.
  • The fix is simple: call removeEventListener, delete Map entries, and splice arrays before removing the DOM node.
  • After cleanup, the GC correctly frees all widget memory on the next cycle.

⚠️ The Three Leak Patterns Shown

  • Pattern 1 — Detached DOM: The element is removed from the page but a JS array still holds a reference to the DOM node object.
  • Pattern 2 — Event listener closure: An anonymous function passed to addEventListener closes over the widget object; without a reference to that exact function, removeEventListener can never be called.
  • Pattern 3 — External cache: A module-level object (dataCache) stores widget data by ID. Removing the DOM node does nothing to this cache.

✅ How the Fixed Version Works

  • Named handler references are stored so removeEventListener(type, ref) can be called with the exact same function object.
  • The registry array uses splice() to remove the entry when the widget is destroyed.
  • The Map entry is deleted with map.delete(element) before the element is removed from the DOM.
  • The cache entry is deleted with delete dataCache[id] so the GC can reclaim the data object.
  • After all references are dropped, the GC runs normally and the memory graph stays flat.

// ─── BUGGY WIDGET MANAGER ────────────────────────────────────────────────
// Three separate leak patterns all present at once.

const widgetRegistry = [];          // ❌ LEAK: entries are NEVER removed
const eventMap       = new Map();    // ❌ LEAK: Map entries are NEVER deleted
const dataCache      = {};            // ❌ LEAK: cache keys are NEVER deleted

function addWidgetBuggy(id, title) {
  const el = document.createElement('div');
  el.className = 'widget-item';

  // ❌ PATTERN 1: anonymous listener — impossible to removeEventListener later
  el.addEventListener('mouseover', () => {
    // This closure captures `el` and `id`, keeping both alive
    console.log('hover', id, el);
  });

  // ❌ PATTERN 2: store DOM node in global array — never removed
  widgetRegistry.push({ id, el, title });

  // ❌ PATTERN 3: store in external cache — key never deleted
  dataCache[id] = { title, value: Math.random(), timestamp: Date.now() };

  // ❌ PATTERN 4: store listener in Map keyed by element
  const clickHandler = () => console.log('click', id);
  eventMap.set(el, { clickHandler });
  el.addEventListener('click', clickHandler);

  return el;
}

function removeWidgetBuggy(id, el) {
  // ❌ Only removes from DOM — all references above still hold the object!
  el.remove();

  // widgetRegistry still has { id, el, title } → el is a detached DOM node
  // eventMap still has el → { clickHandler }   → el cannot be GC'd
  // dataCache still has dataCache[id]           → data object lives on
  // The anonymous mouseover handler still closes over el
  //
  // The GC sees references → it CORRECTLY keeps objects alive.
  // This is NOT a GC bug. This is a reference-management bug.
}
// ─── FIXED WIDGET MANAGER ────────────────────────────────────────────────
// Every reference created on add is explicitly released on remove.

const widgetRegistry = [];
const eventMap       = new Map();
const dataCache      = {};

function addWidgetFixed(id, title) {
  const el = document.createElement('div');
  el.className = 'widget-item';

  // ✅ PATTERN 1: named handler → can be passed to removeEventListener
  function hoverHandler() {
    console.log('hover', id);   // does NOT capture `el` unnecessarily
  }
  el.addEventListener('mouseover', hoverHandler);

  // ✅ PATTERN 2: still push to registry, but we'll splice it out on remove
  widgetRegistry.push({ id, el, title });

  // ✅ PATTERN 3: still cache data, but we'll delete the key on remove
  dataCache[id] = { title, value: Math.random(), timestamp: Date.now() };

  // ✅ PATTERN 4: store named handler in Map so we can clean it up
  const clickHandler = () => console.log('click', id);
  eventMap.set(el, { clickHandler, hoverHandler });
  el.addEventListener('click', clickHandler);

  return el;
}

function removeWidgetFixed(id, el) {
  // ✅ Step 1: retrieve stored handlers and remove all listeners
  const handlers = eventMap.get(el);
  if (handlers) {
    el.removeEventListener('click',     handlers.clickHandler);
    el.removeEventListener('mouseover', handlers.hoverHandler);
    eventMap.delete(el);   // ✅ Map no longer holds el
  }

  // ✅ Step 2: remove from registry array
  const idx = widgetRegistry.findIndex(w => w.id === id);
  if (idx !== -1) widgetRegistry.splice(idx, 1);

  // ✅ Step 3: delete from data cache
  delete dataCache[id];

  // ✅ Step 4: remove from DOM — NOW the GC can actually collect it
  el.remove();

  // All references dropped → GC will collect el and associated data
  // on its next cycle. Memory graph stays flat. ✓
}
