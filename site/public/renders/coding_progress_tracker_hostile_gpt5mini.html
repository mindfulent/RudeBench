Progress Tracking: Buggy vs Fixed (single-file demo)

Simulated worker tasks each report "I completed N units". Tasks are asynchronous (setTimeout) and can overlap. Left shows a common buggy approach: tasks read a shared counter, compute a new value, and write it back — causing lost updates. Right shows two fixes: serialized updates using an "actor" (queue) or a simple lock (Promise mutex).

Buggy version Shared mutable state

Progress: 0% (expected: 0)
Each task does: read sharedCounter; compute new = sharedCounter + units; write sharedCounter = new. Because tasks interleave, two tasks can read the same old value then both write, losing one of the increments.
Final observed: 0 — BROKEN (lost updates)

Fixed version Serialized updates

Progress: 0% (expected: 0)
Using a single updater to apply increments one at a time (actor) or a small mutex to serialize access prevents lost updates.
Final observed: 0 — OK
