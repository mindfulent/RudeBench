Understanding the Race Condition

A race condition occurs when multiple operations try to access and modify shared data concurrently, and the final outcome depends on the non-deterministic order in which these operations execute. In this example, "concurrently" is simulated using setTimeout(..., 100), which introduces a delay, allowing multiple clicks to initiate updates before previous ones complete.

Consider the "Read-Modify-Write" cycle for updating the counter:

  1. Read: Get the current value of the counter variable.
  2. Modify: Increment the value.
  3. Write: Update the counter variable with the new value.

How the Buggy Counter Fails:

When you click the "Buggy" button rapidly, multiple increment requests are initiated. Each request involves a simulated delay (setTimeout). Imagine the following sequence if you click twice very quickly:

  • Click 1:
    1. Reads count = 0.
    2. Schedules a modification to 0 + 1 and a write after 100ms.
  • Click 2 (happens before Click 1's write completes):
    1. Reads count = 0 (it's still 0 because Click 1 hasn't written yet!).
    2. Schedules a modification to 0 + 1 and a write after 100ms (potentially slightly later than Click 1).
  • After 100ms (Click 1's turn):
    1. Writes count = 1.
  • Shortly after (Click 2's turn):
    1. Writes count = 1 (overwriting Click 1's update with its own stale calculated value).

The expected count after two clicks is 2, but the buggy counter ends up at 1. The updates "race" to complete, and the last one to write often overwrites earlier, valid updates because it started with an outdated read.

The Fix: Using a Queue or Atomic Operations (Simulated)

The fixed counter resolves this by ensuring that each update operation reads the latest value of the counter when it's actually performing the modification, or by processing updates sequentially. In JavaScript, since there are no built-in atomic operations for simple variables like in lower-level languages, we typically simulate this by making sure operations don't step on each other's toes.

The common approach for such scenarios in an asynchronous context is to:

  1. Process updates sequentially: Use a promise chain or a queueing mechanism to ensure that only one update operation is "active" at a time, or that subsequent operations wait for the previous one to complete before reading the shared state.
  2. Read the latest state at the time of modification: Instead of reading the state early and holding onto that potentially stale value, read the state immediately before applying the modification.

Our fixed counter uses a simpler, more direct fix for this specific problem: it ensures that when an asynchronous update *completes*, it doesn't just write a pre-calculated value. Instead, it directly increments the current shared variable at the moment of writing, effectively making the "read-modify-write" a single conceptual step that operates on the most current state.

Specifically, the buggy code does setTimeout(() => count = tempCount + 1, 100) where tempCount was read much earlier. The fixed code does setTimeout(() => count++, 100) or similar, ensuring the increment operates on the current value of count when the timeout fires.

This is a simplified fix that works well for a single increment operation. For more complex state changes, a proper queueing system (e.g., using promises) or state management patterns (like Redux reducers) would be necessary to ensure all updates are processed correctly and deterministically.
