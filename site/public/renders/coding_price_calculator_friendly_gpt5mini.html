Shopping Cart: floating-point pitfalls vs fixed cents

This demo shows why using JavaScript Number with naive === comparisons and arithmetic can produce surprising results for decimal prices (e.g., 4.99). The right approach here is to handle money in integer cents (or use a decimal library).

Explanation (short)

    
JavaScript Numbers are IEEE-754 binary floating point. Many decimal fractions like 4.99 or 0.01 do not have an exact binary representation, so arithmetic and equality checks can fail:

    
    
      
  • 4.99 is actually stored as something like 4.9899999999999998 internally.
    • 
      
  • Adding 0.01 may not give exactly 5.00, so (4.99 + 0.01) === 5.00 can be false.
    • 
    

    
Solution: represent money as integer smallest-units (cents) or use a decimal library (decimal.js, big.js, or BigInt with scaled integers). Use formatting only when displaying to users.
