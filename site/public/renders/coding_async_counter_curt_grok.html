Race Condition Counter Demonstration

Buggy Counter

0

This version has a race condition. Rapid clicks trigger overlapping setTimeout calls that read, modify, and write a shared counter variable asynchronously. This can cause the count to display incorrect values, as updates may overwrite each other.

Fixed Counter

0

This version uses a lock to prevent overlapping updates. Even with rapid clicks, the counter ensures that only one update processes at a time, avoiding lost increments.

Explanation of the Race Condition and Fix

Race Condition in Buggy Version: In the buggy counter, each button click schedules a setTimeout function that asynchronously reads the current count, increments it, and writes it back. If you click the button rapidly, multiple setTimeout calls may overlap. For example, if the count is 0, two quick clicks might both read 0, increment it to 1, and write 1 back—resulting in the final count being 1 instead of 2. This happens because the operations are not atomic, leading to lost updates and incorrect values.

Fix in Fixed Version: To resolve this, a simple lock (a boolean flag) is used to ensure that only one update can be scheduled at a time. If an update is already in progress, additional clicks are ignored until the current one completes. This serializes the updates, preventing overlaps and ensuring the counter always reflects the correct value. In a real-world scenario, you might use more robust synchronization like promises, mutexes, or a queue for better handling.

How to Test: Click the "Increment (Buggy)" button rapidly (e.g., 5-10 times quickly) and observe the displayed count—it may not match the number of clicks. Then, do the same with the "Increment (Fixed)" button; it should accurately reflect the clicks.
