🐛 Why the Buggy Version Fails

Each task reads sharedProgress, adds its chunk, then writes back — a classic read-modify-write race:

  • Task A reads progress = 40
  • Task B reads progress = 40 (same snapshot!)
  • Task A writes progress = 60 (+20)
  • Task B writes progress = 60 (+20 again, overwriting A's result)
  • Net effect: only +20 instead of +40 → lost updates
  • Or tasks accumulate beyond 100 → overflow
  • JS is single-threaded but await yields control between tasks, creating the interleaving window

✅ How the Fix Works

Updates are serialized through a Promise-based async queue (mutex):

  • Each task enqueues its update; the queue processes one at a time
  • A lock Promise chains updates sequentially: lock = lock.then(update)
  • No two tasks can be inside the critical section simultaneously
  • Progress increments are clamped to [0, 100] as an extra guard
  • Result: monotonically increasing, no skips, no overflow, correct final value
  • This pattern mirrors Mutex / Semaphore in multi-threaded languages
