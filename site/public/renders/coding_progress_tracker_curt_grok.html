Shared State Concurrency Bug Demo

Explanation of the Shared State Problem

In concurrent programming, a shared state concurrency bug occurs when multiple tasks access and modify a shared variable (like a progress counter) without proper synchronization. This can lead to race conditions, where the final state of the variable is unpredictable due to the order of operations. For example:

  • In the buggy implementation, multiple tasks (simulated with setTimeout and Promises) read and update a shared progress variable at the same time. This might cause:
    • Values to be overwritten (e.g., two tasks read the same value and add to it, resulting in lost updates).
    • Progress to skip percentages (e.g., it jumps from 20% to 60% instead of 40%).
    • Progress to exceed 100% (e.g., due to duplicate or incorrect increments).
  • In the fixed implementation, we use a promise chain to serialize updates, ensuring that tasks wait for the previous one to complete before modifying the shared state. This prevents race conditions and ensures accurate results.

This demo uses 5 concurrent tasks, each intended to add 20% to the progress (totaling 100%). Run both sections to compare the behavior.
