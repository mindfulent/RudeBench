Progress Bar Concurrency Bug Demonstration

Buggy Version

Explanation of the Problem: In this version, multiple concurrent tasks (simulated with setTimeout and Promise) directly access and update a shared progress variable without synchronization. This leads to a race condition, where tasks might read the same outdated value, overwrite each other's changes, or cause the progress to exceed 100% or display incorrect values. For example, if two tasks try to add to the progress at the same time, one might add based on an old state, resulting in inconsistencies.

Corrected Version

Explanation of the Fix: In this version, we use a simple queue to synchronize access to the shared progress variable. Each task adds its update to a queue, and updates are processed one by one using a promise-based loop. This ensures that modifications are atomic and sequential, preventing race conditions. As a result, the progress bar accurately reflects the total (e.g., it will reach exactly 100% without errors). This mimics a basic mutex or lock in a single-threaded environment like JavaScript.
