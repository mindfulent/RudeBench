🔬 Why Parameterized Queries Are Structurally Immune to Injection

💀 String Concatenation (Vulnerable)

The query is built as a single string. User input becomes part of the query's syntax. The parser cannot distinguish between developer-intended SQL and attacker-injected SQL.

// Developer writes: const query = "SELECT * FROM products " + "WHERE name LIKE '%" + userInput // ← DANGER ZONE + "%'"; // User enters: ' OR '1'='1 // Query becomes: SELECT * FROM products WHERE name LIKE '%' OR '1'='1%' // ↑ Always true — returns ALL rows

The database engine receives one string and parses it. By the time it sees the query, it has no way to know which parts were written by the developer and which were injected.

🛡️ Parameterized Query (Safe)

The query structure and the data are sent separately. The database engine compiles the query template first, then binds the value. Syntax is already locked.

// Developer writes: const query = "SELECT * FROM products " + "WHERE name LIKE ?"; // ← placeholder const params = [`%${userInput}%`]; // Sent separately to DB engine // User enters: ' OR '1'='1 // DB receives two separate things: Query: "SELECT * FROM products WHERE name LIKE ?" Param: ["%' OR '1'='1%"] // ↑ Treated as a literal string, not syntax

The query structure is compiled before the value is ever seen. The engine treats the parameter as a plain string value — quotes, semicolons, keywords are all inert.

🧠 The Exact Mechanism — Step by Step

1

Parse phase (parameterized): The DB engine parses SELECT * FROM products WHERE name LIKE ? and builds a query execution plan. The ? is a typed slot — it expects a string value, not SQL syntax.

2

Bind phase: The user's input is bound to that slot as a raw scalar value. The parser is already done. There is no second parse of the user's string. Single quotes in the input are just characters in a string, not SQL delimiters.

3

Execute phase: The engine runs the pre-compiled plan with the bound value. ' OR '1'='1 becomes the literal search term — it finds products whose name literally contains that character sequence. Zero rows, no breach.

4

Why concatenation fails: With string building, there is only one string by the time the DB sees it. The engine parses it all as SQL. It cannot retroactively know what was "data" vs "code" — that distinction was destroyed when you concatenated.

5

Escaping is not sufficient: Manually escaping quotes is fragile — different DB engines have different rules, encoding edge cases exist, and one missed escape is a breach. Parameterization is a structural guarantee, not a sanitization heuristic.
