🔍 Why Does This Break?

JavaScript is single-threaded, but asynchronous operations create interleaving. When you use setTimeout (or any async mechanism), the engine suspends that task and runs others before coming back. This produces a classic read–modify–write race condition.

The Buggy Pattern BUG

// Shared state
let counter = 0;

function buggyIncrement() {
  const current = counter;          // 1. READ  — captures value NOW
  setTimeout(() => {
    counter = current + 1;           // 2. WRITE — uses the stale snapshot
  }, Math.random() * 80);
}

The Race — Step by Step

  1. Call #1 fires. It reads counter = 0 and stores it in current₁ = 0.
  2. Call #2 fires immediately after. It also reads counter = 0 → current₂ = 0.
  3. …calls #3 through #10 all do the same. They all captured 0.
  4. Timeouts begin resolving. Call #1 writes counter = 0 + 1 = 1. ✓
  5. Call #2 writes counter = 0 + 1 = 1. It overwrites call #1's result.
  6. All 10 calls write 1. Final value: 1 instead of 10.

The Fixed Pattern FIX

let counter = 0;

function fixedIncrement() {
  // No local snapshot taken here
  setTimeout(() => {
    counter = counter + 1;            // READ + WRITE happen together at callback time
  }, Math.random() * 80);
}

Because JavaScript callbacks execute on the single-threaded event loop, each callback runs to completion before the next one starts. Reading and writing counter inside the same callback is therefore atomic with respect to other callbacks — no other callback can sneak in between the read and the write.

The Key Insight

The bug is not about when the write happens — it's about which value is being written. Capturing state into a local variable before an async gap means you're scheduling a write of yesterday's truth. The fix keeps the read inside the callback so it always sees today's truth at the moment of execution.

Real-World Equivalents

This same pattern destroys correctness in: React setState calls that close over stale state (fix: use the updater form setState(prev => prev + 1)), database read-modify-write without transactions, and multi-threaded code without locks. The shape of the bug is always identical.
