🧠 Why Your Colleague Is Wrong: A Complete Breakdown

⚠️ The Myth: "It's a GC Problem"

JavaScript's garbage collector uses mark-and-sweep: starting from root objects (window, global scope), it traces every reference. Anything still reachable is not collected — by design. Blaming the GC means assuming it should collect objects you are still pointing to. That's not a GC bug. That's you holding references and not letting go. The GC is working exactly as specified. The leak is 100% in application code.

❌ What the colleague thinks is happening

Widget is deleted from the DOM → GC should automatically clean everything up → GC isn't running → memory grows → solution: force GC or wait longer. This assumes all references disappear when the DOM element is removed. They absolutely do not.

✅ What is actually happening

Widget is deleted from DOM but: (1) the leaking array still holds a direct reference to the detached DOM node, (2) every interval/event listener created for that widget still holds a closure referencing the widget's data object. The GC sees live references and correctly keeps everything in memory. The leak is the references, not the GC.

❌ Why "force GC" won't fix it

Even if you could force a GC cycle (you can't reliably in production), it would find the same live references and collect nothing. Forcing GC on a real leak is like emptying a bucket while the tap is still running. You must close the tap — remove the references — first.

✅ The actual fix

Before removing a widget: call removeEventListener with the exact same function reference, clear all intervals/timers, null out or splice the DOM reference from the tracking array. Now the GC sees zero references to the widget objects and collects them on the very next cycle. The GC was never broken.

❌ Leaking Code Pattern

// Global leak accumulator — never cleaned
const leakedRefs = [];
const leakedListeners = [];

function addWidget(element) {
  // Store DOM ref — never removed on delete
  leakedRefs.push(element);

  // Handler created inline — can't removeEventListener
  // because we lose the reference immediately
  element.addEventListener('click', function() {
    // Closure captures element, widgetData, etc.
    console.log(element.id, widgetData);
  });
  // ^ anonymous fn reference is lost forever

  // Interval never cleared on widget removal
  setInterval(function() {
    element.textContent = getData();
  }, 1000);
  // ^ timer ID not stored, can never clearInterval
}

function removeWidget(element) {
  element.remove();
  // DOM gone from view but:
  // leakedRefs[n] still points to detached node
  // interval still firing on detached node
  // event listener still attached and alive
  // GC cannot collect ANY of this
}

✅ Fixed Code Pattern

// Clean registry — properly managed
const widgetRegistry = new Map();

function addWidget(element, id) {
  // Named handler — reference kept for cleanup
  const handler = (e) => handleClick(e, id);

  // Store timer ID so we can clear it
  const timerId = setInterval(function() {
    element.textContent = getData();
  }, 1000);

  element.addEventListener('click', handler);

  // Registry holds everything needed for cleanup
  widgetRegistry.set(id, {
    element, handler, timerId
  });
}

function removeWidget(id) {
  const entry = widgetRegistry.get(id);
  if (!entry) return;

  // 1. Stop the timer
  clearInterval(entry.timerId);

  // 2. Remove listener with exact same reference
  entry.element.removeEventListener(
    'click', entry.handler
  );

  // 3. Remove DOM node
  entry.element.remove();

  // 4. Delete from registry — zero references left
  widgetRegistry.delete(id);
  // GC now sees no references → collects everything ✓
}
