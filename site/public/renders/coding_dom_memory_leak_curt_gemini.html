The Case of the Growing Memory: It's Not the Garbage Collector!

The Real Culprit: Uncleaned References and Event Listeners

When a dashboard widget application shows growing memory usage over time, even after widgets are "removed," the immediate thought often jumps to a "bad" garbage collector (GC). However, modern JavaScript GCs are highly optimized and efficient. The problem almost always lies in *your code* creating references that prevent the GC from doing its job.

This demo illustrates the most common cause: uncleaned event listeners and lingering DOM references.

  • Event Listener Leaks: When you add an event listener to an element (e.g., a button inside a widget), and then remove that element from the DOM, if the listener itself (or the function it calls) still holds a reference to the element or the widget's data, the element cannot be garbage collected.
  • DOM Reference Leaks: If your JavaScript code keeps an array or object holding references to DOM elements that are no longer in the document, those elements cannot be collected.
  • Closures: Functions created within other functions (closures) can capture variables from their outer scope. If a closure is kept alive (e.g., as an event listener), it can prevent the garbage collection of all the variables it captured, even if they're no longer directly used.

The GC isn't "failing"; it's doing exactly what it's told: keeping alive anything that is still *reachable* from the root (like the `window` object or active execution stacks). If your code maintains a reference, it's reachable.

To observe the leak: Open your browser's Developer Tools (F12), go to the "Memory" tab, and take a heap snapshot. Add and remove many widgets from the "Buggy Dashboard." Take another snapshot and compare. You'll see detached DOM nodes and event listeners accumulating. Do the same for the "Fixed Dashboard" and observe the difference.
