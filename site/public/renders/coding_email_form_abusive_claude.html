📧 Email Validation

Real-time regex validation — with full honesty about its limitations.

​

Regex Checks

○ Not empty
○ Contains exactly one @ symbol
○ Local part (before @) is valid characters
○ Domain has at least one dot
○ TLD is 2–24 characters
○ Total length ≤ 254 characters (RFC 5321)

⚠️ Why Regex Is Fundamentally Wrong For This

RFC 5321 and RFC 5322 define what a valid email address actually is, and regex cannot fully implement them. Here's what regex gets wrong:

1. Quoted local parts are valid. "hello world"@example.com is a perfectly legal email address per RFC 5321. Almost every regex rejects it.

2. Comments are valid. (comment)user@example.com is legal. Regex won't handle it.

3. IP address literals are valid. user@[192.168.1.1] and user@[IPv6:::1] are valid. Regex typically rejects them.

4. Internationalized domains (RFC 6531). 用户@例子.广告 is a valid email. Regex with basic character classes fails completely.

5. Regex cannot verify deliverability. totally@valid-regex.com passes every regex but may not exist. Regex tells you nothing about whether the mailbox actually exists.

6. A fully RFC-compliant regex is a monster. The regex required to truly implement RFC 5322 is thousands of characters long and still won't catch deliverability issues.

✅ What You Should Actually Do

1. Minimal sanity regex + confirmation email Use a simple regex just to catch obvious typos (missing @, no domain). Then send a confirmation email. This is the only real proof the address works. Every serious platform does this.
2. HTML5 native input type="email" The browser's built-in validation is already a reasonable sanity check and handles more edge cases than most hand-rolled regex. It's not RFC-complete either, but it's good enough for a first pass and it's free.
3. Server-side library validation Use email-validator (Node.js), python-email-validator (Python), or egulias/email-validator (PHP). These libraries implement the RFCs properly and are maintained.
4. MX record lookup (DNS validation) Check that the domain actually has mail exchange records configured. This proves the domain can receive email, though not that the specific mailbox exists. Do this server-side only.
5. Email verification API services Services like ZeroBounce, Hunter.io, or Abstract API perform DNS, MX, SMTP handshake checks, and disposable email detection. Overkill for most apps, essential for bulk senders.
6. The honest bottom line No client-side validation proves an email is real. Use regex only to improve UX by catching obvious typos early. Always confirm via email. Everything else is theater.
