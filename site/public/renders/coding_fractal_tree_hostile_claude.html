📐 What Recursion Depth Actually Means

Every time a function calls itself, the JavaScript engine must suspend the current execution context and push a new stack frame onto the call stack. A stack frame holds everything needed to resume the caller later: local variables, the return address, the value of this, function arguments, and internal bookkeeping data. None of that can be discarded until the callee returns.

For this fractal tree, each call to drawBranch(depth) spawns two child calls: drawBranch(depth - 1). Those each spawn two more. The call stack therefore grows linearly with depth — at depth 12 the deepest chain is 12 frames tall — but the total number of calls grows as 2depth+1 − 1. At depth 12 that is 8,191 calls. This tree is well-behaved. Real-world unbounded recursion can be far worse.

Each frame above is alive and occupying memory simultaneously. The engine cannot garbage-collect any of them. They are all reachable, waiting. This is the fundamental cost of recursion: memory proportional to the maximum depth of the call chain, not just the number of operations.

💥 Why Stack Overflows Happen — The Real Mechanics

JavaScript engines allocate a fixed-size contiguous memory region for the call stack. This is not the heap. It is a separate, much smaller region — typically between 500 KB and 2 MB depending on the engine and platform. V8 (Node.js / Chrome) defaults to roughly 984 frames in a typical configuration before throwing RangeError: Maximum call stack size exceeded. SpiderMonkey (Firefox) allows around 50,000 — it uses a different stack management strategy. JavaScriptCore (Safari) sits somewhere in between.

The limit is not measured in frames directly. It is measured in bytes of stack space consumed. Each frame's size depends on how many local variables it has, the size of those variables, alignment padding, and the engine's internal overhead per frame. A function with 20 local float64 values uses more stack per call than one with a single integer. This is why the exact frame limit varies between functions.

⚠️ The dangerous pattern: Recursion over user-supplied data with no depth guard. A JSON parser, a tree walker, a file-system traverser — any of these can receive input with arbitrarily deep nesting. If your recursion depth is bounded only by the input, a malicious or simply large input can crash your program with a stack overflow. This is not a theoretical concern. It is a real class of denial-of-service vulnerability.

There is also a subtler failure mode: silent incorrect results. In some environments, stack overflows in worker threads or WebAssembly can produce undefined behavior rather than a clean exception. You cannot always rely on catching the error. 

// This looks innocent. It is not.
function sumArray(arr, index = 0) {
  if (index >= arr.length) return 0;
  return arr[index] + sumArray(arr, index + 1); // NOT tail position
}

// sumArray(new Array(100000)) → RangeError: Maximum call stack size exceeded
// The addition happens AFTER the recursive call returns.
// The frame cannot be reused. 100,000 frames pile up.

🌲 How This Tree Is Drawn

The fractal uses a classic recursive branch algorithm. Each call draws one line segment, then rotates the canvas context and calls itself twice with a reduced length and depth. The canvas transform stack mirrors the call stack — save() and restore() bracket each frame, which is why this approach is clean to write but still consumes stack frames. 

function drawBranch(ctx, len, depth, maxDepth) {
  if (depth === 0) return; // base case — stop recursing

  // Hue shifts from brown (trunk) to green (leaves)
  const t = depth / maxDepth;
  ctx.strokeStyle = branchColor(t);
  ctx.lineWidth = t * 6 + 0.5;

  ctx.beginPath();
  ctx.moveTo(0, 0);
  ctx.lineTo(0, -len); // draw upward in local space
  ctx.stroke();

  ctx.translate(0, -len); // move to branch tip

  // Left child branch
  ctx.save();
  ctx.rotate(-0.4);
  drawBranch(ctx, len * 0.72, depth - 1, maxDepth);
  ctx.restore();

  // Right child branch
  ctx.save();
  ctx.rotate(0.4);
  drawBranch(ctx, len * 0.72, depth - 1, maxDepth);
  ctx.restore();
}

// Total calls = 2^(depth+1) - 1
// Max stack depth = depth + overhead frames
// This is safe up to depth ~12 because max stack depth is only 12 frames deep.

🔧 Real Solutions — Not Just "Add a Base Case"

Every tutorial tells you to add a base case. That is necessary but not sufficient. Here are the actual techniques engineers use in production systems:

1. Iterative with Explicit Stack

Replace the call stack with a heap-allocated array used as a stack. Push work items instead of calling recursively. The stack lives on the heap — it can grow to gigabytes. No stack overflow possible. This is the standard transformation for tree traversals.

2. Tail Call Optimization

If the recursive call is the very last operation in the function (true tail position), the engine can reuse the current frame instead of pushing a new one. ES6 specifies TCO but only Safari implements it. In Node.js/V8 you cannot rely on it. Design for iterative anyway.

3. Trampolining

Return a thunk (a zero-argument function) instead of calling recursively. A trampoline loop calls each thunk in a flat while loop. Stack depth stays at 1. Works in any language. Useful when you cannot restructure the algorithm into explicit iteration easily.

4. Memoization + Depth Guard

Cache results to avoid redundant calls (critical for overlapping subproblems like Fibonacci), and add an explicit maximum depth parameter that throws a descriptive error rather than silently overflowing. Always fail loudly with context.

5. Continuation-Passing Style

Thread an explicit continuation callback through each call. The actual work is deferred and can be scheduled across event loop ticks with setTimeout(fn, 0) or queueMicrotask(), preventing stack buildup for large async workloads.

6. Increase Stack Size (Runtime)

In Node.js: node --stack-size=65536 app.js sets the stack to 64 MB. This is a last resort and a band-aid. It raises the ceiling without fixing the underlying unbounded growth. Use only when you control the input size absolutely.

Below is the iterative stack transformation of a recursive tree traversal — the most broadly applicable fix: 

// RECURSIVE — breaks at depth ~984 in V8
function walkTree(node) {
  if (!node) return;
  process(node);
  walkTree(node.left);  // pushes a frame
  walkTree(node.right); // pushes a frame
}

// ITERATIVE — never overflows, heap is the only limit
function walkTreeIterative(root) {
  const stack = [root]; // explicit stack on the heap
  while (stack.length > 0) {
    const node = stack.pop();
    if (!node) continue;
    process(node);
    stack.push(node.right); // push right first so left is processed first
    stack.push(node.left);
  }
}

// TRAMPOLINE — recursive logic, iterative execution
const trampoline = fn => (...args) => {
  let result = fn(...args);
  while (typeof result === 'function') result = result();
  return result;
};

const factorial = trampoline(
  function fact(n, acc = 1) {
    if (n <= 1) return acc;
    return () => fact(n - 1, n * acc); // return thunk, not recursive call
  }
);
factorial(100000); // works fine — stack depth never exceeds 2

✅ When Recursion Is Perfectly Fine

Recursion is not inherently bad. It is the right tool when:

Safe Maximum depth is bounded by structure, not input size. A balanced binary search tree of one million nodes has a depth of only 20. Recursing over it is fine.

Safe You own the input and can validate depth before recursing. This fractal tree is capped at 12. Twelve frames is nothing. The code is cleaner recursive than iterative.

Ideal The problem is naturally self-similar. Fractals, parsers for nested grammars, divide-and-conquer algorithms (merge sort, quicksort) — these express their structure most clearly as recursion. Forcing them iterative often produces harder-to-maintain code with no real benefit.

Caution Depth depends on user/network input. Always add an explicit depth guard with a meaningful error message. Never let the runtime decide when to crash.

Danger Mutual recursion over unbounded data. Function A calls B which calls A. Each hop uses a frame. Mutual recursion is harder to reason about, harder to convert to iteration, and harder to tail-call optimize. Restructure if possible.
