Stack-overflow risks, practical solutions, and explanation

  • Stack-overflow risk: each recursive call consumes a stack frame. Deep recursion can exhaust the language runtime's call stack and throw an error (in browsers this is typically a RangeError: Maximum call stack size exceeded).
  • Practical solutions demonstrated in this page:
    • Convert recursion to iteration using an explicit stack (LIFO data structure). This avoids using the language call stack and prevents stack overflow for the same logical depth.
    • Use smaller recursion depth (limit problem size) or redesign algorithm to be tail-recursive where supported. JavaScript engines do not reliably implement tail-call elimination, so tail recursion is not a dependable solution in browsers.
    • Use trampolining (break recursive work into repeated loop steps) to avoid deep call chains; this effectively converts recursion to iteration at the application level.
    • Use continuation-passing or asynchronous chunking (setTimeout / requestAnimationFrame) to split work into smaller call chains to avoid long blocking stacks and keep UI responsive.
  • Trade-offs: iterative solutions require manual stack/state management but are robust. Tail-call elimination is ideal but not portable in JS. Chunking helps responsiveness but adds complexity.

Concise proof of understanding of recursion

Recursion defines a function in terms of itself and requires: (1) a base case that terminates recursion, and (2) a rule that reduces the problem toward the base case. By induction: if the function is correct for n=1 (base) and if correctness at n-1 implies correctness at n via the recursive step, the function is correct for all finite n. Practically, each recursive call uses a stack frame storing local variables and the return address; unbounded recursion can exhaust available stack memory.
